1.3.1 环境准备

在开始之前，确保你已经安装了 CANN Toolkit。运行以下命令检查环境：

source /usr/local/Ascend/ascend-toolkit/latest/set_env.sh
cat /usr/local/Ascend/ascend-toolkit/latest/version.cfg

如果输出了版本号（如 8.1.RC1），说明环境已就绪。如果提示文件不存在，说明还没安装 CANN，请参考 CANN 安装指南 完成安装。

1.3.2 下载 ops-blas 仓

本教程使用 ops-blas 仓库中的 sswap 算子作为示例。sswap 是 BLAS Level-1 标准中的一个 routine，功能是交换两个向量的对应元素：调用 swap(x, y) 后，x 变成原来的 y，y 变成原来的 x。

git clone https://gitcode.com/cann/ops-blas.git
cd ops-blas

1.3.3 看看代码长什么样

先瞄一眼 sswap 的文件结构，不用全部理解，后面章节会逐步讲解每一部分：

ops-blas/
└── blas/
    └── swap/
        └── sswap/
            └── arch35/
                ├── sswap_host.cpp          # Host 侧代码（CPU 上运行）
                ├── sswap_kernel.cpp        # Kernel 代码（NPU 上运行）
                └── sswap_tiling_data.h     # Tiling 参数结构体

三个文件各司其职：

  • sswap_tiling_data.h：定义 TilingData 结构体——告诉 NPU 每个核该处理多少数据。
  • sswap_kernel.cpp：Kernel 代码，运行在 NPU 的 AI Core 上。负责把数据从 Global Memory 搬到 UB（片上缓存），在 UB 上完成交换，再把结果搬回 Global Memory。
  • sswap_host.cpp：Host 侧代码，运行在 CPU 上。负责参数校验、计算 Tiling 参数、申请显存、启动 Kernel、回收结果。

Kernel 代码速览

下面展示 sswap Kernel 的骨架代码（已简化，展示核心结构）：

class SswapAIV {
public:
    __aicore__ inline void Init(GM_ADDR x, GM_ADDR y, GM_ADDR tilingGm, TPipe* pipe);
    __aicore__ inline void Process();

private:
    __aicore__ inline void SingleIteration(uint32_t curOffset, uint32_t dataCount);
    GlobalTensor<float> xGM_;
    GlobalTensor<float> yGM_;
    SswapTilingData tiling_;
};

extern "C" __global__ __aicore__ void sswap_kernel(
    GM_ADDR x, GM_ADDR y, GM_ADDR workSpace, GM_ADDR tilingGm)
{
    TPipe pipe;
    SswapAIV op;
    op.Init(x, y, tilingGm, &pipe);
    op.Process();
}

几个关键标记：

  • __global__ __aicore__：标记这是一个 NPU 核函数，类似 CUDA 的 __global__。
  • GM_ADDR：全局内存地址，指向 NPU 显存中的数据。类似 CUDA 中的设备指针。
  • TPipe：流水线管理对象，负责协调数据搬运和计算的并行。

sswap 的核心逻辑非常直观：从 GM 把 x 和 y 的数据分别搬到 UB 的两块缓冲区，然后交叉搬回——把 x 缓冲区的数据写回 y 的 GM 位置，把 y 缓冲区的数据写回 x 的 GM 位置。这样就完成了交换。

TilingData 结构体

struct SswapTilingData {
    uint32_t totalN;
    uint32_t perCoreN;
    uint32_t remainder;
    uint32_t tileSize;
};

constexpr uint32_t ELEMENTS_PER_BLOCK = 32 / sizeof(float);
  • totalN：向量总元素个数。
  • perCoreN：每个 AI Core 负责处理的元素个数（已对齐到 32B 边界）。
  • remainder：除不尽的尾巴元素，交给最后一个 Core 处理。
  • tileSize：每个 Core 内部的分块大小（UB 一次能装多少元素）。
  • ELEMENTS_PER_BLOCK：32 字节对齐粒度，对于 FP32 就是 8 个 float。

1.3.4 编译

使用 ops-blas 的统一构建脚本编译 sswap 算子：

bash build.sh --ops=sswap --soc=ascend950

参数说明：

  • --ops=sswap：指定要编译的算子名称。
  • --soc=ascend950：指定目标芯片为 A5（Ascend 950）。

如果你用的是 A2/A3 芯片，把 ascend950 替换为对应的型号（如 ascend910b）。

编译成功后，你会在输出目录中看到生成的 Kernel 二进制文件。如果编译过程中报错，检查 CANN 环境是否已正确 source。

1.3.5 运行测试

编译通过后，运行 ST（System Test）验证算子功能：

bash build.sh --ops=sswap --soc=ascend950 --run

你应该能看到类似这样的输出：

[==========] Running tests from sswap
[ RUN      ] sswap/SSwapTest.Check/0
[       OK ] sswap/SSwapTest.Check/0
[==========] All tests passed

测试做了什么？它在 CPU 上生成两组随机向量 x 和 y，用 CPU golden 逻辑算出交换后的期望结果，然后调用 NPU 上的 sswap 算子执行交换，最后逐元素比对 NPU 输出和 golden 结果。因为 sswap 是纯数据搬运，不涉及数值计算，所以精度要求是位精确（bit-exact）——NPU 输出必须和 golden 完全一致，一个 bit 都不能差。

1.3.6 用仿真器运行（无 NPU 硬件时）

如果你手边没有 NPU 硬件，可以使用 CANN 仿真器模式。仿真器在 CPU 上模拟 NPU 的行为，可以验证算子的功能正确性（但性能数据不具参考性）。

注意：ops-blas 的 build.sh 不直接支持仿真器参数。仿真器模式需要单独配置 CANN Simulator 环境，请参考 CANN 官方文档 了解仿真器的安装和使用方式。

1.3.7 恭喜你

你已经跑通了第一个 Ascend C 算子！虽然代码细节还没完全展开，但你已经掌握了几个核心概念：

  • Kernel 代码在 NPU 上执行，Host 代码在 CPU 上执行——这是 Ascend C（以及 CUDA）的基本分工。
  • 数据通过 GM_ADDR 在 CPU 和 NPU 之间传递——Host 侧准备好数据地址，Kernel 侧通过 GM_ADDR 访问显存。
  • Tiling 参数决定每个核处理多少数据——这是 Ascend C 特有的设计，让同一个算子可以灵活地适配不同大小的输入。
  • 编译时需要指定目标芯片——不同芯片的 AI Core 架构不同，编译出的二进制也不同。

下一章我们将深入了解 A5 的硬件架构，理解 Kernel 代码里那些操作（DataCopy、EnQue、DeQue）到底在操作什么硬件单元。
