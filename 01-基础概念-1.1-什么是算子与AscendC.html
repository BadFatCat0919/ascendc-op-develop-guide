1.1.1 算子是什么

想象你在雪王奶茶打工。一杯"满杯百香果"从点单到出餐，要经过好几道工序：煮茶底、加果酱、加冰、封口。每道工序就是一个算子（Operator）——它接收上一步的半成品（数据），加工一下，传给下一步。你写的 PyTorch 模型就是那张菜单，框架就是店里的 POS 系统，而算子就是每一道实际动手的工序。torch.add(a, b) 就是"加两勺糖"，torch.matmul(A, B) 就是"把茶底和果酱摇匀"。

用更正式的话说：算子是对张量（多维数组）执行的一种数学运算，它是深度学习框架中最小的计算单元。你在 PyTorch 里写 torch.add(a, b)，这个 add 就是一个算子；写 torch.matmul(A, B)，这个 matmul 也是一个算子。

算子 vs 函数 vs 层

这三个词经常混用，但它们的视角不同：

概念视角举例
函数编程概念add(a, b) 是一个 C++ 函数
层（Layer）网络结构概念"卷积层"是网络架构图里的一个方框
算子（Operator）硬件执行概念Conv2D 算子是 NPU 上实际运行的一段 Kernel 代码

一个"层"往往包含多个算子。比如一个典型的"卷积层"实际上由三个算子串联而成：Conv2D（卷积计算）→ BiasAdd（加偏置）→ ReLU（激活函数）。框架在调度时，会把它们拆成独立的算子节点依次执行。

算子在框架中的角色

当你在 PyTorch 或 MindSpore 里写模型代码时，框架会在内部构建一张计算图（Compute Graph）——图中的每个节点就是一个算子，边代表数据的流向。

  • 前向传播：框架按拓扑序依次执行每个算子，从输入数据算出预测结果。
  • 反向传播：框架执行每个算子的"梯度算子"，从损失函数倒推出每个参数的梯度。

整个调用链条从上到下大致是这样的：

用户代码 (PyTorch / MindSpore)
    ↓
框架算子调度层
    ↓
算子接口层（Host 侧，运行在 CPU 上）
    ↓
Tiling（数据切分：决定每个 NPU 核处理哪些数据）
    ↓
Kernel（NPU 上实际执行的核函数）

算子开发者的工作，就是实现最底层那个 Kernel——让它在 NPU 上高效地跑起来。上面的几层（框架调度、接口封装）由 CANN 和框架团队负责，你只需要关注"数据怎么切分"和"核函数怎么写"。

常见算子分类

按照计算特征，算子可以分为以下几大类：

类别典型算子主要计算单元特点
逐元素（Element-wise）Add、Mul、ReLUVector最简单，每个元素独立计算，互不依赖
归约（Reduction）Sum、Max、SoftmaxVector沿某个轴做聚合，比如对一行求和
矩阵乘（MatMul）GEMM、GEMVCube计算密集型，算力是瓶颈
卷积（Convolution）Conv2DCube本质也是矩阵运算（im2col + GEMM）
数据搬运（Data Move）Transpose、Gather、SwapMTE不涉及数值计算，搬运效率是关键

以 sswap 为例：sswap 是 BLAS Level-1 中的一个数据搬运算子，功能是交换两个向量的对应元素——swap(x, y) 执行后，x 变成原来的 y，y 变成原来的 x。它不做任何加减乘除，纯粹是把数据从 A 搬到 B、从 B 搬到 A。对于这类算子，关键不在于"算什么"，而在于"怎么搬得更快"。

为什么需要"开发算子"

框架自带的算子库已经很丰富了，为什么还要自己写算子？主要有三个原因：

  1. 框架没覆盖：你需要的算子（比如 FlashAttention 这种融合算子）框架里还没有，得自己实现。
  2. 性能优化：通用算子为了兼容各种场景，不一定是最快的。针对特定硬件（如 A5 芯片）和特定形状（如固定 batch size）做定制优化，可以快很多。
  3. 算子融合：把多个小算子融合成一个大算子，减少数据在显存和片上缓存之间的来回搬运，显著提升效率。

1.1.2 Ascend C 是什么

Ascend C 是华为昇腾 NPU 上的算子开发框架——它提供了一套 C++ API，让你可以用接近写普通 C++ 的方式来编写运行在 NPU 上的核函数（Kernel）。

类比一下：就像 CUDA 之于 NVIDIA GPU，Ascend C 之于昇腾 NPU。如果你之前接触过 CUDA 编程，会发现两者的思路非常相似——都是"在 CPU 上准备数据、启动核函数、回收结果，在加速器上执行计算密集的部分"。

与 CUDA 的类比

概念CUDA (NVIDIA GPU)Ascend C (昇腾 NPU)
硬件计算单元GPU SM（流多处理器）AI Core
编程语言CUDA C/C++Ascend C (C++)
软件栈CUDA ToolkitCANN Toolkit
核函数标记__global____global__ __aicore__
核索引threadIdx / blockIdxblock_idx
片上共享内存Shared MemoryUnified Buffer (UB)
全局内存Global MemoryGlobal Memory (GM)

两者的核心区别在于硬件架构：GPU 的 SM 擅长大量线程并行做简单运算；昇腾 NPU 的 AI Core 则集成了专用的 Cube（矩阵乘）和 Vector（向量）计算单元，对矩阵运算有硬件级加速。

CANN 软件栈

CANN 全称 Compute Architecture for Neural Networks，是昇腾的统一计算架构。你可以把它理解为"NPU 世界的操作系统 + 开发工具包"。

从用户代码到硬件，整个软件栈的分层如下：

┌─────────────────────────────────────┐
│  应用框架 (PyTorch / MindSpore)      │
├─────────────────────────────────────┤
│  算子接口层 (aclnn / 直调)           │
├─────────────────────────────────────┤
│  Ascend C 算子库                    │
│  (ops-blas / ops-nn / ops-math ...) │
├─────────────────────────────────────┤
│  opbase 基础框架                    │
├─────────────────────────────────────┤
│  CANN Toolkit (编译器/运行时/工具链) │
├─────────────────────────────────────┤
│  昇腾 NPU 硬件                      │
└─────────────────────────────────────┘
  • 应用框架层：PyTorch、MindSpore 这些你熟悉的框架，它们通过算子接口层调用底层算子。
  • 算子接口层：提供统一的 C 接口（aclnn 前缀），框架通过它来调用具体的算子实现。
  • Ascend C 算子库：这就是我们要开发算子的地方。算子库按功能域分成多个仓库。
  • opbase 基础框架：所有算子仓库共同依赖的底层库，提供公共头文件、调度框架、内存管理等基础设施。
  • CANN Toolkit：编译器、运行时库和各种调试工具。
  • NPU 硬件：最终执行计算的芯片。

算子仓库

算子按功能域分成不同的代码仓库：

仓库功能域典型算子
ops-blas线性代数（BLAS）sswap、sgemm、saxpy
ops-nn神经网络Conv2D、BatchNorm
ops-math数学运算Exp、Log、Sqrt
ops-cv计算机视觉Resize、Crop
ops-transformerTransformer 专用FlashAttention

sswap 属于 ops-blas 仓，因为它是 BLAS（Basic Linear Algebra Subprograms）标准中 Level-1 的一个 routine——BLAS Level-1 处理的都是向量与向量之间的操作。

opbase 是所有算子仓库的公共依赖，提供公共头文件、调度框架等基础设施，你可以把它理解为算子仓库的"标准库"。

1.1.3 算子开发流程概览

一个算子从零到上线，大致经历以下环节：

需求分析 → Tiling 设计 → Kernel 编码 → 编译 → 仿真验证 → 上板测试 → 精度/性能验收 → 上库
  1. 需求分析：明确算子的接口签名、支持的数据类型、精度要求等。
  2. Tiling 设计：决定数据怎么切分给多个 AI Core、每个 Core 的片上缓存怎么分配。这是 Ascend C 开发中最关键的设计环节。
  3. Kernel 编码：用 Ascend C API 实现核函数，包括数据搬运和计算逻辑。
  4. 编译：使用 CANN 编译器将代码编译成 NPU 可执行的二进制。
  5. 仿真验证：在没有 NPU 硬件的情况下，用仿真器验证功能正确性。
  6. 上板测试：在真实 NPU 上运行 ST（System Test）用例，与 golden 数据比对。
  7. 精度/性能验收：确认精度达标、性能满足要求。
  8. 上库：代码合入主分支。

后续章节会逐步展开每个环节的细节。下一章我们先认识数据类型和排布格式——这是写 Kernel 代码之前必须搞清楚的基础知识。然后我们立刻动手，跑通第一个真正的算子。
