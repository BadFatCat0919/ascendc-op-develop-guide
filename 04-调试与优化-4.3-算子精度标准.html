4.3.1 精度标准概述

算子精度标准不是"一刀切"的，而是根据算子类型和使用场景分为两套体系：

商用标准（更严格）— 适用于标准工程开发的算子，采用双标杆比对（NPU 结果同时与 CPU golden 和三方芯片结果比较），用 Ratio 指标评估。这套标准分为 L0/L1/L2 三个精度等级，L2 最严格，适用于核心业务算子。

社区标准（相对宽松）— 适用于简易工程开发的生态算子，采用单标杆比对（只与 CPU golden 比较），用 MERE/MARE 阈值评估。

sswap 的例子：sswap 属于 非计算类算子——它只做数据交换，不涉及任何数值计算。对于这类算子，精度标准最简单也最严格：二进制一致（Bitwise Match），即 NPU 输出必须和 CPU golden 的每一个 bit 都完全相同。没有"差不多就行"的空间。

4.3.2 精度比对公式

对于浮点计算类算子，精度比对使用以下公式：

对于每个输出元素：
|actual - expected| <= atol + rtol * |expected|

这个公式由两部分组成：

  • atol（绝对误差容限）：不管期望值多大，至少允许这么大的偏差
  • rtol * |expected|（相对误差容限）：期望值越大，允许的绝对偏差也越大

两者取和，意味着对于很小的期望值，主要看 atol；对于很大的期望值，主要看 rtol。

对于非计算类算子（如 sswap），atol = 0，rtol = 0，即要求完全一致。

4.3.3 非计算类精度标准

非计算类算子包括数据搬运（Copy、Swap）、类型转换（Cast）等。它们的共同特点是：不涉及数值计算，输出应该是输入的精确副本或确定性变换。

通过标准：二进制一致（Bitwise Match）

is_pass = np.array_equal(npu_output, golden_output)

只要有一个 bit 不同，就算失败。

sswap 就是这种情况：交换两个向量的元素，不应该改变任何一个 bit。所以 sswap 的 ST 测试使用了 PrecisionMode::EXACT 模式：

>

```cpp

VerifyConfig cfg;

cfg.mode = PrecisionMode::EXACT;

EXPECT_TRUE(Verifier::verifyVector(xPtr, goldenX.data(), ...));

EXPECT_TRUE(Verifier::verifyVector(yPtr, goldenY.data(), ...));

```

4.3.4 浮点计算类精度标准

浮点计算类算子是最常见的类型，精度标准根据数据类型不同而不同。

社区标准（单标杆 Threshold）

数据类型MERE ThresholdMARE Threshold（= 10 × MERE）
FP322^-13 ≈ 0.0001222^-13 × 10 ≈ 0.00122
FP162^-10 ≈ 0.0009772^-10 × 10 ≈ 0.00977
BF162^-7 ≈ 0.007812^-7 × 10 ≈ 0.0781
HiFP322^-11 ≈ 0.0004882^-11 × 10 ≈ 0.00488
FP8 E4M32^-3 ≈ 0.1252^-3 × 10 ≈ 1.25
FP8 E5M22^-2 ≈ 0.252^-2 × 10 ≈ 2.5

判定条件：MERE < Threshold 且 MARE < 10 × Threshold，两项同时满足才算通过。

其中误差指标定义为：

  • MERE（平均相对误差）= avg(|actual - golden| / (|golden| + 1e-7))
  • MARE（最大相对误差）= max(|actual - golden| / (|golden| + 1e-7))

商用标准（双标杆 Ratio）

商用标准更严格，需要同时与 CPU golden 和三方芯片结果比较，计算 Ratio：

精度等级适用场景用例规模MARE RatioMERE RatioRMSE Ratio
L0常规算子≥5,000≤10≤2≤2
L1重要算子（LLM/多模态）≥10,000≤5≤1.5≤1.5
L2关键算子（核心业务）≥30,000≤2≤1.2≤1.2

Ratio = NPU 误差指标 / 三方芯片误差指标。Ratio ≤ 阈值意味着 NPU 的精度不能比三方芯片差太多。

4.3.5 整数计算类精度标准

整型计算（如整数加减、位运算）通常要求精确匹配：atol = 0，rtol = 0。

整数运算没有舍入误差，结果应该是确定性的。如果整型算子的输出和 golden 不一致，那一定是代码有 bug，而不是精度问题。

4.3.6 量化计算类精度标准

涉及整型与浮点之间转换的算子（如量化、反量化），精度标准更复杂。因为量化本身就会引入误差（把连续的浮点值映射到离散的整型值），所以需要考虑量化误差的累积。

具体标准取决于输入输出的数据类型组合和量化策略，请参考 opbase 精度标准文档中的量化计算类章节。

4.3.7 特殊场景处理

小值域

当 golden 值接近 0 时，相对误差的计算会变得不稳定（分母太小，一点偏差就会被放大）。此时需要启用小值域标准：

数据类型小值阈值误差阈值
FP322^-142^-30
FP162^-112^-16
BF162^-82^-16
HiFP322^-122^-28
FP8 E4M32^-42^-6
FP8 E5M22^-32^-5

小值域标准使用 ErrorCount（超出误差阈值的元素个数）来评估，而不是相对误差。

INF/NAN

当输出中包含 INF 或 NAN 时，需要特殊处理：

  • 如果 golden 是 INF/NAN，NPU 输出也应该是对应的 INF/NAN
  • 不同芯片架构对 INF 的处理可能不同（Ascend 910A 及之前的芯片，INF 不参与精度比较）

精度复检

当单个用例不满足通过标准时，不要急着判定失败。由于 NPU 执行存在一定的随机性，可以启动复检流程：更换随机种子运行多次（推荐 1000 次），用 Bootstrap 方法计算 95% 置信区间，如果置信区间下限 > 1.0 才判定为真正失败。

4.3.8 Golden 构造方法

Golden 就是 CPU 上的参考实现，是判断 NPU 输出是否正确的"标准答案"。构造一个好的 Golden，是精度验证的基础。

构造原则

  1. 使用更高精度：Golden 的计算精度应该高于被测算子。比如被测的是 FP16 算子，Golden 至少用 FP32 计算。
  2. 避免同源错误：不要使用与被测算子相同的库来实现 Golden，否则可能两者犯同样的错误，测试就失去意义了。
  3. 覆盖边界值：对 0、INF、NAN、极大/极小值单独验证。

标杆选择优先级

优先级标杆类型适用场景
1业界 CPU / 三方芯片同等功能算子标准算子（如 BLAS）
2小算子拼接组合实现融合算子
3自行构造的 CPU 实现非标准数据类型

sswap 的 Golden 实现

sswap 的 Golden 非常简单直接——因为它只是交换元素，CPU 实现只需要一个 for 循环：

inline aclblasStatus_t aclblasSswap_cpu(
    aclblasHandle_t handle, int n, float* x, int incx, float* y, int incy)
{
    if (handle == nullptr) return ACLBLAS_STATUS_HANDLE_IS_NULLPTR;
    if (n <= 0) return ACLBLAS_STATUS_SUCCESS;
    if (x == nullptr || y == nullptr) return ACLBLAS_STATUS_INVALID_VALUE;
    if (incx == 0 || incy == 0) return ACLBLAS_STATUS_INVALID_VALUE;
    if (incx < 0 || incy < 0) return ACLBLAS_STATUS_INVALID_VALUE;
    for (int i = 0; i < n; i++) {
        std::swap(x[i * incx], y[i * incy]);
    }
    return ACLBLAS_STATUS_SUCCESS;
}

这个实现有几个值得注意的地方：

  1. 参数校验先行：先检查 handle、n、指针、inc 参数的合法性，和 NPU 侧的参数校验逻辑保持一致。
  2. 支持 stride：通过 incx 和 incy 支持非连续存储的向量，这是 BLAS 标准的要求。
  3. 使用 std::swap：标准库的 swap 保证位精确交换，不会引入任何精度损失。

对于 BLAS 标准算子，也可以直接使用 OpenBLAS 或 MKL 作为 Golden，它们的实现经过了大量验证，可靠性更高。
