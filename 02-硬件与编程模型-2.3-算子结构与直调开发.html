2.3.1 算子的两层结构

一个 Ascend C 算子由两层代码组成：Host 侧运行在 CPU 上，Kernel 侧运行在 NPU 的 AI Core 上。

可以用丑团外卖来理解：Host 是你在 APP 上下单（准备参数、选好地址、确认支付），Kernel 是骑手（接到单后去店里取餐、送到你手上）。你不需要知道骑手走哪条路，只需要把订单信息准备好就行。

┌─────────────────────────────────────┐
│  Host 侧代码 (CPU 上运行)           │
│  - 准备输入数据                      │
│  - 计算 Tiling 策略（怎么切分数据）   │
│  - 启动 Kernel                      │
├─────────────────────────────────────┤
│  Kernel 侧代码 (NPU AI Core 上运行) │
│  - 接收 Tiling 参数和 GM 地址        │
│  - 从 GM 搬数据到本地 Buffer         │
│  - 执行计算                          │
│  - 将结果搬回 GM                    │
└─────────────────────────────────────┘

在实际工程中，这两层代码通常分别放在不同的文件里。以一个典型的算子为例，文件结构如下：

算子目录/
└── arch35/
    ├── xxx_host.cpp          # Host 侧：参数校验、Tiling 计算、启动 Kernel
    ├── xxx_kernel.cpp        # Kernel 侧：AI Core 上的计算逻辑
    └── xxx_tiling_data.h     # TilingData 结构体（Host 和 Kernel 共用）

xxx_tiling_data.h 是 Host 和 Kernel 之间的"通信协议"——Host 侧把计算好的 Tiling 参数填入这个结构体，Kernel 侧从 GM 中读取并解析它。

2.3.2 Kernel 函数

Kernel 是运行在 AI Core 上的核函数，由 Host 侧通过 <<<>>> 语法启动。

Kernel 函数签名

extern "C" __global__ __aicore__ void my_op_kernel(
    GM_ADDR x,          // 输入数据的 GM 地址
    GM_ADDR y,          // 输出数据的 GM 地址
    GM_ADDR workspace,  // 工作空间
    GM_ADDR tiling      // Tiling 参数
)

这个签名中有几个关键修饰符：

  • extern "C" — 使用 C 链接，防止 C++ 的 name mangling（名称修饰）。如果不加，C++ 编译器会把函数名改成一串包含参数类型的编码，导致 Host 侧找不到这个函数。
  • __global__ — 标记这是一个可以从 Host 侧启动的全局核函数。
  • __aicore__ — 标记这个函数运行在 AI Core 上，编译器会用昇腾的指令集来编译它。
  • GM_ADDR — 全局内存地址指针，指向 HBM 上的数据。Kernel 通过它来访问 Host 侧传入的输入、输出、工作空间和 Tiling 参数。

一个典型的 Kernel 签名如下：

extern "C" __global__ __aicore__ void my_kernel(
    GM_ADDR x,
    GM_ADDR y,
    GM_ADDR workSpace,
    GM_ADDR tilingGm)
{
    KERNEL_TASK_TYPE_DEFAULT(KERNEL_TYPE_AIV_ONLY);
    MyOpAIV op;
    op.Init(x, y, tilingGm);
    op.Process();
}

KERNEL_TASK_TYPE_DEFAULT(KERNEL_TYPE_AIV_ONLY) 告诉硬件只启动 AIV（AI Vector）核，不启动 AIC（AI Cube）核。这是因为纯数据搬运算子不需要矩阵乘能力。MyOpAIV 是算子的核心类，Init 负责初始化（解析 Tiling、设置 GM 地址、分配 Buffer），Process 负责执行计算逻辑。

Kernel 内部流程

一个典型的 Kernel 内部流程如下：

{
    // 1. 获取当前核编号
    auto blockIdx = AscendC::GetBlockIdx();

    // 2. 解析 Tiling 数据（使用框架宏）
    GET_TILING_DATA(tilingData, tiling);

    // 3. 计算当前核负责的数据范围
    uint32_t offset = blockIdx * tilingData.tileSize;

    // 4. 搬入 → 计算 → 搬出
    // ...（后续章节详细讲解）
}

上面的 GET_TILING_DATA(tilingData, tiling) 是使用框架宏的写法，适用于 aclnn 框架托管的算子。直调方式则通过手动 reinterpret_cast 解析 Tiling 数据：

```cpp

__gm__ SswapTilingData tilingPtr = reinterpret_cast<__gm__ SswapTilingData>(tilingGm);

tiling_.totalN = tilingPtr->totalN;

```

每个 AI Core 通过 GetBlockIdx() 知道自己的编号，再结合 Tiling 数据算出自己该处理哪段数据。这就是 SPMD 模型在 Kernel 内部的具体体现。

2.3.3 Tiling 函数

Tiling 是数据切分策略——决定总共多少数据、分给多少个核、每个核处理多少。它在 Host 侧计算，通过 GM 传递给 Device 侧。

你可以把 Tiling 理解为雪王奶茶的分单策略：公司团建点了 totalN 杯奶茶，有 coreNum 家门店来分（核数），每家分 perCoreN 杯，最后可能剩下 remainder 杯分给某几家门店多喝几杯。

Tiling 数据结构

一个典型的 TilingData 结构体定义如下：

struct SswapTilingData {
    uint32_t totalN;     // 向量总元素数
    uint32_t perCoreN;   // 每个核处理的基础元素数
    uint32_t remainder;  // 余数（totalN % coreNum）
    uint32_t tileSize;   // 每次迭代搬运的元素数（受 UB 容量限制）
};

同时定义了一个常量：

constexpr uint32_t ELEMENTS_PER_BLOCK = 32 / sizeof(float);

ELEMENTS_PER_BLOCK 是 32 Bytes 对齐要求下的最小搬运单位——对于 float（4 Bytes），每次至少搬 8 个元素。

Tiling 函数

Tiling 函数在 Host 侧执行，根据输入规模和硬件资源计算切分方案。下面是一个 aclnn 框架下的 Tiling 函数示例（通过框架提供的 context 对象获取核数和输入 shape）：

// aclnn 框架下的 Tiling 函数示例
void my_op_tiling(const MyOpTilingContext* context, MyOpTilingData* tilingData)
{
    uint32_t coreNum = context->GetCoreNumAiv();
    tilingData->totalLength = context->GetInputShape(0)[0];
    tilingData->tileSize = ceil(tilingData->totalLength / coreNum);
    tilingData->tileNum = coreNum;
}

直调方式不依赖 aclnn 框架的 Tiling 上下文，而是通过自定义函数直接获取核数，由 Host 侧代码自行计算所有 Tiling 参数。

在直调方式的 Host 侧代码中，Tiling 计算逻辑大致如下：先获取可用的 Vector 核数，然后将 n 个元素均分给各个核（perCoreN = n / coreNum），余数 remainder = n % coreNum 分配给前几个核多处理一个元素。tileSize 则根据 UB 容量计算，确保每次迭代的数据量不超过 UB 的承载能力。

2.3.4 直调方式：最简单的开发入口

直调（Direct Call）是最简单的算子调用方式——直接从 Host 侧调用 Kernel 函数，不经过 aclnn 框架的封装。

直调适合学习和快速验证。你不需要理解 aclnn 的两段式接口（先获取 workspace 大小、再执行计算），只需要准备好数据、算好 Tiling、启动 Kernel 就行。生产环境最终需要对接 aclnn 框架，但入门时先掌握直调可以让你更快理解算子的运行机制。

直调 vs aclnn

维度直调aclnn
复杂度低高
灵活性高中
框架对接需要自己处理框架自动调度
适用场景学习、验证生产环境

直调的 Host 侧代码

一个典型的 Host 侧入口函数整体流程如下：

#include "acl/acl.h"

int main() {
    // 1. 初始化 ACL
    aclInit(nullptr);
    aclrtSetDevice(0);

    // 2. 准备输入数据（在 GM 上分配内存）
    size_t size = n * sizeof(float);
    void* devX = nullptr;
    void* devY = nullptr;
    aclrtMalloc(&devX, size, ACL_MEM_MALLOC_HUGE_FIRST);
    aclrtMalloc(&devY, size, ACL_MEM_MALLOC_HUGE_FIRST);

    // 3. 拷贝数据到 GM
    aclrtMemcpy(devX, size, hostX, size, ACL_MEMCPY_HOST_TO_DEVICE);

    // 4. 启动 Kernel
    my_op_kernel<<<coreNum, stream>>>(devX, devY, workspace, tiling);

    // 5. 拷贝结果回 CPU
    aclrtMemcpy(hostY, size, devY, size, ACL_MEMCPY_DEVICE_TO_HOST);

    // 6. 清理
    aclrtFree(devX);
    aclrtFree(devY);
    aclrtResetDevice(0);
    aclFinalize();
}

流程非常直观：初始化设备 → 分配 GM 内存 → 拷入数据 → 启动 Kernel → 拷出结果 → 清理资源。<<>> 是 Ascend C 的 Kernel 启动语法，coreNum 指定启动多少个 AI Core，stream 是执行流。

Host 侧通常还会封装一个 C 风格的启动函数：

void my_kernel_do(GM_ADDR x, GM_ADDR y, GM_ADDR workSpace,
                  GM_ADDR tilingGm, uint32_t numBlocks, void* stream)
{
    my_kernel<<<numBlocks, nullptr, stream>>>(x, y, nullptr, tilingGm);
}

这个函数把 Kernel 启动的细节封装起来，Host 侧的 aclblasSswap 只需要传入参数和核数就可以启动算子。

2.3.5 完整算子文件结构

在 ops-blas 仓中，算子按家族（family）、算子名（operator）和架构（arch）组织：

blas/{family}/
└── {operator_name}/
    └── archXX/
        ├── {operator_name}_host.cpp       # Host 侧：参数校验、Tiling、直调 Kernel
        ├── {operator_name}_kernel.cpp     # Kernel 侧：AscendC 类 + Kernel 入口
        └── {operator_name}_tiling_data.h  # TilingData 结构体（Host/Kernel 共用）

典型的文件结构如下：

算子目录/
└── arch35/
    ├── xxx_host.cpp
    ├── xxx_kernel.cpp
    └── xxx_tiling_data.h

三个文件各司其职：tiling_data.h 定义通信协议，kernel.cpp 实现 AI Core 上的计算逻辑，host.cpp 负责参数校验、Tiling 计算和 Kernel 启动。

2.3.6 动手试试：写一个最简 Kernel

这是 Tutorial 部分，让我们跟着写一个最简单的 Kernel 骨架，把前面讲的概念串起来。

// kernel/add_kernel.cpp
#include "kernel_operator.h"

extern "C" __global__ __aicore__ void add_kernel(
    GM_ADDR a, GM_ADDR b, GM_ADDR c,
    GM_ADDR workspace, GM_ADDR tiling)
{
    // 获取 tiling 数据
    GET_TILING_DATA(tilingData, tiling);
    auto blockIdx = AscendC::GetBlockIdx();

    // 计算当前核的数据范围
    uint32_t offset = blockIdx * tilingData.tileSize;
    uint32_t length = tilingData.tileSize;

    // 声明 UB Buffer
    AscendC::TPipe pipe;
    AscendC::TQue<AscendC::QuePosition::VECIN, 1> queIn;
    AscendC::TQue<AscendC::QuePosition::VECOUT, 1> queOut;

    // 初始化 Buffer
    pipe.InitBuffer(queIn, 2, length * sizeof(float));
    pipe.InitBuffer(queOut, 1, length * sizeof(float));

    // 搬入
    auto inA = queIn.AllocTensor<float>();
    auto inB = queIn.AllocTensor<float>();
    AscendC::DataCopy(inA, a + offset, length);
    AscendC::DataCopy(inB, b + offset, length);
    queIn.EnQue(inA);
    queIn.EnQue(inB);

    // 计算
    auto valA = queIn.DeQue<float>();
    auto valB = queIn.DeQue<float>();
    auto valC = queOut.AllocTensor<float>();
    AscendC::Add(valC, valA, valB, length);
    queOut.EnQue(valC);

    // 搬出
    auto outC = queOut.DeQue<float>();
    AscendC::DataCopy(c + offset, outC, length);

    // 释放
    queIn.FreeTensor(valA);
    queIn.FreeTensor(valB);
    queOut.FreeTensor(outC);
}

这段代码展示了 Kernel 的基本骨架：搬入 → 计算 → 搬出。

TPipe 是流水线管理器，负责分配和管理 UB 空间。TQue 是队列，用来组织 UB 中的 Buffer——VECIN 队列存放搬入的数据，VECOUT 队列存放待搬出的结果。AllocTensor 从队列中分配一块 Buffer，EnQue 将填好数据的 Buffer 入队（表示"我写完了，可以给别人用了"），DeQue 从队列中取出一块 Buffer（表示"我要开始读了"），FreeTensor 释放 Buffer。

这套 EnQue/DeQue 机制实现了流水线的同步：搬入线程把数据 EnQue 到 VECIN，计算线程从 VECIN DeQue 出来做计算，算完 EnQue 到 VECOUT，搬出线程再从 VECOUT DeQue 搬回 GM。

swap 算子的 Kernel 结构与此类似，但更简单——它没有"计算"步骤，只有"搬入"和"搬出"。swap 使用同一块 Buffer 既用于搬入也用于搬出。swap 的巧妙之处在于交叉写回：

>

```

CopyIn: xGM → inQueueX (UB) CopyOut: inQueueX (UB) → yGM (x 的数据写到 y 的位置)

yGM → inQueueY (UB) inQueueY (UB) → xGM (y 的数据写到 x 的位置)

```

>

搬入时，x 的数据进入 inQueueX_，y 的数据进入 inQueueY_。搬出时，inQueueX_ 的内容写到 yGM（y 的 GM 地址），inQueueY_ 的内容写到 xGM（x 的 GM 地址）。这样就完成了两个向量的元素交换，全程不需要任何算术运算。

现在你只需要理解整体流程，不用纠结每个 API 的细节。后续章节会逐一讲解 TPipe、TQue、DataCopy 等 API 的详细用法。
