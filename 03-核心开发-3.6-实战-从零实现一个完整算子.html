3.6.1 实战目标

本文以 ops-blas 仓中的 sswap 算子（arch35） 为例，完整走通一个 Ascend C 算子的开发全流程。

sswap 是 BLAS Level-1 接口，功能是交换两个向量的对应元素：

x[i] ↔ y[i]，对所有 i ∈ [0, N)

它的计算特征非常鲜明：纯数据搬运，不涉及任何算术运算。从 GM 把 x 和 y 搬到 UB，然后把 x 的内容写回 y 的 GM 地址、y 的内容写回 x 的 GM 地址。正因为逻辑简单，sswap 是理解 Ascend C 算子开发框架的最佳入门案例——你可以把全部注意力集中在 Tiling、搬运、队列、同步这些框架机制上，而不用被复杂的计算逻辑分心。

关键设计选择：

  • 使用 Vector 单元（纯搬运，不用 Cube）
  • 单 Buffer（无计算阶段，双 Buffer 无额外收益）
  • 精度位精确（数据搬运不产生精度误差）
  • 目标架构：arch35（A5 芯片）

3.6.2 需求分析

输入：两个长度为 N 的 FP32 向量 x 和 y，存储在 GM 中。

输出：交换后的 x 和 y（原地修改，不分配额外输出空间）。

计算类型：纯数据搬运。不涉及 Vector 算术运算，不涉及 Cube 矩阵乘。

精度要求：精确。数据搬运不改变任何值，输出必须与输入完全一致（只是 x 和 y 互换了位置）。

硬件目标：A5 (950PR)，arch35。每核 UB 248KB，Vector 核数通过运行时接口获取。

对齐约束：DataCopy 要求搬运长度按 32B 对齐，FP32 下即 8 个元素的倍数。

3.6.3 Tiling 设计

多核切分

sswap 是逐元素算子，每个元素的交换互不依赖，所以按元素数量均分给所有 Vector Core 即可。

切分时需要满足 32B 对齐约束，所以每核处理量要向下对齐到 ELEMENTS_PER_BLOCK = 8：

uint32_t rawPerCore = n / coreNum;
tiling.perCoreN = (rawPerCore / alignUnit) * alignUnit;

对齐后剩余的元素全部交给最后一个核：

tiling.remainder = n - tiling.perCoreN * coreNum;

UB 切分

sswap 需要两个 Buffer（xBuffer 和 yBuffer），UB 空间平分给它们：

constexpr uint32_t ubSize = 248 * 1024;
constexpr uint32_t bufferCount = 2;
uint32_t maxElements = ubSize / (bufferCount * sizeof(float));
tiling.tileSize = (maxElements / alignUnit) * alignUnit;

248KB 平分给两个 Buffer，每个 Buffer 约 124KB，可以装约 31K 个 FP32 元素。对齐到 8 的倍数后得到 tileSize。

Tiling 数据结构

struct SswapTilingData {
    uint32_t totalN;     // 总元素个数
    uint32_t perCoreN;   // 每核处理量（已对齐到 ELEMENTS_PER_BLOCK）
    uint32_t remainder;  // 尾块大小，分配给最后一个核
    uint32_t tileSize;   // UB 单轮处理量（已对齐到 ELEMENTS_PER_BLOCK）
};

四个字段的设计理由：

  • totalN：Kernel 需要知道总长度来设置 SetGlobalBuffer 的边界。
  • perCoreN：每个核的基础工作量，用于计算数据起始偏移。
  • remainder：最后一个核的额外工作量。
  • tileSize：控制内层循环的轮数。

完整的 Tiling 计算函数

static SswapTilingData CalSswapTilingData(uint32_t n, uint32_t coreNum)
{
    SswapTilingData tiling;
    tiling.totalN = n;
    constexpr uint32_t alignUnit = ELEMENTS_PER_BLOCK;
    uint32_t rawPerCore = n / coreNum;
    tiling.perCoreN = (rawPerCore / alignUnit) * alignUnit;
    tiling.remainder = n - tiling.perCoreN * coreNum;
    constexpr uint32_t ubSize = 248 * 1024;
    constexpr uint32_t bufferCount = 2;
    uint32_t maxElements = ubSize / (bufferCount * sizeof(float));
    tiling.tileSize = (maxElements / alignUnit) * alignUnit;
    return tiling;
}

3.6.4 Kernel 编码

Kernel 是运行在 AI Core 上的核函数，负责实际的数据搬运和交换。sswap 的 Kernel 包含三个核心方法：Init（初始化）、SingleIteration（单轮交换）、Process（主循环）。

Kernel 入口

Kernel 入口函数是 __global__ __aicore__ 修饰的全局函数，接收 GM 指针作为参数：

extern "C" __global__ __aicore__ void sswap_kernel(GM_ADDR x, GM_ADDR y, GM_ADDR workSpace, GM_ADDR tilingGm)
{
    KERNEL_TASK_TYPE_DEFAULT(KERNEL_TYPE_AIV_ONLY);
    TPipe pipe;
    SswapAIV op;
    op.Init(x, y, tilingGm, &pipe);
    op.Process();
}

入口函数非常简洁：声明任务类型、创建 Pipe 和 Kernel 类实例，调用 Init 初始化，调用 Process 执行主循环。

  • KERNEL_TASK_TYPE_DEFAULT(KERNEL_TYPE_AIV_ONLY) 声明当前 Kernel 仅使用 AIV（AI Vector）单元，不使用 AIC（AI Cube）单元。sswap 是纯搬运算子，不涉及矩阵乘，所以只需要 AIV。
  • 变量名使用 op 而非 aiV，与 ops-blas 仓内其他算子保持一致的命名风格。

Init：初始化阶段

Init 负责三件事：解析 Tiling 数据、计算当前核的数据范围、初始化 Queue 和 Buffer。

__aicore__ inline void SswapAIV::Init(GM_ADDR x, GM_ADDR y, GM_ADDR tilingGm, TPipe* pipe)
{
    pipe_ = pipe;
    blockIdx_ = GetBlockIdx();
    ParseTilingData(tilingGm);
    myOffset_ = blockIdx_ * tiling_.perCoreN;
    myCount_ = tiling_.perCoreN;
    if (blockIdx_ == GetBlockNum() - 1) {
        myCount_ += tiling_.remainder;
    }
    xGM_.SetGlobalBuffer((__gm__ float*)x, tiling_.totalN);
    yGM_.SetGlobalBuffer((__gm__ float*)y, tiling_.totalN);
    uint32_t bufSize = tiling_.tileSize * sizeof(float);
    pipe_->InitBuffer(inQueueX_, BUFFER_NUM, bufSize);
    pipe_->InitBuffer(inQueueY_, BUFFER_NUM, bufSize);
}

逐行解读：

  1. GetBlockIdx() 获取当前核的编号（0, 1, 2, ...）。
  2. ParseTilingData 从 GM 中读出 Tiling 结构体。
  3. myOffset_ = blockIdx_ * perCoreN 计算当前核负责的数据起始位置。
  4. myCount_ 计算当前核要处理的元素总数。最后一个核额外加上 remainder。
  5. SetGlobalBuffer 把 GM 指针包装成 GlobalTensor，后续可以用偏移量访问。
  6. InitBuffer 为 x 和 y 各初始化一个队列，BUFFER_NUM = 1（单 Buffer）。

队列类型：TQueBind

sswap 使用的队列类型是 TQueBind，而不是普通的 TQue。两者的关键区别：

  • TQue：标准的单方向队列，数据从 MTE2 搬入 UB，经 Vector 计算后，再由 MTE3 从 UB 搬出到 GM。入队/出队各自独立。
  • TQueBind：绑定输入（VECIN）和输出（VECOUT）的共用队列。数据从 MTE2 搬入后，可以直通到 MTE3 搬出，不需要额外的计算阶段介入。

sswap 是纯搬运算子，不涉及任何 Vector 计算，所以使用 TQueBind 让数据从搬入直接流转到搬出，是最自然的选择。队列声明如下：

TQueBind<QuePosition::VECIN, QuePosition::VECOUT, 1> inQueueX_;
TQueBind<QuePosition::VECIN, QuePosition::VECOUT, 1> inQueueY_;

模板参数 的含义：输入端绑定到 VECIN（MTE2 搬入），输出端绑定到 VECOUT（MTE3 搬出），队列深度为 1（单 Buffer）。

SingleIteration：单轮交换

这是 sswap 的核心逻辑：把一轮数据从 GM 搬到 UB，交换后写回 GM。

__aicore__ inline void SswapAIV::SingleIteration(uint32_t curOffset, uint32_t dataCount)
{
    uint32_t alignedCount = (dataCount / ELEMENTS_PER_BLOCK) * ELEMENTS_PER_BLOCK;
    uint32_t tailCount = dataCount - alignedCount;

首先把 dataCount 分成两部分：alignedCount（32B 对齐的主体）和 tailCount（不足 32B 的尾部）。

搬入阶段：

    LocalTensor<float> xLocal = inQueueX_.AllocTensor<float>();
    LocalTensor<float> yLocal = inQueueY_.AllocTensor<float>();
    if (alignedCount > 0) {
        DataCopy(xLocal, xGM_[curOffset], alignedCount);
        DataCopy(yLocal, yGM_[curOffset], alignedCount);
    }
    if (tailCount > 0) {
        uint8_t paddingNum = ELEMENTS_PER_BLOCK - tailCount;
        DataCopyExtParams copyParams{1, static_cast<uint32_t>(tailCount * sizeof(float)), 0, 0, 0};
        DataCopyPadExtParams<float> padParams{true, 0, paddingNum, 0.0f};
        DataCopyPad(xLocal[alignedCount], xGM_[curOffset + alignedCount], copyParams, padParams);
        DataCopyPad(yLocal[alignedCount], yGM_[curOffset + alignedCount], copyParams, padParams);
    }

主体部分用普通 DataCopy 搬运（长度已对齐）。尾部用 DataCopyPad 搬运，自动填充不足 32B 的部分。

入队 + 出队：

    inQueueX_.EnQue<float>(xLocal);
    inQueueY_.EnQue<float>(yLocal);
    LocalTensor<float> xIn = inQueueX_.DeQue<float>();
    LocalTensor<float> yIn = inQueueY_.DeQue<float>();

EnQue 表示"数据已经搬入完毕"，DeQue 表示"我要开始使用这些数据了"。在 A5 上，这对操作隐含了 MTE2→MTE3 的同步语义。

搬出阶段（交叉写回）：

    if (alignedCount > 0) {
        DataCopy(yGM_[curOffset], xIn, alignedCount);
        DataCopy(xGM_[curOffset], yIn, alignedCount);
    }
    if (tailCount > 0) {
        DataCopyExtParams copyParams{1, static_cast<uint32_t>(tailCount * sizeof(float)), 0, 0, 0};
        DataCopyPad(yGM_[curOffset + alignedCount], xIn[alignedCount], copyParams);
        DataCopyPad(xGM_[curOffset + alignedCount], yIn[alignedCount], copyParams);
    }

注意这里的"交叉写回"：x 的内容写到 y 的 GM 地址，y 的内容写到 x 的 GM 地址。这就是 swap 的本质——不需要临时变量，直接交叉赋值。

释放 Buffer：

    inQueueX_.FreeTensor(xIn);
    inQueueY_.FreeTensor(yIn);
}

FreeTensor 归还 Buffer，供下一轮使用。

Process：主循环

Process 控制多轮 SingleIteration 的执行：

__aicore__ inline void SswapAIV::Process()
{
    if (myCount_ == 0) return;
    uint32_t tileLoop = myCount_ / tiling_.tileSize;
    uint32_t tileTail = myCount_ % tiling_.tileSize;
    uint32_t curOffset = myOffset_;
    for (uint32_t i = 0; i < tileLoop; i++) {
        SingleIteration(curOffset, tiling_.tileSize);
        curOffset += tiling_.tileSize;
    }
    if (tileTail > 0) {
        SingleIteration(curOffset, tileTail);
    }
}

逻辑非常清晰：先算完整轮数 tileLoop 和最后一轮的元素数 tileTail。完整轮每轮处理 tileSize 个元素，最后一轮处理剩余的 tileTail 个。

3.6.5 Host 侧代码

Host 侧代码运行在 CPU 上，负责四件事：参数校验、动态获取核数、Tiling 计算、调用 Kernel。

参数校验

Host 侧在调用 Kernel 之前需要校验输入参数：

  • N 必须大于 0
  • x 和 y 的指针不能为空
  • x 和 y 的数据类型必须一致

动态获取核数

Vector Core 的数量不能硬编码，必须通过运行时接口动态获取，以适配不同型号的芯片：

int32_t aivCoreNum = 0;
aclrtGetDeviceInfo(deviceId, ACL_DEV_ATTR_VECTOR_CORE_NUM, &aivCoreNum, sizeof(int32_t));

获取到核数后，还需要处理小 N 场景——当数据量很小时，不需要启动所有核，否则大量核会空跑浪费资源：

uint32_t numBlocks = (totalN < aivCoreNum) ? totalN : aivCoreNum;

这样当 N 小于核数时，只启动 N 个核（每个核处理 1 个对齐块）；当 N 足够大时，才使用全部 Vector Core。

Tiling 计算

Host 侧调用 CalSswapTilingData 计算切分方案（代码见 3.6.3 节），然后通过 aclrtMalloc + aclrtMemcpy 把 Tiling 数据写入 GM，供 Kernel 读取。

Kernel 调用

Host 侧通过 sswap_kernel_do 包装函数调用 Kernel，该函数封装了 <<<>>> 直调语法和 stream 管理：

sswap_kernel_do(xDev, yDev, workspaceDev, tilingDev, numBlocks, stream);

sswap_kernel_do 内部通过 <<<>>> 语法启动 Kernel：

sswap_kernel<<<numBlocks, stream>>>(xDev, yDev, workspaceDev, tilingDev);

其中 numBlocks 是实际使用的 AI Core 数量（经过小 N 处理后的值），stream 是执行流。使用包装函数的好处是可以统一管理 stream 同步、错误处理等逻辑，避免在每个调用点重复编写。

3.6.6 编译

ops-blas 使用统一的构建系统。编译 sswap 算子：

bash build.sh --ops=sswap --soc=ascend950

构建系统会自动处理 arch35 目录下的源文件，生成 Kernel 二进制和 Host 侧动态库。

如果编译报错，常见原因有：

  • 头文件找不到：检查 CANN Toolkit 环境变量是否正确配置。
  • SOC_VERSION 不匹配：确认 --soc 参数与目标芯片一致。
  • 语法错误：Ascend C 的 __gm__、__aicore__ 等关键字需要特定的编译器支持，确保使用 CANN 提供的编译器。

3.6.7 仿真验证

在没有 NPU 硬件的开发环境中，可以使用 CANN Simulator 进行功能验证：

  1. 准备 CPU golden 数据：在 CPU 上实现 sswap 的参考逻辑，生成期望输出。
  2. 运行仿真：用 Simulator 执行 Kernel，得到 NPU 侧的输出。
  3. 对比结果：逐元素比较 NPU 输出和 CPU golden，确认完全一致。

sswap 是精确算子，输出必须与 golden 完全一致（atol = 0, rtol = 0），任何差异都说明实现有 bug。

3.6.8 上板测试

在真实 NPU 上运行 ST（System Test）用例，验证算子的功能正确性：

bash build.sh --ops=sswap --soc=ascend950
# 运行 ST 用例
./build/sswap_test

ST 用例会自动生成多种规模的测试数据（小、中、大、边界），调用算子并与 golden 结果比对。全部通过才算验证成功。

3.6.9 常见问题排查

精度不对：sswap 是精确算子，如果出现精度问题，大概率是 DataCopy 对齐错误或 Buffer 越界。检查 ELEMENTS_PER_BLOCK 的计算是否正确，DataCopyPad 的填充参数是否匹配。详见 4.1 精度不对怎么办。

性能不好：sswap 是纯搬运算子，性能瓶颈在 GM 带宽。检查多核切分是否均匀（remainder 是否过大），tileSize 是否充分利用了 UB 空间。详见 4.2 性能不达标怎么办。

编译报错：检查 CANN 环境变量、SOC_VERSION 配置、头文件路径。常见的 __gm__ 相关报错通常是因为没有使用 CANN 提供的编译器。

运行时崩溃：检查 GM 指针是否为空、N 是否为 0、Tiling 数据是否正确写入 GM。可以通过 plog 日志定位具体的崩溃位置。
