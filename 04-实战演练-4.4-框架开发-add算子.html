4.4.1 add 算子概述

在前面章节中，我们用直调方式跑通了一个简单的 Kernel。但在实际工程中，算子最终要对接 aclnn 框架，才能被 PyTorch 等上层框架调用。本节我们以 add 算子（两个向量逐元素相加）为例，手把手走一遍 aclnn 框架下的完整开发流程。

add 算子的功能：给定两个相同长度的向量 A 和 B，逐元素相加得到输出向量 C。

C[i] = A[i] + B[i],  i = 0, 1, ..., N-1

这是最简单的 Element-wise 算子，计算逻辑只有一行加法。但麻雀虽小五脏俱全——它包含了 aclnn 框架开发的全部环节：文件结构、TilingData 设计、Kernel 实现、Host 侧接口注册、CMakeLists 配置、编译和测试。

4.4.2 aclnn 框架简介

aclnn 是 CANN 提供的算子接口框架，上层框架（如 PyTorch）通过它来调用底层算子。它采用两段式接口设计：

第一段：获取 workspace 大小
  aclnnXxxGetWorkspaceSize(inputDesc, outputDesc, &workspaceSize, &tilingData)
      ↓
第二段：执行算子
  aclnnXxx(workspace, workspaceSize, tilingData, stream)

为什么要分两段？因为有些算子需要额外的临时内存（workspace）来做中间计算。第一段告诉调度器"我需要多大的 workspace"，调度器分配好内存后，第二段才真正执行计算。对于 add 这种简单算子，workspace 大小为 0，但接口形式仍然保留。

两段式接口的命名规则：

接口命名格式作用
第一段aclnnXxxGetWorkspaceSize计算 workspace 大小和 Tiling 数据
第二段aclnnXxx执行算子计算

其中 Xxx 是算子名的大驼峰形式，比如 add 算子对应 aclnnAdd。

4.4.3 文件结构

aclnn 框架下的 add 算子，文件结构和直调方式基本一致，但 Host 侧的实现方式不同：

blas/blas1/
└── add/
    └── arch35/
        ├── add_tiling_data.h     # TilingData 结构体（Host/Kernel 共用）
        ├── add_kernel.cpp        # Kernel 侧：AscendC 类 + Kernel 入口
        └── add_host.cpp          # Host 侧：aclnn 两段式接口实现

三个文件的职责和直调方式相同，区别在于 add_host.cpp 不再直接调用 <<<>>> 启动 Kernel，而是通过 aclnn 框架的宏和接口来注册和调度。

4.4.4 TilingData 设计

TilingData 是 Host 和 Kernel 之间的通信协议。对于 add 算子，我们需要传递以下信息：

// add_tiling_data.h
#ifndef ADD_TILING_DATA_H
#define ADD_TILING_DATA_H

#include <cstdint>

struct AddTilingData {
    uint32_t totalLength;    // 向量总元素数
    uint32_t tileNum;        // 使用的核数
    uint32_t tileSize;       // 每个核处理的元素数
    uint32_t alignSize;      // 对齐后的元素数（32B 对齐）
};

#endif // ADD_TILING_DATA_H

各字段含义：

  • totalLength：输入向量的总长度 N
  • tileNum：实际使用的 AI Core 数量（不超过硬件上限）
  • tileSize：每个核处理的元素数（= ceil(totalLength / tileNum)）
  • alignSize：DataCopy 要求 32B 对齐，alignSize 是 tileSize 向上对齐到 32B 边界的元素数

对于 float（4 字节），32B 对齐意味着每次至少搬 8 个元素（32 / 4 = 8）。

4.4.5 Kernel 实现

Kernel 侧使用 Ascend C 的 Vector 计算 API 实现逐元素加法。整体流程是经典的搬入 → 计算 → 搬出三步走。

// add_kernel.cpp
#include "kernel_operator.h"
#include "add_tiling_data.h"

class AddKernel {
public:
    __aicore__ inline AddKernel() {}

    __aicore__ inline void Init(GM_ADDR a, GM_ADDR b, GM_ADDR c,
                                 GM_ADDR tilingGm)
    {
        // 解析 Tiling 数据
        __gm__ AddTilingData* tilingPtr =
            reinterpret_cast<__gm__ AddTilingData*>(tilingGm);
        totalLength_ = tilingPtr->totalLength;
        tileSize_ = tilingPtr->tileSize;
        alignSize_ = tilingPtr->alignSize;

        // 计算当前核的数据范围
        blockIdx_ = AscendC::GetBlockIdx();
        offset_ = blockIdx_ * tileSize_;

        // 最后一个核可能处理的数据不足 tileSize，需要截断
        processLength_ = tileSize_;
        if (offset_ + tileSize_ > totalLength_) {
            processLength_ = totalLength_ - offset_;
        }

        // 初始化 GM 地址
        aGm_ = reinterpret_cast<__gm__ float*>(a) + offset_;
        bGm_ = reinterpret_cast<__gm__ float*>(b) + offset_;
        cGm_ = reinterpret_cast<__gm__ float*>(c) + offset_;

        // 分配 UB Buffer（双 Buffer 流水线）
        pipe_.InitBuffer(inQueueA_, 2, alignSize_ * sizeof(float));
        pipe_.InitBuffer(inQueueB_, 2, alignSize_ * sizeof(float));
        pipe_.InitBuffer(outQueue_, 1, alignSize_ * sizeof(float));
    }

    __aicore__ inline void Process()
    {
        CopyIn();
        Compute();
        CopyOut();
    }

private:
    __aicore__ inline void CopyIn()
    {
        // 从 GM 搬入数据到 UB
        auto aLocal = inQueueA_.AllocTensor<float>();
        auto bLocal = inQueueB_.AllocTensor<float>();
        AscendC::DataCopy(aLocal, aGm_, alignSize_);
        AscendC::DataCopy(bLocal, bGm_, alignSize_);
        inQueueA_.EnQue(aLocal);
        inQueueB_.EnQue(bLocal);
    }

    __aicore__ inline void Compute()
    {
        // 从队列取出数据，执行加法
        auto aLocal = inQueueA_.DeQue<float>();
        auto bLocal = inQueueB_.DeQue<float>();
        auto cLocal = outQueue_.AllocTensor<float>();

        AscendC::Add(cLocal, aLocal, bLocal, processLength_);

        outQueue_.EnQue(cLocal);
        inQueueA_.FreeTensor(aLocal);
        inQueueB_.FreeTensor(bLocal);
    }

    __aicore__ inline void CopyOut()
    {
        // 将结果从 UB 搬回 GM
        auto cLocal = outQueue_.DeQue<float>();
        AscendC::DataCopy(cGm_, cLocal, processLength_);
        outQueue_.FreeTensor(cLocal);
    }

    // 成员变量
    AscendC::TPipe pipe_;
    AscendC::TQue<AscendC::QuePosition::VECIN, 1> inQueueA_;
    AscendC::TQue<AscendC::QuePosition::VECIN, 1> inQueueB_;
    AscendC::TQue<AscendC::QuePosition::VECOUT, 1> outQueue_;

    __gm__ float* aGm_ = nullptr;
    __gm__ float* bGm_ = nullptr;
    __gm__ float* cGm_ = nullptr;

    uint32_t totalLength_ = 0;
    uint32_t tileSize_ = 0;
    uint32_t alignSize_ = 0;
    uint32_t processLength_ = 0;
    uint32_t blockIdx_ = 0;
    uint32_t offset_ = 0;
};

// Kernel 入口函数
extern "C" __global__ __aicore__ void add_kernel(
    GM_ADDR a, GM_ADDR b, GM_ADDR c,
    GM_ADDR workspace, GM_ADDR tiling)
{
    AddKernel op;
    op.Init(a, b, c, tiling);
    op.Process();
}

代码要点：

  1. Init 阶段：解析 Tiling 数据 → 计算当前核的数据偏移 → 初始化 Buffer。注意最后一个核的数据量可能不足 tileSize，需要截断。
  2. CopyIn：使用 DataCopy 从 GM 搬入 A 和 B 两个向量到 UB。搬入长度使用 alignSize（32B 对齐），即使实际数据不足也要搬对齐长度（多出的部分会被忽略）。
  3. Compute：调用 AscendC::Add 执行逐元素加法，计算长度使用 processLength（实际数据长度）。
  4. CopyOut：将结果搬回 GM，搬出长度同样使用 processLength。

搬入和计算/搬出的长度不同：DataCopy 要求 32B 对齐，所以搬入用 alignSize；但实际计算和搬出只需要 processLength 个元素。这是 add 算子中最容易踩的坑之一。

4.4.6 Host 侧实现

Host 侧负责两件事：计算 Tiling 数据和注册 aclnn 接口。

Tiling 函数

Tiling 函数在 aclnn 框架的上下文中执行，通过 context 获取硬件信息和输入 shape：

// add_host.cpp
#include "add_tiling_data.h"
#include "tiling/tiling_api.h"

// Tiling 函数：计算切分策略
static void AddTilingFunc(const aclnnTensorDesc* inputADesc,
                           const aclnnTensorDesc* inputBDesc,
                           const aclnnTensorDesc* outputCDesc,
                           AddTilingData* tilingData)
{
    // 获取输入长度
    uint32_t totalLength = inputADesc->numElements;

    // 获取可用核数
    uint32_t coreNum = 0;
    aclrtGetCoreNum(&coreNum);

    // 计算每个核处理的元素数
    uint32_t tileSize = (totalLength + coreNum - 1) / coreNum;

    // 32B 对齐
    uint32_t alignSize = ((tileSize * sizeof(float) + 31) / 32) * (32 / sizeof(float));

    // 填充 TilingData
    tilingData->totalLength = totalLength;
    tilingData->tileNum = coreNum;
    tilingData->tileSize = tileSize;
    tilingData->alignSize = alignSize;
}

aclnn 两段式接口注册

通过 opbase 框架提供的宏来注册两段式接口：

// 第一段接口：计算 workspace 大小
ACLNN_API aclnnStatus aclnnAddGetWorkspaceSize(
    const aclnnTensorDesc* inputADesc,
    const aclnnTensorDesc* inputBDesc,
    const aclnnTensorDesc* outputCDesc,
    uint64_t* workspaceSize,
    aclOpExecutor** executor)
{
    // add 算子不需要 workspace
    *workspaceSize = 0;

    // 创建 executor 并注册 Tiling 函数和 Kernel
    auto exec = new aclOpExecutor();
    exec->SetTilingFunc(AddTilingFunc);
    exec->SetKernelFunc("add_kernel");
    exec->SetBlockDim(tilingData.tileNum);
    *executor = exec;

    return ACLNN_SUCCESS;
}

// 第二段接口：执行算子
ACLNN_API aclnnStatus aclnnAdd(
    const aclnnTensorDesc* inputADesc, void* inputA,
    const aclnnTensorDesc* inputBDesc, void* inputB,
    const aclnnTensorDesc* outputCDesc, void* outputC,
    uint64_t workspaceSize, void* workspace,
    aclOpExecutor* executor, aclrtStream stream)
{
    // 框架会自动调用 Tiling 函数计算 TilingData，
    // 然后通过 <<<>>> 启动 Kernel
    return executor->Execute(inputA, inputB, outputC,
                              workspace, stream);
}

上面的代码是简化后的伪代码，实际的 aclnn 接口注册需要使用 opbase 框架提供的宏（如 IMPL_ADD_OP）来完成。具体写法请参考 opbase 文档和仓内已有算子的实现。

4.4.7 CMakeLists 配置

算子需要注册到构建系统中才能被编译。在 ops-blas 仓中，每个算子需要在对应的 CMakeLists.txt 中添加编译配置：

# CMakeLists.txt（简化示例）

# 添加 Kernel 源文件
set(KERNEL_SRC
    ${CMAKE_CURRENT_SOURCE_DIR}/add/arch35/add_kernel.cpp
)

# 添加 Host 源文件
set(HOST_SRC
    ${CMAKE_CURRENT_SOURCE_DIR}/add/arch35/add_host.cpp
)

# 编译 Kernel 二进制
ascendc_library(add_kernel OBJECT ${KERNEL_SRC})
target_include_directories(add_kernel PRIVATE
    ${CMAKE_CURRENT_SOURCE_DIR}/add/arch35
)

# 编译 Host 侧代码
add_library(add_host OBJECT ${HOST_SRC})
target_include_directories(add_host PRIVATE
    ${CMAKE_CURRENT_SOURCE_DIR}/add/arch35
    ${OPBASE_INCLUDE_DIRS}
)

实际工程中，CMakeLists 的配置由 opbase 框架统一管理，开发者通常只需要在算子目录的 CMakeLists.txt 中添加源文件路径即可。

4.4.8 编译和测试

编译

使用 ops-blas 仓提供的 build.sh 脚本编译：

# 编译 add 算子（指定目标芯片）
bash build.sh --ops=add --soc=Ascend910B

# 编译成功后，Kernel 二进制会生成在 out/ 目录下
ls out/kernel/add_kernel.o

编译过程会依次执行：Ascend C 编译器编译 Kernel → C++ 编译器编译 Host 侧 → 链接生成动态库。

测试

编译通过后，运行 ST（System Test）用例验证功能正确性：

# 运行 add 算子的 ST 测试
cd build && ctest -R add --output-on-failure

ST 测试会自动生成随机输入数据，调用 CPU golden 计算期望结果，然后在 NPU 上执行算子，最后比对 NPU 输出和 golden 是否在精度范围内。

4.4.9 与直调方式的对比

下面对比 aclnn 框架开发和直调方式的关键差异：

维度直调方式aclnn 框架
Kernel 启动Host 侧直接 <<<coreNum, stream>>>框架自动调度，通过 executor 启动
Tiling 计算Host 侧手动计算，手动 aclrtMemcpy 到 GM框架调用 Tiling 函数，自动传递到 Kernel
Workspace自行分配和管理两段式接口自动管理
参数校验自行实现框架提供统一的校验机制
上层对接需要自己封装 C 接口框架自动生成 aclnn 接口，PyTorch 直接调用
适用场景学习、验证、原型开发生产环境、正式算子入库

建议：入门阶段先用直调方式理解算子的运行机制（参考 2.3 节），掌握后再切换到 aclnn 框架进行正式开发。两者的 Kernel 侧代码几乎完全相同，主要区别在 Host 侧。

4.4.10 完整开发 Checklist

开发一个 aclnn 框架算子，按以下顺序逐项完成：

  1. ✅ 定义 TilingData 结构体（add_tiling_data.h）
  2. ✅ 实现 Kernel 类和入口函数（add_kernel.cpp）
  3. ✅ 实现 Tiling 函数和 aclnn 两段式接口（add_host.cpp）
  4. ✅ 配置 CMakeLists.txt
  5. ✅ 编译通过，确认 Kernel 二进制生成
  6. ✅ ST 测试通过，精度达标
  7. ✅ 性能测试，确认达到预期带宽利用率
