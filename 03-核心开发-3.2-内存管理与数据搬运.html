3.2.1 存储层次回顾

昇腾 NPU 的存储体系是一个多层级的金字塔结构。从底层到顶层，容量越来越小，但访问速度越来越快：

  • GM（Global Memory）：片外 HBM，容量最大（几十 GB），但延迟最高。所有 AI Core 共享，是输入输出数据的存放地。
  • L1：片上缓存，容量比 GM 小得多，但访问速度快。主要用于 Cube 计算的中间存储。
  • L0A / L0B / L0C：Cube 计算单元的专用寄存器，容量最小但速度最快。
  • UB（Unified Buffer）：Vector 计算的主战场，每核独享，A5 上为 248KB。

数据在不同层级之间搬运，就像雪王奶茶的备料流程：原料从总仓（GM）拉到门店冷库（L1），再摆到吧台（UB），调饮师在操作台上加工（Vector/Cube 计算），做好的成品再打包送出。每次搬运都有时间成本，好的算子设计要尽量减少不必要的搬运，并让搬运和加工重叠执行——调饮师做上一杯的时候，下一杯的原料已经在路上了。

对于 swap 这种纯搬运算子，数据路径非常简单：从 GM 搬到 UB，在 UB 里交换两个 Buffer 的内容，再从 UB 搬回 GM。整个过程不涉及 Vector 计算和 Cube 计算，只用到 MTE2（搬入）和 MTE3（搬出）两个搬运引擎。

3.2.2 Global Memory (GM)

GM 是片外 HBM（高带宽内存），相当于 NPU 的"主内存"。它的容量很大（通常几十 GB），所有 AI Core 共享同一块 GM 空间。算子的输入数据和输出结果都存放在 GM 中。

在 Kernel 函数中，GM 通过 GM_ADDR 指针传入。你可以把它理解为一个指向 HBM 的裸指针，但访问它需要特殊的语法：

extern "C" __global__ __aicore__ void my_kernel(
    GM_ADDR x, GM_ADDR y,
    GM_ADDR workspace, GM_ADDR tiling)

Kernel 内部通过 SetGlobalBuffer 把 GM_ADDR 包装成 GlobalTensor，方便后续用偏移量访问：

xGM_.SetGlobalBuffer((__gm__ float*)x, tiling_.totalN);
yGM_.SetGlobalBuffer((__gm__ float*)y, tiling_.totalN);

包装之后，就可以用 xGM_[offset] 的方式访问 GM 中任意位置的数据，就像访问普通数组一样。

3.2.3 Unified Buffer (UB)

UB 是每个 AI Core 的片上高速缓存，是 Vector 计算的主战场。与 GM 不同，UB 是每核独享的——核 0 的 UB 和核 1 的 UB 是完全独立的两块内存，互不干扰。

A5 芯片每核 UB 容量为 248KB。听起来不少，但对于大规模计算来说仍然很紧张，需要精心规划 Buffer 的分配。

UB 内存管理

Ascend C 提供了三种 UB 内存管理方式：

TBuf（静态 Buffer）：在编译时确定大小，适合大小固定的临时存储。用法简单，但灵活性差。

LocalTensor（动态 Tensor）：在运行时从 UB 中分配一块内存，适合大小在运行时才能确定的场景。通过 AllocTensor 分配，FreeTensor 释放。

pipe.InitBuffer（队列式管理）：最常用的方式。通过 TPipe 对象初始化队列，框架自动管理 Buffer 的分配和回收。swap 就使用这种方式：

pipe_->InitBuffer(inQueueX_, BUFFER_NUM, bufSize);
pipe_->InitBuffer(inQueueY_, BUFFER_NUM, bufSize);

这里为 x 和 y 各初始化了一个队列，单 Buffer，每个 Buffer 的大小为 tileSize * sizeof(float)。

对齐要求

UB 中有一个硬性约束：Buffer 的起始地址必须 32 字节对齐，搬运长度也必须是 32 字节的倍数。这是因为 MTE（Memory Transfer Engine）以 32B 为最小搬运粒度。

对于 FP32（4 字节/元素），32B 对齐意味着元素个数必须是 8 的倍数。所有 Tiling 计算中的切分粒度都要对齐到这个值。

如果数据量不是 32B 的倍数怎么办？可以用 DataCopyPad 来处理，后面会详细讲。

3.2.4 DataCopy：数据搬运核心 API

DataCopy 是 Ascend C 中最基础、最常用的数据搬运 API。它负责在不同存储层级之间搬运数据，是整个算子执行流程的"物流系统"。

基本用法

DataCopy 的调用非常直观，根据参数类型自动判断搬运方向：

// GM → UB（搬入）
DataCopy(dstLocal, srcGlobal, length);

// UB → GM（搬出）
DataCopy(dstGlobal, srcLocal, length);

length 是搬运的元素个数（不是字节数）。搬运的源和目标可以是 GlobalTensor（GM 侧）或 LocalTensor（UB 侧），框架会根据参数类型自动选择正确的 MTE 引擎。

在 swap 中，搬入阶段把 x 和 y 的数据从 GM 搬到 UB：

DataCopy(xLocal, xGM_[curOffset], alignedCount);
DataCopy(yLocal, yGM_[curOffset], alignedCount);

搬出阶段把交换后的数据从 UB 写回 GM——注意这里是"交叉写回"：x 的内容写到 y 的 GM 地址，y 的内容写到 x 的 GM 地址：

DataCopy(yGM_[curOffset], xIn, alignedCount);
DataCopy(xGM_[curOffset], yIn, alignedCount);

DataCopyPad：带填充的搬运

当数据量不是 32B 对齐时，直接用 DataCopy 会报错。DataCopyPad 通过自动填充（padding）来解决这个问题：搬运时把不足 32B 的尾部数据补齐到 32B 边界，填充的值不影响计算结果。

swap 在处理尾块时使用了 DataCopyPad。当 tailCount（尾部元素数）不是 8 的倍数时：

uint8_t paddingNum = ELEMENTS_PER_BLOCK - tailCount;
DataCopyExtParams copyParams{1, static_cast<uint32_t>(tailCount * sizeof(float)), 0, 0, 0};
DataCopyPadExtParams<float> padParams{true, 0, paddingNum, 0.0f};
DataCopyPad(xLocal[alignedCount], xGM_[curOffset + alignedCount], copyParams, padParams);

这段代码的含义是：从 GM 搬运 tailCount 个元素到 UB，不足 32B 的部分用 0.0f 填充。paddingNum 是需要填充的元素个数，copyParams 指定实际搬运的字节数。

搬出时同样需要处理尾部，但不需要填充（因为 UB 里已经有填充好的数据了）：

DataCopyPad(yGM_[curOffset + alignedCount], xIn[alignedCount], copyParams);

3.2.5 队列（Queue）机制

队列（Queue）是 Ascend C 流水线的核心数据结构。它扮演了"生产者"和"消费者"之间的缓冲区角色：搬运单元（MTE2）作为生产者把数据放入队列（EnQue），计算单元（Vector/Cube）作为消费者从队列取出数据（DeQue），计算完成后再把结果放入输出队列，由 MTE3 搬出。

队列的价值不仅在于缓冲，更在于隐式同步：EnQue 操作会自动等待搬运完成，DeQue 操作会自动等待数据就绪。开发者不需要手动写 SetFlag/WaitFlag，队列机制帮你处理了大部分同步需求。

队列使用流程

swap 的队列使用分为三步：

第一步：初始化。 在 Init 阶段，通过 pipe_->InitBuffer 为队列分配 UB 空间：

pipe_->InitBuffer(inQueueX_, BUFFER_NUM, bufSize);
pipe_->InitBuffer(inQueueY_, BUFFER_NUM, bufSize);

第二步：生产者（搬入 + 入队）。 从 GM 搬入数据，然后 EnQue：

LocalTensor<float> xLocal = inQueueX_.AllocTensor<float>();
LocalTensor<float> yLocal = inQueueY_.AllocTensor<float>();
DataCopy(xLocal, xGM_[curOffset], alignedCount);
DataCopy(yLocal, yGM_[curOffset], alignedCount);
inQueueX_.EnQue<float>(xLocal);
inQueueY_.EnQue<float>(yLocal);

第三步：消费者（出队 + 使用 + 释放）。 DeQue 取出数据，使用完毕后 FreeTensor：

LocalTensor<float> xIn = inQueueX_.DeQue<float>();
LocalTensor<float> yIn = inQueueY_.DeQue<float>();
// ... 使用 xIn 和 yIn 做搬出 ...
inQueueX_.FreeTensor(xIn);
inQueueY_.FreeTensor(yIn);

多队列协作

swap 有两个输入队列，分别管理 x 和 y 的数据流。对于更复杂的算子，可能还需要输出队列、中间结果队列等。不同队列之间相互独立，各自管理自己的 Buffer 生命周期。

队列机制的一个关键优势是：它把同步逻辑封装在了 EnQue/DeQue 的语义中。你不需要关心"MTE2 搬完了没有"，只要 DeQue 能返回数据，就说明搬运一定已经完成了。这将在 3.5 同步与流水线 中深入讨论。

3.2.6 各层存储之间的搬运路径

不同的搬运路径由不同的 MTE（Memory Transfer Engine）负责。下表列出了 A5 架构上所有可用的搬运路径：

源目标MTE 引擎说明
GMUBMTE2输入数据搬入（Vector 类算子）
GML1MTE2Cube 输入搬入
L1L0AMTE1Cube 左矩阵搬入
L1L0BMTE1Cube 右矩阵搬入
L0CUBMTE1Cube 输出到 Vector（A5 新增）
UBGMMTE3结果搬出
UBL1MTE1Vector 到 Cube（A5 新增）
L1UBMTE1Cube 输入到 Vector

swap 只用到了两条路径：GM→UB（MTE2 搬入）和 UB→GM（MTE3 搬出）。这两条路径分别由 MTE2 和 MTE3 引擎驱动，可以并行执行——也就是说，MTE2 在搬入下一轮数据的同时，MTE3 可以搬出上一轮的结果。这就是流水线并行的基础。

A5 相比 A2/A3 新增了两条重要通路：L0C→UB 和 UB↔L1。这意味着 Cube 的计算结果可以直接送到 Vector 做后处理，Vector 的中间结果也可以直接送给 Cube 做下一轮矩阵乘，不需要绕道 GM。这大大提升了混合计算场景的效率。

3.2.7 内存对齐详解

32B 对齐是昇腾 NPU 的硬性要求，不是建议，是必须遵守的规则。违反对齐要求会导致两种后果：一种是 DataCopy 直接报错（运行时异常），另一种更隐蔽——不报错但数据错位，导致精度问题。

对齐要求涉及两个方面：

  1. 地址对齐：UB 中 Buffer 的起始地址必须是 32B 的倍数。通过 InitBuffer 分配的 Buffer 自动满足这个要求。
  2. 长度对齐：每次 DataCopy 搬运的字节数必须是 32B 的倍数。对于 FP32 就是 8 个元素的倍数，FP16 是 16 个元素的倍数。

当数据量不满足对齐要求时，有两种处理方案：

  • Tiling 阶段对齐：在切分时就把每轮的数据量对齐到 32B 边界，尾块单独处理。swap 的切分粒度都做了对齐。
  • DataCopyPad 填充：对于尾块，用 DataCopyPad 自动补齐不足的部分。swap 在尾部元素数不是 8 的倍数时使用了这种方式。

详细的对齐规则速查表见 附录 B：对齐规则速查。
