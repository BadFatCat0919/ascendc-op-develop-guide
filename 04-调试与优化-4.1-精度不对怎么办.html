4.1.1 精度问题的常见表现

算子跑完了，结果不对——这是开发中最让人头疼的事情。就像你在大绿书搜"三亚旅游攻略"，搜到的内容可能是：全篇广告（输出全 0）、大致靠谱但酒店价格差了两倍（偏差大）、有的景点对有的景点错（部分错误）。精度问题也有这三种表现形式，每种背后对应着不同的根因方向：

输出全为 0 或随机值 — 这往往意味着数据根本没有被正确处理。最常见的原因是 Buffer 未初始化，或者流水线同步出了问题：数据还没从 GM 搬到 UB，计算单元就已经开始读了，读到的全是垃圾值。

输出与 golden 偏差大 — 数据流是对的，但数值算错了。常见于类型转换（比如 FP32 先转成 FP16 再转回 FP32，中间丢失了精度）或累加器精度不够（FP16 累加大量数据时溢出）。

部分元素正确、部分错误 — 这类问题最隐蔽。通常是 DataCopy 不对齐（导致数据错位）、尾块 mask 算错（最后几个元素没处理到），或者多核切分时边界计算有误。

sswap 的例子：sswap 是位精确算子（PrecisionMode::EXACT），它只做数据交换，不涉及任何数值计算。所以如果 sswap 的精度不对，问题一定出在数据搬运环节——要么地址写错了，要么数据没搬完。

4.1.2 排查流程

遇到精度问题，不要慌，按照下面这棵决策树一步步排查：

精度不对
  ├── 1. 检查输入数据是否正确
  │     └── 用 printf 打印前几个元素，对比预期
  ├── 2. 检查 DataCopy 是否对齐
  │     └── 长度和地址是否 32B 对齐？→ 见附录 B
  ├── 3. 检查流水线同步
  │     └── EnQue/DeQue 是否配对？SetFlag/WaitFlag 是否正确？
  ├── 4. 检查类型转换
  │     └── FP32→FP16→FP32 往返转换有精度损失
  ├── 5. 检查累加器精度
  │     └── FP16 累加大量数据可能溢出，改用 FP32 累加器
  └── 6. 检查尾块处理
        └── 最后一个 tile 的 mask 是否正确？

4.1.3 问题1：输出全为 0

可能原因：

  • Buffer 未初始化
  • 流水线同步错误（数据还没搬完就开始计算）
  • EnQue/DeQue 不配对

排查步骤：

  1. 在 DataCopy 之后加 printf，确认数据已搬入 UB
  2. 检查 SetFlag/WaitFlag 的 Pipe 参数是否正确
  3. 检查 Queue 的 stageNum 是否与实际使用匹配

4.1.4 问题2：输出与 golden 偏差大

可能原因：

  • FP16 累加溢出（大量数据累加时 FP16 精度不够）
  • 类型转换损失（FP32→FP16→FP32 的往返转换）
  • 量化/反量化参数错误

排查步骤：

  1. 将累加器改为 FP32，看精度是否改善
  2. 检查 Cast API 的源/目标类型是否正确
  3. 对比中间结果，定位精度损失发生在哪一步

4.1.5 问题3：部分元素正确、部分错误

可能原因：

  • DataCopy 不对齐（导致数据错位）
  • 尾块 mask 计算错误
  • 多核切分时边界计算错误

排查步骤：

  1. 检查 DataCopy 的长度是否是 32B 的倍数
  2. 检查最后一个 tile 的 mask 值
  3. 用 printf 打印每个核的 block_idx 和处理范围

4.1.6 实战案例：sswap 的交叉写回问题

sswap 的功能是交换两个向量 x 和 y 的对应元素。在 Kernel 内部，数据流是这样的：

GM(x) ──CopyIn──→ UB(bufX) ──Swap──→ UB(bufX_out)  ──CopyOut──→ GM(y)
GM(y) ──CopyIn──→ UB(bufY) ──Swap──→ UB(bufY_out)  ──CopyOut──→ GM(x)

注意最后一步：x 的数据要写回到 y 的地址，y 的数据要写回到 x 的地址。这就是"交叉写回"。

如果你在写 CopyOut 时把地址搞反了——比如把 bufX_out 写回了 GM(x) 而不是 GM(y)——结果就是：算子跑完，x 和 y 的内容完全没变。表面上看"好像算子没干活"，实际上是写回地址错了。

这类问题在 sswap 的 ST 测试中会被精确捕获，因为测试代码使用了 PrecisionMode::EXACT 模式做位精确比对：

VerifyConfig cfg;
cfg.mode = PrecisionMode::EXACT;
EXPECT_TRUE(Verifier::verifyVector(xPtr, goldenX.data(), ...));
EXPECT_TRUE(Verifier::verifyVector(yPtr, goldenY.data(), ...));

只要有一个 bit 不对，测试就会失败。对于 sswap 这种纯搬运算子，这是完全合理的——交换操作不应该改变任何一位数据。

4.1.7 精度比对方法

精度验证的核心思路很简单：把 NPU 的输出和 CPU golden 的输出逐元素对比。

对于浮点类算子，使用 atol（绝对误差）和 rtol（相对误差）来评估精度：

对于每个元素：
|actual - expected| <= atol + rtol * |expected|

这个公式的含义是：允许的误差由两部分组成——一个固定的绝对误差 atol，加上一个与期望值成正比的相对误差 rtol。当期望值很大时，允许的绝对误差也相应放大。

这里用 atol/rtol 做简化说明，便于快速理解。实际 opbase 标准使用 MERE/MARE 聚合指标，详见 4.3 算子精度标准。

对于 sswap 这类位精确算子，atol 和 rtol 都是 0——不允许任何偏差。详细的精度标准见 4.3 算子精度标准。

4.1.8 调试工具

排查精度问题时，有几个趁手的工具：

printf — 最简单粗暴的方式。直接在 Device 侧的 Kernel 代码里加 PRINTF，打印 UB 中的数据。优点是所见即所得，缺点是会严重影响性能，而且大量打印会刷屏。适合在开发初期快速验证数据流。

仿真器 — CANN Simulator 可以在 CPU 上模拟 NPU 的执行行为，支持单步调试、断点、变量查看。最大的优势是可以和 CPU golden 逐步对比，精确到每一条指令。适合定位复杂的流水线同步问题。

plog — 昇腾的结构化日志系统，支持日志级别控制（INFO/WARN/ERROR）。相比 printf，plog 的输出更规范，可以按级别过滤，适合在生产环境中排查问题。
