3.2.1 A5 架构升级概述

A5（DAV_3510）是昇腾架构的一次重大升级。相比 A2/A3（DAV_2201），A5 在数据通路、编程模型、数据类型、同步机制等方面都引入了重要变化。这些变化直接影响算子的写法和性能上限。

用一句话概括：A5 让 Cube 和 Vector 之间的数据流动更自由了，让程序员可选的编程方式更多了，让低精度计算的支持更丰富了，让多流水同步的代码更简单了。

3.2.2 新增数据通路

A5 新增了几条重要的数据通路，大幅提升了核内数据流转的灵活性。这些通路在 A2/A3 上不存在，是 A5 架构独有的能力。

L0C → UB

Cube 的计算结果可以直接送到 Vector 单元做后处理（如激活函数、类型转换），不再需要先写回 GM 再由 Vector 读回来。这对 FlashAttention / FIA 类融合算子意义重大，可以消除 workspace 的 GM 读写，显著降低延迟。

UB ↔ L1

Vector 与 Cube 之间建立了双向直连通道，数据可以在两个计算单元之间自由流动。UB→L1 是新增方向，让 Vector 的中间结果可以直接送入 Cube 的输入缓存。

SSBuffer 消息通路

核间细粒度同步机制，允许多个 AI Core 之间通过消息传递进行协作，而不需要经过 GM 中转。SSBuffer 容量约 256KB，是 DAV_3510 新增的 Buffer 类型。

3.2.3 新增编程模型

A5 引入了多种全新的编程模型，给算子开发者提供了更多选择。这些编程模型在 2.2 编程模型 中有详细讲解，这里做一个总览。

编程模型说明适用场景
SIMT线程级并行方式，每个线程处理一个或少量数据元素，硬件按 Warp（32 线程一组）调度离散访问（Gather/Scatter）、排序、Hash
SIMD-RegBase中间计算结果保存在寄存器中而非 UB，减少 UB 读写次数，支持 OOO 指令双发多步融合计算（如 Gelu+Cast）
NDDMA高维 DMA 指令，支持多维张量的直接搬运，不需要拆成一维循环多维张量搬运、转置

3.2.4 新增数据类型

A5 新增了一系列低精度数据类型，以支持大模型推理中的量化需求。精度越低，Cube 的 MAC 阵列越大，吞吐越高。

数据类型格式Cube MAC 阵列说明
FP8 E5M21 符号 + 5 指数 + 2 尾数16×32×16适合梯度表示
FP8 E4M31 符号 + 4 指数 + 3 尾数16×32×16适合权重/激活
MXFP8微缩浮点，32 个 Data 共享 1 个 Scale16×32×16进一步压缩数据
MXFP44-bit 浮点（E2M1/E1M2）16×64×16最低精度，最高吞吐
HiF8华为自定义 8 位浮点16×32×16华为优化格式

MXFP 类型有一个特殊的 Tiling 要求：32 个 Data 共享 1 个 Scale，如果 Scale 采用与 Data 相同的 StepK 会导致 Scale 的 TileSize 过小、带宽利用率低，需要采用独立的 scaleFactor 缓存。

3.2.5 BufferID 同步机制

A2/A3 上，多流水并行时需要严格配对使用 SetFlag / WaitFlag 来做同步——一个流水段 Set 了某个 flag，另一个流水段必须 Wait 同一个 flag，配对关系不能乱。这就像两个人传纸条：A 写了纸条举起来（SetFlag），B 看到纸条举起来才能去拿（WaitFlag），如果 A 举了两张纸条但 B 只拿了一张，就会出问题。

A5 引入了 BufferID 同步机制，每个 Buffer 自带独立的同步标识，硬件自动管理依赖关系。就像给每个包裹贴了追踪码——系统自动知道哪个包裹到了、谁该去取，不需要人工配对。这消除了手动配对的需求，大幅简化了多流水算子的同步代码。

Q：BufferID 同步机制意味着 A5 上完全不需要 SetFlag/WaitFlag 了吗？

A：不是完全不需要，而是大部分场景下不需要手动配对了。BufferID 机制让硬件自动管理 Buffer 之间的依赖关系，但在某些特殊的同步场景下（比如跨 Buffer 的全局同步），仍然可能需要显式的同步操作。

3.2.6 存储容量变化

A5 在存储容量上有两处显著增大：

BufferA2 (DAV_2201)A5 (DAV_3510)变化
L0C128 KB256 KB翻倍 — Cube 输出缓冲区更大，减少写回次数
UB192 KB248 KB增加 56KB — Vector 工作区更大
BT1 KB4 KB4 倍 — FixPipe Bias 表更大
SSBuffer—~256 KB新增 — 核间消息通路

3.2.7 不再支持的特性

A5 在引入新特性的同时，也移除了一些 A2/A3 上的特性。如果你的算子依赖这些特性，需要在 A5 上改用替代方案。

移除的特性影响替代方案
4:2 稀疏矩阵计算原依赖该特性提速的 kernel 无法使用改为稠密计算或其他支持的稀疏策略
GM → L0A/L0B 直连通路不能直接从 GM 搬数据到 L0A/L0B改为 GM → L1 → L0A/L0B（经过 L1 中转）
L1 → GM 直连通路不能直接从 L1 搬数据到 GM改为 L1 → UB → GM（经过 UB 中转）

Q：我在 A2 上开发的算子，能直接在 A5 上跑吗？

A：大部分情况下可以。如果你只用了 SIMD-MemBase 编程模型，并且没有依赖 GM→L0A/L0B 或 L1→GM 的直连通路，那么 arch22 的代码通常可以在 A5 上直接运行。但如果想充分利用 A5 的新特性（SIMT、RegBase、FP8 等），需要在 arch35 目录下写专门的优化版本。

Q：A5 删除了 GM→L0A/L0B 通路，对矩阵乘算子有什么影响？

A：在 A2 上，矩阵乘的数据可以直接从 GM 搬到 L0A/L0B。在 A5 上，这条路径被删除了，数据需要先经过 L1 中转（GM → L1 → L0A/L0B）。这意味着 A5 的矩阵乘算子需要多一步搬运，但 L0C 翻倍到 256KB 和 L0C→UB 直连通路的引入，在整体上弥补了这一步的开销。
