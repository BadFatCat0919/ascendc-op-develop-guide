3.4.1 Cube 计算单元

如果说 Vector 单元是"逐个加工零件"的流水线工人，那 Cube 单元就是"一次搬一车砖"的重型机械。Cube 是昇腾 NPU 的矩阵乘加（MAC）计算单元，内部是一个 16×16×16 的 MAC 阵列，每个时钟周期可以完成一次 16×16 的矩阵乘法。

Cube 的算力远大于 Vector。以 A5 (950PR Server) 为例，Vector 的 FP16 算力是 54T，而 Cube 的 BF16/FP16 算力高达 432T——差了将近 8 倍。这是因为矩阵乘是深度学习中计算密度最高的操作（全连接层、注意力机制、卷积等本质上都是矩阵乘），芯片设计时把大部分晶体管面积给了 Cube。

不过，不是所有算子都需要 Cube。我们的贯穿示例 swap 就是一个鲜明的对比：swap 是纯数据搬运型算子，只用到 MTE2/MTE3 搬运引擎，完全不涉及 Cube 计算。而矩阵乘（MatMul）、卷积（Conv2D）、注意力（Attention）等算子则是 Cube 的主力用户。理解这两类算子的区别，有助于你在设计 Tiling 和流水线时做出正确的选择。

A5 Cube 算力

芯片核数频率BF16/FP16FP8MXFP4
A2 (910B2)241.8 GHz353T--
A5 (950PR PCIE)281.65 GHz378T757T1514T

A5 相比 A2 新增了 FP8 和 MXFP4 低精度支持，算力翻倍甚至翻四倍。这对于大模型推理场景（如 LLM 的 KV-Cache 量化）非常关键。

3.4.2 分块矩阵乘原理

大矩阵乘 C[M,N] = A[M,K] × B[K,N] 不可能一次性塞进 Cube 的 16×16 阵列里。就像搬一车砖，一次只能搬 16 块，需要分多次搬。Cube 的做法是把大矩阵切成 16×16 的小块（Tile），每次计算一个小块，累加到输出矩阵的对应位置。

基本块（Tile）

C[M,N] = A[M,K] × B[K,N]

切分为：
C[m,n] = Σ A[m,k] × B[k,n]

其中 m, n, k 是基本块大小，通常为 16 的倍数

每个基本块的大小是 16 的倍数（因为 Cube 阵列是 16×16 的），多个基本块的计算结果累加起来，就得到了完整的输出矩阵。

三层循环

分块矩阵乘的核心是一个三层嵌套循环：

for each k_block:        // K 维度循环（受 L0A/L0B 容量限制）
    for each m_block:    // M 维度循环（受 L0C 容量限制）
        for each n_block: // N 维度循环（受 L0C 容量限制）
            C[m,n] += A[m,k] × B[k,n]
  • K 维度是累加维度：同一个输出块 C[m,n] 需要对 K 上的所有块做累加。K 维度的循环次数取决于 L0A 和 L0B 能装下多大的块。
  • M 和 N 维度是输出维度：不同的 (m,n) 对应输出矩阵的不同块，可以分配给不同的 AI Core 并行计算。

Tiling 设计的关键就是确定这三层循环的块大小，使得每块数据恰好能装进对应的 Buffer（L0A/L0B/L0C）中。

3.4.3 L0A / L0B / L0C 的使用

Cube 计算有三个专用 Buffer，各司其职：

  • L0A：存放左矩阵 A 的当前块。就像一个"原料托盘"，每次放一块 A 的数据。
  • L0B：存放右矩阵 B 的当前块。另一个"原料托盘"。
  • L0C：存放输出矩阵 C 的累加结果。像一个"成品托盘"，每次矩阵乘的结果都累加到这里。

数据流

数据从 GM 出发，经过 L1 中转，最终到达 L0A/L0B，在 Cube 中计算后写入 L0C，最后搬出到 UB 或 GM：

注意这条路径比 swap 复杂得多。swap 只需要 GM↔UB 两条路径，而矩阵乘需要 GM→L1→L0A/L0B→L0C→UB→GM 一整条链路，涉及 MTE2、MTE1、MTE3 三个搬运引擎的协调。

容量限制

Buffer容量可存放的 FP16 矩阵大小
L0A64 KB16 × 2048 或 128 × 256
L0B64 KB16 × 2048 或 128 × 256
L0C128 KB (A2) / 256 KB (A5)128 × 128 (FP32 累加)

L0A 和 L0B 各 64KB，L0C 在 A5 上有 256KB（是 A2 的两倍）。L0C 更大是因为输出通常用 FP32 累加（即使输入是 FP16），每个元素占 4 字节而不是 2 字节。

这些容量限制直接决定了 Tiling 中 K/M/N 维度的块大小上限。比如 L0A 只能放 64KB，如果你选择 FP16 输入，那 m × k × 2 ≤ 65536，即 m × k ≤ 32768。

3.4.4 MatMul API

Ascend C 提供了高层的 MatMul API，封装了 Cube 计算的底层细节。开发者不需要手动管理 L0A/L0B/L0C 的分配和搬运，只需要调用几个函数就能完成矩阵乘。

基本用法

// 从 L1 搬入 L0A/L0B
CopyIn(leftMatrix, rightMatrix);
// 执行矩阵乘
Matmul(outputMatrix, leftMatrix, rightMatrix);
// 从 L0C 搬出
CopyOut(outputMatrix);

这三步对应了 Cube 计算的完整生命周期：搬入→计算→搬出。CopyIn 负责把数据从 L1 搬到 L0A 和 L0B，Matmul 执行矩阵乘并把结果累加到 L0C，CopyOut 把 L0C 的结果搬到 UB 或 GM。

融合算子

为了进一步提升性能，Ascend C 还提供了一些融合算子，把矩阵乘和后处理操作合并成一步执行：

  • MatmulLeakyRelu：矩阵乘 + LeakyReLU 激活函数融合。省去了把中间结果搬到 UB 做 ReLU 再搬回来的开销。
  • Matmul + FixPipe：矩阵乘 + 后处理流水线融合。FixPipe 可以在 Cube 输出的同时做量化、Bias 加法等操作，不额外占用计算时间。

3.4.5 FixPipe：后处理流水线

FixPipe 是 Cube 输出端的后处理单元，你可以把它理解为 Cube 的"下游加工站"。当 Cube 把矩阵乘的结果写入 L0C 后，FixPipe 可以在数据搬出之前对它做一系列后处理操作。

FixPipe 支持的操作包括：

  • 量化（Quant）：把 FP32 累加结果量化为 INT8，用于量化推理。
  • 反量化（Dequant）：把 INT8 输入反量化为浮点。
  • Bias 加法：给输出加上偏置向量。
  • Relu / LeakyRelu：激活函数。

FixPipe 的最大优势是与 Cube 流水线并行执行。当 Cube 在计算下一个块时，FixPipe 可以同时处理上一个块的输出。这意味着后处理操作不额外占用计算时间，几乎是"免费"的。

3.4.6 Cube 计算的数据格式要求

Cube 对输入数据的排布格式有特殊要求。与 Vector 单元直接使用普通的 NCHW/NHWC 格式不同，Cube 要求数据按照特定的分块格式存储（如 FracZ、5HD 等），以匹配其内部的 16×16 MAC 阵列。

如果你的输入数据不符合 Cube 要求的格式，需要先调用 TransData 做格式转换。但格式转换本身有开销（需要额外的搬运和计算），好的 Tiling 设计应该尽量减少转换次数。

一种常见的优化策略是：在算子图的编译阶段就规划好数据格式，让上游算子的输出格式恰好是下游 Cube 算子的输入格式，避免运行时的格式转换。

更多关于 Cube 计算的高级话题（如 NDDMA、CCU 通算融合），详见 8.2 RegBase 编程模型。
