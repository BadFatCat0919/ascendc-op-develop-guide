3.1.1 什么是 Tiling

Tiling 的本质是"分而治之"。想象公司团建点了 1000 杯雪王奶茶，一家门店做不完，于是你把订单分给 20 家门店同时做——每家分 50 杯。但问题来了：1000 除 20 刚好整除，那如果是 1003 杯呢？最后 3 杯给谁做？而且每家门店的吧台（UB）就那么大，一次放不下 50 杯的原料，得分批从总仓库（GM）进货。Tiling 就是解决这些问题的"分单策略"——怎么分、每家分多少、分几批做。

一个好的 Tiling 策略需要同时满足三个目标：

  1. 负载均衡：每个 AI Core 分到的工作量尽量相等，避免出现"有人闲着、有人忙死"的情况。
  2. 充分利用 UB 空间：A5 芯片每核有 248KB 的 Unified Buffer，这是计算的主战场。Tiling 要让每次搬运进来的数据尽量填满 UB，减少搬运次数。
  3. 减少搬运开销：每次 GM↔UB 的数据搬运都有延迟，好的 Tiling 应该让"搬运时间"和"计算时间"尽量重叠，把搬运开销藏起来。

Tiling 的三个核心问题

设计 Tiling 时，你需要回答三个问题：

  1. 多核切分：总数据量怎么分给 N 个 AI Core？每个核处理哪一段？
  2. UB 切分：每个核拿到的数据可能很大，UB 一次装不下，需要分几轮搬进来？每轮搬多少？
  3. 循环切分：UB 内部的多轮搬运如何组织成循环？最后一轮不足一整轮时怎么处理？

下面我们以 sswap（向量交换）算子为例，逐一解答这三个问题。sswap 的功能是交换两个向量 x 和 y 的对应元素，属于纯数据搬运型算子，逻辑简单，非常适合用来理解 Tiling 的基本框架。

3.1.2 多核切分策略

多核切分的目标很直接：把总数据量尽量均匀地分给所有可用的 AI Core。

基本切分公式

最直观的做法是用总数据量除以核数：

perCoreN = totalN / coreNum

但这里有一个关键细节：昇腾 NPU 的 DataCopy 要求搬运长度按 32 字节对齐。对于 FP32（4 字节/元素），这意味着每次搬运的元素个数必须是 8 的倍数（32 ÷ 4 = 8）。所以我们不能简单地均分，还需要把每核的处理量向下对齐到对齐单元。

sswap 的做法是：

constexpr uint32_t ELEMENTS_PER_BLOCK = 32 / sizeof(float); // = 8

uint32_t rawPerCore = n / coreNum;
tiling.perCoreN = (rawPerCore / alignUnit) * alignUnit;

先用 n / coreNum 算出粗略的每核处理量，再向下取整到 ELEMENTS_PER_BLOCK 的倍数。这样每个核处理的元素个数一定是 8 的倍数，满足 DataCopy 的对齐要求。

尾块处理

向下对齐之后，必然会有剩余数据。比如 1000 个元素分给 32 个核，每核对齐后只能分 24 个（假设对齐单元为 8），总共才处理了 768 个，还剩 232 个没着落。

sswap 的策略是把所有剩余数据都交给最后一个核处理：

tiling.remainder = n - tiling.perCoreN * coreNum;

在 Kernel 侧，每个核先按 perCoreN 计算自己的处理量，最后一个核额外加上 remainder：

myCount_ = tiling_.perCoreN;
if (blockIdx_ == GetBlockNum() - 1) {
    myCount_ += tiling_.remainder;
}

这种"尾块归最后一核"的策略简单有效。最后一个核虽然多干了一点活，但 remainder 通常不大（最多 coreNum × alignUnit - 1 个元素），对整体负载均衡影响很小。

3.1.3 UB 空间切分

多核切分解决了"每个核处理多少数据"的问题，但每个核分到的数据量可能远大于 UB 能装下的量。比如 A5 的 UB 是 248KB，对于 FP32 最多装约 62K 个元素，如果每核分到 10 万个元素，就需要分多轮搬入 UB 处理。

UB 切分要回答的问题是：UB 空间怎么分配给各个 Buffer？每轮能处理多少元素？

Buffer 规划示例（sswap）

sswap 需要两个输入 Buffer（xBuffer 和 yBuffer），没有中间计算结果，也不需要输出 Buffer（直接把 x 的内容写回 y 的 GM 地址，y 的内容写回 x 的 GM 地址）。所以 UB 空间的分配很简单：

UB 总空间 = 248 KB (A5)

xBuffer:  ~124 KB
yBuffer:  ~124 KB
──────────────────────
总计:     ~248 KB

对应的代码：

constexpr uint32_t ubSize = 248 * 1024;
constexpr uint32_t bufferCount = 2;
uint32_t maxElements = ubSize / (bufferCount * sizeof(float));
tiling.tileSize = (maxElements / alignUnit) * alignUnit;

maxElements 是 UB 能装下的总元素数，除以 2 得到每个 Buffer 的大小，再向下对齐到 ELEMENTS_PER_BLOCK。这个 tileSize 就是 UB 单轮能处理的最大元素数。

双 Buffer 策略

对于有 Vector 计算的算子（如 Add、Mul），通常会采用双 Buffer 策略：把每个 Buffer 分成两块，一块用于当前轮的计算，另一块用于下一轮数据的搬入，两者交替使用，让搬运和计算重叠执行。

时间线：
Buffer[0]: [搬运] [计算]        [搬运] [计算]
Buffer[1]:        [搬运] [计算]        [搬运] [计算]

不过 sswap 是纯搬运算子，没有 Vector 计算阶段，所以双 Buffer 对它没有额外收益。sswap 使用单 Buffer（BUFFER_NUM = 1），每轮搬入→交换→搬出，逻辑更简单。这将在 3.5 同步与流水线 中详细讨论。

3.1.4 Tiling 数据结构设计

Tiling 数据是 Host 侧和 Device 侧之间的"合同"。Host 侧根据输入规模和硬件参数计算出切分方案，打包成一个结构体写入 GM；Kernel 侧从 GM 读出这个结构体，就知道自己该处理哪段数据、每轮搬多少。

这个结构体需要包含 Kernel 运行所需的全部切分信息，不多不少。字段太少，Kernel 无法正确工作；字段太多，增加 GM 读写开销和代码复杂度。

sswap 的 Tiling 结构

struct SswapTilingData {
    uint32_t totalN;     // 总元素个数
    uint32_t perCoreN;   // 每核处理量（已对齐到 ELEMENTS_PER_BLOCK）
    uint32_t remainder;  // 尾块大小，分配给最后一个核
    uint32_t tileSize;   // UB 单轮处理量（已对齐到 ELEMENTS_PER_BLOCK）
};

四个字段各有分工：

  • totalN：Kernel 需要用总长度来设置 SetGlobalBuffer 的范围，防止越界访问。
  • perCoreN：每个核的基础工作量，用于计算当前核的数据起始偏移 myOffset_ = blockIdx_ * perCoreN。
  • remainder：最后一个核的额外工作量，myCount_ = perCoreN + remainder。
  • tileSize：UB 单轮能处理的最大元素数，用于控制内层循环的轮数。

Kernel 侧的主循环（Process）用这四个字段就能完整描述：

uint32_t tileLoop = myCount_ / tiling_.tileSize;  // 完整轮数
uint32_t tileHead = myCount_ % tiling_.tileSize;  // 最后一轮的元素数
for (uint32_t i = 0; i < tileLoop; i++) {
    SingleIteration(curOffset, tiling_.tileSize);
    curOffset += tiling_.tileSize;
}
if (tileHead > 0) {
    SingleIteration(curOffset, tileHead);
}

3.1.5 获取硬件参数

Tiling 函数运行在 Host 侧（CPU），但它需要知道 Device 侧（NPU）的硬件参数，比如一共有多少个 AI Core、每核 UB 有多大。这些参数必须通过运行时接口获取，禁止硬编码。

原因很简单：不同芯片型号的核数和 UB 大小不同。A2 有 48 个 Vector 核，A5 有 56 或 64 个；UB 大小也可能随芯片版本变化。如果硬编码成某个固定值，换一块芯片就会出错。

auto ascendcPlatform = platform_ascendc::PlatformAscendC(context->GetPlatformInfo());
uint32_t coreNum = ascendcPlatform.GetCoreNumAiv();
uint64_t ubSize;
ascendcPlatform.GetCoreMemSize(platform_ascendc::CoreMemType::UB, ubSize);

反例：硬编码

// ❌ 错误：硬编码典型值
constexpr uint32_t CORE_NUM = 32;
constexpr uint32_t UB_SIZE = 248 * 1024;

// ✅ 正确：运行时获取
uint32_t coreNum = ascendcPlatform.GetCoreNumAiv();
uint64_t ubSize;
ascendcPlatform.GetCoreMemSize(platform_ascendc::CoreMemType::UB, ubSize);

sswap 的 Tiling 函数中，coreNum 和 UB 大小（248KB）虽然看起来是固定值，但仍然通过接口获取，保证代码的可移植性。在实际的 ops-blas 工程中，GetVectorCoreCount 等封装函数会统一处理这些查询。

3.1.6 不同算子类型的 Tiling 策略

不同类型的算子，Tiling 的侧重点不同。

逐元素算子

逐元素算子（如 Add、Mul、sswap）是最简单的情况。每个输出元素只依赖对应位置的输入元素，元素之间没有依赖关系。因此 Tiling 只需要按元素数量均分，不需要考虑元素间的关联。

sswap 就是典型的逐元素算子：x[i] 和 y[i] 交换，跟 x[j] 和 y[j] 完全无关。所以它的 Tiling 策略非常直接——按 n 均分，对齐到 32B 边界，尾块给最后一核。

归约算子

归约算子（如 Sum、Max）需要沿某个维度聚合数据。Tiling 时需要区分"归约轴内"和"归约轴外"：轴内的数据必须累加到一起，不能随意切分；轴外的数据可以独立分配给不同核。通常的做法是：每个核先做局部归约，把中间结果写入 workspace，最后由一个核做全局归约。

矩阵乘算子

矩阵乘 C[M,N] = A[M,K] × B[K,N] 需要在 M、N、K 三个维度上同时切分。M 和 N 维度的切分类似逐元素算子（分给不同核），但 K 维度是累加维度，切分后需要累加中间结果。此外，分块大小受 L0A/L0B/L0C 容量限制，Tiling 设计更加复杂。这部分将在 3.4 Cube 计算 中详细讨论。
