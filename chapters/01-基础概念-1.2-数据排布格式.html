1.2.1 为什么数据排布很重要

无论数据在逻辑上分几维（1 维向量、2 维矩阵、4 维张量……），最终在内存中都是连续排布的一维数组。就像爬黄山，无论多少家庭、多少旅行团，最后到了"一线天"那个狭窄的通道，都得一个个地过。数据排布格式就是规定大家按什么顺序压缩成一条线通过。

同样的一批数据，在内存中可以按不同的顺序排列。就像你刷大绿书，同样一堆帖子，按"最新"排和按"最热"排，你刷到想看内容的效率完全不同。数据排布格式决定了 NPU 访问内存时的效率——如果排布方式和硬件的计算模式匹配，就能连续读取、一次搬完；不匹配就会产生大量碎片化的随机访问，效率大打折扣。

在算子开发中，数据排布格式直接影响两件事：

  • 搬运效率：NPU 的 MTE（Memory Transfer Engine）喜欢连续的大块数据搬运，碎片化的数据会让它反复启动，浪费时间。
  • 计算效率：Cube 单元（矩阵乘单元）一次计算一个固定大小的矩阵块（如 16×16），如果数据已经按计算块排好，就能直接喂进去；否则需要额外的重排操作。

1.2.2 张量与维度基础

在深度学习中，数据的基本组织形式是张量（Tensor）——一个多维数组。你可以把张量理解为"数组的数组的数组……"，维度的数量决定了张量的"阶"：

维度数名称形状示例深度学习中的对应
0 维标量（Scalar）()一个损失值
1 维向量（Vector）(N,)一个特征向量
2 维矩阵（Matrix）(M, N)全连接层权重
3 维3 阶张量(C, H, W)一张彩色图片
4 维4 阶张量(N, C, H, W)一批彩色图片（Batch）
5 维5 阶张量(N, C, D, H, W)一批 3D 医学影像

维度的顺序很重要。同样一张 3×224×224 的图片，按 (C, H, W) 排列和按 (H, W, C) 排列，在内存中的存放顺序完全不同——前者是先放完整个红色通道、再放绿色通道、最后放蓝色通道；后者是每个像素点的 RGB 三个值紧挨着放。这就是"数据排布格式"要解决的问题。

1.2.3 统一的数据排布格式

格式维度顺序 / 说明用途备注
ND任意维度，Row-major（行优先）最通用的格式，Ascend C 中最常用类似 C 语言数组存储
NCHWBatch-Channel-Height-Width卷积、图像处理PyTorch 默认格式
NHWCBatch-Height-Width-Channel卷积、图像处理TensorFlow 默认格式
NCDHWBatch-Channel-Depth-Height-Width3D 卷积场景
NC1HWC0 (5HD)将 Channel 维拆为 C1（外层）+ C0（内层），C0=16卷积输入特征图昇腾专用，匹配 Cube 的 16×16 计算粒度
FracZ按 Cube 计算分块重新排列全连接层权重昇腾专用
FracNZFracZ 的改进版，支持双维分块Transformer QKV 权重昇腾专用

ND 格式是 Ascend C 开发中最常用的格式。它不限制维度数和维度含义，数据按行优先（Row-major）顺序连续存放——就像 C 语言中二维数组 a[i][j] 的存储方式：第 0 行所有元素、第 1 行所有元素……依次排列。

表中标注"昇腾专用"的格式是为了匹配 Cube 单元（矩阵乘单元）的硬件计算需求而设计的。Cube 单元一次计算的是一个固定大小的矩阵块（如 16×16），如果数据按 ND 格式存放，每次取一个矩阵块就需要跨行跳跃读取，效率低。专用格式把数据预先按 Cube 的计算块重排，让硬件可以连续读取。

Q：为什么 ND 格式最常用，而不是 NCHW 或 NHWC？

A：因为 Ascend C 开发的大部分算子（如 Add、Mul、Swap 等逐元素和搬运算子）并不关心数据的"语义"（哪个维度是 Channel、哪个是 Height）。它们只关心"总共有多少个元素、怎么连续搬运"。ND 格式不限定维度含义，对这些算子来说最灵活。只有涉及卷积、矩阵乘等需要匹配 Cube 计算粒度的算子时，才需要使用 5HD、FracZ 等专用格式。

1.2.3.1 格式转换路径

格式转换本身是有开销的（需要搬运和重排数据），所以好的算子设计应尽量减少不必要的格式转换。如果一个算子的输入已经是 5HD 格式，就没必要先转回 ND 再处理。

1.2.4 大语言模型中的排布格式

在大语言模型（LLM）中，注意力机制（Attention）涉及多维张量的复杂变换，因此衍生了一系列专用的排布格式。这些格式的核心区别在于 Batch（B）、Sequence（S）、Num Heads（N）、Head Dim（D） 四个维度的排列顺序。

格式维度顺序说明典型场景
BSNDBatch-Sequence-NumHeads-HeadDim最常用的 Attention 输入格式Multi-Head Attention 的 Q/K/V 输入
BNSDBatch-NumHeads-Sequence-HeadDimPyTorch 的 nn.MultiheadAttention 内部格式PyTorch 原生 Attention 实现
BSHBatch-Sequence-HiddenHidden = NumHeads × HeadDim，不区分 HeadFFN 层、Embedding 层
BHSBatch-HeadDim-SequenceHeadDim 和 Sequence 交换某些优化的 Attention Kernel
SBNDSequence-Batch-NumHeads-HeadDimSequence 在最外层某些推理框架的 KV Cache 格式

这些格式之间的转换在 LLM 推理中非常频繁。例如，输入是 BSH 格式（Embedding 输出），需要先 Reshape 成 BSND 才能送入 Attention 计算，Attention 输出再 Reshape 回 BSH 送入 FFN 层。理解这些格式对于开发 LLM 相关算子（如 FlashAttention、RoPE）至关重要。

Q：为什么 Attention 要用 BSND 而不是 BSH？

A：因为 Multi-Head Attention 需要把输入拆成多个 Head 分别计算。BSH 格式把 NumHeads 和 HeadDim 合并成了 Hidden 维，无法区分不同的 Head。BSND 格式显式地保留了 NumHeads 维，让 Kernel 可以按 Head 并行处理——每个 AI Core 处理一个或几个 Head，实现多核并行。

1.2.5 对齐要求

昇腾 NPU 的内存访问有一个硬性要求：数据必须 32 字节（32B）对齐。这意味着每次从 Global Memory 搬运数据到 UB 时，搬运的起始地址和数据长度都必须是 32B 的整数倍。

为什么是 32B？因为 NPU 的 MTE（Memory Transfer Engine）一次搬运的最小粒度就是 32 字节——就像你在狗西购物凑满减，满 199 减 30，你购物车里 198 块的东西得硬凑个 1 块钱的数据线才能享受优惠。MTE 也一样，不够 32 字节的数据得用"填充物"补齐才能发车。

对齐粒度的计算方式：

对齐粒度（元素个数）= 32B / 每个元素的字节数
数据类型每元素字节数对齐粒度（元素个数）
FP32 (float)4B8 个元素
FP16 (half)2B16 个元素
BF162B16 个元素
INT81B32 个元素

对齐后长度的计算公式：

对齐后长度 = ceil(原始长度 / 对齐粒度) × 对齐粒度

以 swap 算子为例，如果使用 FP32（float）类型，每个 float 占 4 字节，所以对齐粒度 = 32 / 4 = 8 个 float。如果向量长度不是 8 的倍数，就需要在末尾补齐（padding）到 8 的倍数才能用 DataCopy 搬运。

Q：对齐填充（padding）的数据会影响计算结果吗？

A：不会。填充的数据只是用来凑够 32B 搬运粒度的"占位符"，搬运时会一起搬到 UB，但计算时算子会根据实际的向量长度（Tiling 参数中传入）忽略这些填充元素。写回结果时也只写回实际长度的数据，填充部分不会写回 Global Memory。

更多对齐规则的细节见 附录 B：对齐规则速查。
