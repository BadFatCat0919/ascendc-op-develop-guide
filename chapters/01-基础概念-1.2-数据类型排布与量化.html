1.2.1 为什么需要不同的数据类型

在深度学习中，数据类型的选择本质上是一个精度-速度-显存的三角权衡：

  • 精度越高（如 FP32），数值表示越准确，但每个数占 4 字节，显存消耗大、搬运慢。
  • 精度越低（如 INT8、INT4），每个数只占 1 字节甚至半字节，显存省了、搬运快了，但数值表示会有损失。

实际工程中的选择策略：

  • 训练通常用 FP32 或 BF16，因为梯度累积对精度敏感。
  • 推理可以用 FP16、INT8 甚至 INT4，因为最终输出对小幅误差容忍度更高。

A5（Ascend 950 / DAV_3510）对不同数据类型的算力差异很大。以 Cube 单元（矩阵乘）为例：FP8 的算力是 FP16 的 2 倍，MXFP4 的算力是 FP16 的 4 倍。所以选对数据类型，不仅省显存，还能直接翻倍算力。

1.2.2 浮点数基础

所有浮点数格式都遵循 IEEE 754 标准的基本思想：一个数由三部分组成——符号位 + 指数位 + 尾数位。

类比一下：你可以把浮点数想象成科学计数法 ±d.dd × 10^e。符号位决定正负，指数位决定数的大小范围（相当于 10^e 中的 e），尾数位决定有效数字的精度（相当于 d.dd 中有几位小数）。

类型位宽符号指数尾数动态范围精度典型用途
FP32 (float)321823大高训练主精度
FP16 (half)161510小中推理、混合精度训练
BF16 (bfloat16)16187大低混合精度训练
FP8 E4M38143很小很低推理量化
FP8 E5M28152小很低训练梯度量化
INT88-----量化推理
INT44-----极致量化推理

BF16 和 FP16 的区别是面试和实战中的高频考点。它们都是 16 位，但"天赋点点的位置不同"——就像王者里同一个英雄，有人主加攻击（精度），有人主加血量（动态范围）。所谓"天赋点"，就是游戏中升级时可以分配的属性点，选择不同的加点方向会让英雄有不同的特点：

  • BF16 把 8 位给了指数（和 FP32 一样），所以动态范围和 FP32 一样大，不容易溢出；但尾数只有 7 位，精度比 FP16 低。适合训练场景——梯度值忽大忽小，需要大的动态范围来兜住。
  • FP16 只把 5 位给指数，动态范围较小（最大约 65504），但尾数有 10 位，精度更高。适合推理场景——权重值分布相对稳定，不需要太大的动态范围。

Q：为什么训练用 BF16 而推理用 FP16？

A：训练时梯度值忽大忽小，需要大的动态范围来兜住，BF16 的 8 位指数和 FP32 一样大，不容易溢出。推理时权重值分布相对稳定，不需要太大的动态范围，FP16 的 10 位尾数精度更高，输出更准确。

1.2.4 A5 特有的数据类型

A5（DAV_3510 架构）相比 A2/A3 新增了多种低精度数据类型，主要服务于量化推理和高效训练。

命名规则解读

看到 FP8 E4M3、MXFP4 E2M1 这种名字，可能一头雾水。其实命名规则很直观：

  • E = Exponent（指数位）：决定数值的大小范围，E 越大能表示的数越大/越小
  • M = Mantissa（尾数位）：决定数值的精度，M 越大精度越高

所以 FP8 E4M3 就是：8 位浮点数，4 位指数 + 3 位尾数（还有 1 位符号位）。MXFP4 E2M1 就是：4 位微缩浮点数，2 位指数 + 1 位尾数（还有 1 位符号位）。

MXFP 中的 MX = Microscaling（微缩），表示这是一种"分组共享指数"格式——多个数据元素共享同一个 Scale 因子，进一步压缩数据体积。

A5 Cube MMAD 支持的数据类型

格式A2/A3 支持A5 支持备注
FP16 / BF16 / HF32 / FP32 / S8✅✅通用类型，两代都支持
FP8 E5M2 / E4M3❌✅支持静态/动态量化
MXFP8 E5M2/E4M3❌✅32 个 Data 共享 1 个 Scale
MXFP4 E2M1/E1M2❌✅4-bit 浮点，极致压缩
HiF8❌✅华为自定义 FP8 格式

MXFP（Microscaling Floating Point） 是 A5 引入的一种"分组共享指数"格式：32 个数据元素共享 1 个 Scale 因子。这样做的好处是进一步压缩数据体积——32 个数只需要额外存 1 个 Scale，而不是每个数都带完整的指数位。在 Tiling 设计时，需要把 scaleFactor 作为独立的数据来处理，不能和 Data 混在一起搬运。

1.2.5 数据排布格式

同样的一批数据，在内存中可以按不同的顺序排列。就像你刷大绿书，同样一堆帖子，按"最新"排和按"最热"排，你刷到想看内容的效率完全不同。数据排布格式决定了 NPU 访问内存时的效率——如果排布方式和硬件的计算模式匹配，就能连续读取、一次搬完；不匹配就会产生大量碎片化的随机访问，效率大打折扣。

格式维度顺序 / 说明用途备注
ND任意维度，Row-major（行优先）最通用的格式，Ascend C 中最常用
NCHWBatch-Channel-Height-Width卷积、图像处理PyTorch 默认格式
NHWCBatch-Height-Width-Channel卷积、图像处理TensorFlow 默认格式
NCDHWBatch-Channel-Depth-Height-Width3D 卷积场景
NC1HWC0 (5HD)将 Channel 维拆为 C1（外层）+ C0（内层），C0=16卷积输入特征图昇腾专用，匹配 Cube 的 16×16 计算粒度
FracZ按 Cube 计算分块重新排列全连接层权重昇腾专用
FracNZFracZ 的改进版，支持双维分块Transformer QKV 权重昇腾专用

ND 格式是 Ascend C 开发中最常用的格式。它不限制维度数和维度含义，数据按行优先（Row-major）顺序连续存放——就像 C 语言中二维数组 a[i][j] 的存储方式：第 0 行所有元素、第 1 行所有元素……依次排列。

表中标注"昇腾专用"的格式是为了匹配 Cube 单元（矩阵乘单元）的硬件计算需求而设计的。Cube 单元一次计算的是一个固定大小的矩阵块（如 16×16），如果数据按 ND 格式存放，每次取一个矩阵块就需要跨行跳跃读取，效率低。专用格式把数据预先按 Cube 的计算块重排，让硬件可以连续读取。

格式转换路径

格式转换本身是有开销的（需要搬运和重排数据），所以好的算子设计应尽量减少不必要的格式转换。如果一个算子的输入已经是 5HD 格式，就没必要先转回 ND 再处理。

1.2.6 对齐要求

昇腾 NPU 的内存访问有一个硬性要求：数据必须 32 字节（32B）对齐。这意味着每次从 Global Memory 搬运数据到 UB 时，搬运的起始地址和数据长度都必须是 32B 的整数倍。

为什么是 32B？因为 NPU 的 MTE（Memory Transfer Engine）一次搬运的最小粒度就是 32 字节——就像你在狗西购物凑满减，满 199 减 30，你购物车里 198 块的东西得硬凑个 1 块钱的数据线才能享受优惠。MTE 也一样，不够 32 字节的数据得用"填充物"补齐才能发车。

对齐粒度的计算方式：

对齐粒度（元素个数）= 32B / 每个元素的字节数
数据类型每元素字节数对齐粒度（元素个数）
FP32 (float)4B8 个元素
FP16 (half)2B16 个元素
BF162B16 个元素
INT81B32 个元素

对齐后长度的计算公式：

对齐后长度 = ceil(原始长度 / 对齐粒度) × 对齐粒度

以 swap 算子为例，如果使用 FP32（float）类型，每个 float 占 4 字节，所以对齐粒度 = 32 / 4 = 8 个 float。如果向量长度不是 8 的倍数，就需要在末尾补齐（padding）到 8 的倍数才能用 DataCopy 搬运。

更多对齐规则的细节见 附录 B：对齐规则速查。

1.2.7 量化（Quantization）

量化是将高精度数据（如 FP32）映射到低精度数据（如 INT8）的技术。目的是减小模型体积、加速推理。

类比一下：一幅 2400 万像素的照片（FP32）占 70MB，压缩成 JPEG（INT8）只占 3MB，虽然损失了一些细节，但人眼看起来差别不大。

量化公式

q = round(x / scale) + zero_point

其中 scale（缩放因子）的计算方式为：

scale = (max_val - min_val) / (qmax - qmin)

max_val 和 min_val 是原始数据的范围，qmax 和 qmin 是目标量化类型的范围（如 INT8 是 -128 到 127）。

对称 vs 非对称

类型zero_point量化范围适用场景
对称量化0[-max, max]权重（通常以 0 为中心对称分布）
非对称量化≠0[min, max]激活值（ReLU 输出全为非负，不对称）

对称量化更简单高效（不需要处理 zero_point），但要求数据分布以 0 为中心。非对称量化更灵活，但计算时需要额外处理 zero_point 的偏移。

按粒度分类

量化的"粒度"指的是多少元素共享一组 scale 和 zero_point：

  • Per-tensor：整个张量共享一组 scale/zero_point。最粗糙，精度损失最大，但实现最简单。
  • Per-channel：每个输出通道一组 scale/zero_point。精度更好，是权重量化的常用方案。
  • Per-group：每 G 个元素一组 scale/zero_point（如 G=128）。精度最好，是大模型量化的主流方案（如 GPTQ、AWQ 都用分组量化）。

1.2.8 反量化（Dequantization）

反量化是量化的逆过程——把低精度数据还原回高精度：

x_approx = (q - zero_point) × scale

注意这里写的是 x_approx 而不是 x，因为反量化是有损的——量化时 round() 操作引入了舍入误差，反量化无法恢复原始值。精度损失的大小取决于量化方案：粒度越细（Per-group < Per-channel < Per-tensor），损失越小。

在实际推理流程中，量化和反量化经常成对出现：输入用 INT8 计算（快），输出反量化回 FP16/FP32（准）。

1.2.9 伪量化（Fake Quantization）

伪量化是一种训练时模拟量化误差的技术，也叫"量化感知训练（Quantization-Aware Training, QAT）"。

核心思想：在训练阶段，前向传播时做一次"量化→反量化"（模拟推理时的精度损失），让模型在训练过程中就学会适应这种噪声；反向传播时，梯度直接穿过量化操作（因为 round() 函数不可导，用 Straight-Through Estimator 近似为恒等映射）。

这样训练出来的模型，在真正部署为 INT8 推理时，精度损失会比"训练后直接量化（PTQ）"小很多。
