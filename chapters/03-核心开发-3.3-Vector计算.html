3.3.1 Vector 计算单元

Vector 单元是昇腾 NPU 上的向量计算引擎，每个时钟周期可以处理一条 256 字节宽的向量指令。你可以把它想象成一条"超宽的流水线"：普通 CPU 一次处理 1 个或几个浮点数，Vector 单元一次处理一整排——256 字节可以装下 128 个 FP16 或 64 个 FP32 元素。

Vector 单元支持多种运算类型：算术运算（加减乘除）、比较运算（取大取小）、归约运算（求和求最值）、类型转换（FP32↔FP16↔BF16）等。它是逐元素算子（Add、Mul、Exp 等）的核心执行引擎。

值得一提的是，并非所有算子都需要 Vector 计算。我们的贯穿示例 swap 就是一个典型的反例：swap 的功能是交换两个向量的对应元素，整个过程只涉及数据搬运（从 GM 搬到 UB，交换后再搬回 GM），完全不需要 Vector 单元参与。swap 的执行逻辑里只有 DataCopy 调用，没有任何 Add、Mul 之类的 Vector API。

这提醒我们一个重要的设计原则：选择计算单元要看算子的实际需求。纯搬运型算子（swap、copy）、索引型算子（argmax）等，可能根本不需要 Vector 单元。把 Vector 留给真正需要它的算子，才能充分发挥芯片算力。

3.3.2 逐元素运算

逐元素运算是最简单的计算模式：对输入 Tensor 的每个元素独立地做相同的运算，输出 Tensor 的每个位置只依赖对应位置的输入。就像一条流水线上的工人，每个人只负责加工自己面前的零件，互不干扰。

常用逐元素 API

API功能示例
Add逐元素加C = A + B
Sub逐元素减C = A - B
Mul逐元素乘C = A * B
Div逐元素除C = A / B
Max逐元素取大C = max(A, B)
Min逐元素取小C = min(A, B)
Abs逐元素取绝对值B = |A|
Exp逐元素指数B = exp(A)
Ln逐元素对数B = ln(A)
Sqrt逐元素开方B = sqrt(A)
Reciprocal逐元素倒数B = 1/A

这些 API 的调用格式基本一致：API(dst, src0, src1, length, repeatTimes, repeatParams)。其中 length 是要处理的元素总数，repeatTimes 是向量指令的重复次数（详见下一节）。

标量运算

除了向量与向量的运算，Vector 单元还支持向量与标量的运算。比如 Adds 给每个元素加一个常数，Muls 给每个元素乘一个常数：

// C[i] = A[i] + scalar
Adds(dst, src, scalar, length, repeatTimes, repeatParams);

// C[i] = A[i] * scalar
Muls(dst, src, scalar, length, repeatTimes, repeatParams);

标量运算在深度学习推理中非常常见，比如 BatchNorm 的缩放和偏移、量化中的 scale 和 zero-point 等。

3.3.3 repeat 与 mask 机制

Vector 指令的最小处理单位是 256 字节（一条向量）。但实际数据量可能远大于 256B，也可能不足 256B。repeat 和 mask 机制就是用来处理这两种情况的。

repeat 机制

当数据量超过 256B 时，需要让向量指令重复执行多次。每次 repeat 处理 256B 数据，N 次 repeat 就处理 N×256B。

举个例子：处理 1024 个 FP16 元素，总共 2048 字节。每次 repeat 处理 256B，需要 ceil(2048 / 256) = 8 次 repeat：

Add(dst, src0, src1, 1024, 8, {1, 1, 1, 8, 8, 8});

最后一个参数 {1, 1, 1, 8, 8, 8} 是 repeat 参数包，控制源和目标的地址步进方式。在大多数简单场景下，源和目标的步进倍数都设为 repeat 次数即可。

mask 机制

当数据量不是 256B 的整数倍时，最后一次 repeat 会多出一些"空位"。mask 机制用一个位掩码告诉硬件：哪些元素需要计算，哪些跳过。

比如处理 100 个 FP16 元素（200 字节），不足一条完整的 256B 向量（128 个 FP16 元素）。这时 mask 设为 100，表示只计算前 100 个元素，后 28 个空位跳过：

Add(dst, src0, src1, 100, 1, {100, 1, 1, 8, 8, 8});

repeat + mask 组合

实际使用中，repeat 和 mask 经常配合使用。前 N-1 次 repeat 用满 mask（每次处理完整的 256B），最后一次 repeat 用部分 mask（只处理剩余的尾部数据）。

这种组合在 Tiling 设计中非常重要：如果你的 tileSize 恰好是 256B 的倍数，就不需要 mask，效率最高；如果有余数，就需要用 mask 处理尾部，会有一点额外开销。

3.3.4 归约运算（Reduction）

归约运算是沿某个维度聚合数据，把多个元素"压缩"成一个或几个结果。比如求和（ReduceSum）把所有元素加起来，求最大值（ReduceMax）找出最大的那个。

常用归约 API

API功能
ReduceSum求和
ReduceMax求最大值
ReduceMin求最小值

归约的实现思路

归约运算的特殊之处在于：输出元素依赖多个输入元素，不能像逐元素运算那样随意切分。一个常用的策略是两阶段归约：

  1. 核内归约：每个 AI Core 对自己分到的那段数据做局部归约，得到一个中间结果。
  2. 跨核归约：所有核把中间结果写入 GM 的 workspace 区域，最后由一个核（通常是核 0）读取所有中间结果，做全局归约。

两阶段归约的难点在于跨核同步：必须等所有核都写完中间结果之后，才能开始全局归约。这通常通过 workspace 中的标志位或原子操作来实现。

3.3.5 类型转换

类型转换在不同数据类型之间架起桥梁。深度学习模型中经常需要在 FP32、FP16、BF16、INT8 等类型之间切换，比如混合精度训练中前向用 FP16 加速、反向用 FP32 保精度。

常用转换 API

API功能
Cast通用类型转换

精度转换注意事项

类型转换不是无损的，需要注意以下几点：

  • FP32 → FP16/BF16：精度必然有损失，因为 FP16 只有 10 位尾数，BF16 只有 7 位尾数。如果原始数据的动态范围很大，截断误差可能很显著。
  • FP16 → BF16：不能直接转换，需要先转成 FP32（保留完整精度），再从 FP32 转成 BF16。这是因为 FP16 和 BF16 的指数位宽度不同，直接转换会丢失信息。
  • 量化/反量化：本质上也是类型转换（浮点↔整数），但涉及 scale 和 zero-point 的计算，比普通 Cast 更复杂。

3.3.6 Vector 计算最佳实践

要让 Vector 计算发挥最大效率，有几个关键原则：

对齐优先：尽量让每轮处理的数据量对齐到 256B 的倍数，避免 mask 开销。在 Tiling 阶段就把 tileSize 设计成 256B 对齐的值，可以省去运行时的 mask 计算。

计算融合：如果一个算子需要多步 Vector 运算（比如先乘再加），尽量让数据留在 UB 中连续执行，不要中间搬回 GM 再搬进来。每次 GM↔UB 搬运都有延迟，减少搬运次数就是提升性能。

RegBase 优化（A5）：A5 架构支持 RegBase 模式，可以更精细地控制寄存器分配，进一步优化多步融合计算。RegBase 模式下，开发者手动管理 UB 中的寄存器，避免框架自动分配带来的开销。适合对性能要求极高的场景。

更多关于 Vector 计算的高级用法（如 RegBase 模式、Vector Function），详见 6.2 进阶主题。
