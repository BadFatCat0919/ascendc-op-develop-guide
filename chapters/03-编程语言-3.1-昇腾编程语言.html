3.1 昇腾编程语言

3.1.1 昇腾编程语言的演进

昇腾 NPU 从诞生到现在，编程语言经历了几代演进。每一代语言的出现，都是在"开发效率"和"性能控制"之间重新找平衡的结果。了解这段演进史，有助于你理解为什么今天要用 Ascend C，以及它在整个生态中处于什么位置。

用奶茶店来类比：最早的时候，你得自己种茶树、炒茶叶、调配方（机器指令），效率极低但完全可控。后来有了半成品原料包（CCE），省了一些工序但还是要自己动手。再后来有了全自动奶茶机（TBE），按个按钮就出杯，但口味只能从预设菜单里选。现在有了智能调饮台（Ascend C），既可以一键出杯，也可以手动微调每个步骤。最新的中央厨房调度系统（PyPTO），则能同时管理多家门店的出品。

3.1.2 五代编程语言

(1) 机器指令

最底层的编程方式，直接编写 NPU 的二进制指令。就像用 0 和 1 写程序——效率最高（每一条指令都精确到硬件行为），但开发效率极低，只有芯片设计者自己才会用。对普通开发者来说，这基本等于用汇编语言写操作系统。

(2) CCE（C Cube Engine）

华为早期的底层编程框架。CCE 直接操作 Cube 和 Vector 单元，需要手动管理流水线、同步信号和 Buffer 分配。你可以把它理解为"用 C 语言写汇编"——有了一些抽象，但离硬件还是很近，开发门槛极高。写一个矩阵乘算子，你得自己安排数据从 GM 到 L1 到 L0A/L0B 的每一步搬运，手动设置每条流水线的 SetFlag/WaitFlag 同步，稍有不慎就会死锁。CCE 的性能上限很高，但能驾驭它的人极少。

(3) TBE（Tensor Boost Engine）

基于 Python 的 DSL（领域特定语言），用 TVM 编译框架做自动调度。开发者只需要用 Python 描述"做什么"（比如 C = A × B），TVM 的编译器会自动决定数据怎么切分、怎么搬运、怎么安排流水线。开发效率大幅提升——写一个算子可能只需要几十行 Python 代码。但代价是灵活性受限：编译器自动生成的调度策略不一定是最优的，对于复杂的融合算子或特殊的访存模式，性能优化空间有限。就像全自动奶茶机——标准饮品一键出杯，但想搞个创意特调就没辙了。

(4) Ascend C

当前主推的 C++ 编程框架，也是本书的重点。Ascend C 在开发效率和性能控制之间找到了一个很好的平衡点：它提供了高层 API（如 DataCopy、Add、MatMul）让常见操作变得简单，同时保留了底层优化能力（如手动管理 Buffer、控制流水线、选择编程模型）。用 C++ 编写，编译后直接在 NPU 上运行，没有 Python 解释器的开销。

Ascend C 的核心优势：

  • 开发效率高：高层 API 封装了常见的数据搬运和计算操作，不需要像 CCE 那样手动管理每个细节。
  • 性能可控：开发者可以精确控制 Tiling 策略、Buffer 分配、流水线编排，性能优化空间大。
  • 生态完善：与 PyTorch（通过 torch_npu）和 MindSpore 深度集成，算子开发完可以直接在框架中调用。
  • 跨架构兼容：同一份代码可以编译到 A5 以前的架构（arch22）和 A5（arch35），通过条件编译利用各架构的特有特性。

(5) PyPTO（Parallel Tensor/Tile Operation）

面向复杂融合算子和整网模型的 MPMD 编程框架。PyPTO 采用基于 Tile 的编程模型，用 Python 描述计算逻辑，通过多层次编译优化将代码编译为高效的 NPU 可执行代码，并以 MPMD 方式调度到设备上——不同的 AI Core 可以运行不同的计算逻辑。

PyPTO 适合的场景比较特殊：当你的需求超出了单个算子的范畴（比如需要开发一个复杂的融合算子，甚至整个模型网络），或者需要不同核运行不同的计算逻辑时，PyPTO 是更好的选择。对于日常的单个算子开发，Ascend C 就足够了。

也就是说，无论你用哪种语言写算子，最终都要经过毕昇编译器编译为 CCE 指令（一种中间表示），再由后端编译器生成 NPU 可执行的二进制文件。CCE 指令是连接上层编程语言和底层硬件的桥梁——这也是为什么 CCE 虽然作为编程框架在逐步淘汰，但 CCE 指令本身仍然是整个编译链的核心。

Q：既然最终都编译成 CCE 指令，那不同语言写的算子性能一样吗？

A：不一定。编译器能自动优化的程度不同。Ascend C 允许开发者手动控制 Tiling、Buffer 分配和流水线编排，编译器可以直接把这些决策翻译成高效的 CCE 指令。TBE 依赖 TVM 自动调度，编译器生成的 CCE 指令不一定是最优的。所以同样的算子，Ascend C 手写的版本通常比 TBE 自动生成的版本性能更好。

3.1.6 Ascend C 的优势

对于绝大多数算子开发场景，Ascend C 是最通用的选择。原因很简单：

  • 够用：Ascend C 提供的 API 足以覆盖 95% 以上的算子开发需求，从简单的逐元素运算到复杂的矩阵乘融合算子。
  • 够快：C++ 编译为原生代码，没有运行时解释开销；开发者可以精确控制数据搬运和计算流水线，性能上限高。
  • 够灵活：支持 SIMD-MemBase、SIMD-RegBase、SIMT 三种编程模型，可以根据算子特点选择最优方案。
  • 够方便：与主流框架（PyTorch、MindSpore）深度集成，算子开发完可以直接在模型中使用。

如果你的需求是开发复杂的融合算子或整网模型，并且需要 MPMD（不同核运行不同代码）的能力，可以考虑 PyPTO。但对于日常的算子开发工作，Ascend C 就是你的主力工具。

Q：TBE 和 Ascend C 可以同时使用吗？

A：可以，但不建议混用。TBE 和 Ascend C 是两套独立的编程体系，编译工具链不同。新项目建议直接使用 Ascend C，老项目如果已经有 TBE 实现且运行良好，可以暂时保留，但长期来看迁移到 Ascend C 是更好的选择。

Q：CCE 写的算子还能在最新的 A5 芯片上跑吗？

A：CCE 代码的兼容性取决于具体的指令集。A5（DAV_3510）引入了新的指令和编程模型，部分 CCE 代码可能需要适配。建议新开发的算子直接使用 Ascend C，它可以充分利用 A5 的新特性（SIMT、RegBase 等）。
