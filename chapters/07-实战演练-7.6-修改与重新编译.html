4.2c 修改与重新编译

前两节我们完成了 swap 算子的获取、编译、运行和结果查看。本节将通过两个动手练习，让你熟悉修改代码 → 重新编译 → 重新运行的开发循环，这是日常算子开发中最频繁的操作。

练习一：添加调试日志

在 Kernel 中添加 printf 是最直接的调试手段。NPU 上的 printf 输出会记录在 plog 日志中。

# 1. 打开 swap_kernel.cpp
vim blas/swap/arch35/swap_kernel.cpp

# 2. 在 Process 函数中添加日志
# 找到 Process() 函数，添加：
printf("Kernel started, block_idx=%d, totalN=%d\n", block_idx, tiling.totalN);

# 3. 重新编译（增量编译，速度更快）
bash build.sh --ops=swap --soc=ascend950

# 4. 重新运行测试
cd test/swap && bash run_test.sh

# 5. 查看日志中的新输出
grep "Kernel started" plog/plog-*.log

Q&A：NPU 上的 printf 输出在哪里？

Q：我在 Kernel 代码里加了 printf，但终端上看不到输出？

A：NPU 上的 printf 不会直接输出到终端，而是写入 plog 日志文件（位于运行目录下的 plog/ 子目录）。使用 grep 在 plog 文件中搜索你的日志关键词即可。注意：过多的 printf 会显著影响性能，调试完成后记得删除。

练习二：修改数据规模

通过修改测试配置文件，可以添加自定义的测试用例来验证不同数据规模下的算子行为。

# 1. 打开测试配置文件
vim test/swap/swap_test.csv

# 2. 添加一行新的测试用例
# 格式：dtype,N,incx,incy
# 添加：float32,8192,1,1

# 3. 重新运行测试（无需重新编译！）
bash run_test.sh

# 4. 查看新测试用例的结果
cat results/swap_test_report.txt | grep 8192

Q&A：修改测试数据需要重新编译吗？

Q：我只是修改了 CSV 测试配置，没有改算子代码，还需要重新编译吗？

A：不需要。CSV 文件是运行时读取的测试参数配置，不参与编译。只有修改了 .cpp 或 .h 源码文件才需要重新编译。但如果修改了 CMakeLists.txt 或切换了目标芯片，则需要加 --clean 做全量编译。

增量编译 vs 全量编译

场景命令耗时
只修改了 Kernel 代码bash build.sh --ops=swap --soc=ascend950~10 秒
修改了 CMakeLists.txtbash build.sh --ops=swap --soc=ascend950 --clean~30 秒
切换了目标芯片bash build.sh --ops=swap --soc=ascend910 --clean~30 秒

开发流程总结

至此，你已经体验了算子开发的完整循环。以下是核心要点回顾：

  • 获取代码：从 ops-blas 仓库克隆，按 blas/{算子名}/arch{XX}/ 定位源码
  • 编译：bash build.sh --ops={算子名} --soc={芯片型号}，产物在 out/ 目录
  • 运行测试：ST 框架自动验证正确性，结果与 CPU golden 比对
  • 查看结果：测试报告在 test/ 目录，运行日志在 plog/ 目录
  • 修改迭代：改代码 → 编译 → 测试，形成开发闭环
