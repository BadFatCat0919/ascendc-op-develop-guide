2.3.1 算子的两层结构

一个 Ascend C 算子由两层代码组成：Host 侧运行在 CPU 上，Kernel 侧运行在 NPU 的 AI Core 上。

可以用丑团外卖来理解：Host 是你在 APP 上下单（准备参数、选好地址、确认支付），Kernel 是骑手（接到单后去店里取餐、送到你手上）。你不需要知道骑手走哪条路，只需要把订单信息准备好就行。

2.3.2 Kernel 函数签名

Kernel 是运行在 AI Core 上的核函数，由 Host 侧通过 <<<>>> 语法启动。

标准签名

extern "C" __global__ __aicore__ void my_op_kernel(
    GM_ADDR x,          // 输入数据的 GM 地址
    GM_ADDR y,          // 输出数据的 GM 地址
    GM_ADDR workspace,  // 工作空间
    GM_ADDR tiling      // Tiling 参数
)

这个签名中有几个关键修饰符：

  • extern "C" — 使用 C 链接，防止 C++ 的 name mangling（名称修饰）。如果不加，C++ 编译器会把函数名改成一串包含参数类型的编码，导致 Host 侧找不到这个函数。
  • __global__ — 标记这是一个可以从 Host 侧启动的全局核函数。
  • __aicore__ — 标记这个函数运行在 AI Core 上，编译器会用昇腾的指令集来编译它。
  • GM_ADDR — 全局内存地址指针，指向 HBM 上的数据。Kernel 通过它来访问 Host 侧传入的输入、输出、工作空间和 Tiling 参数。

典型 Kernel 实现

extern "C" __global__ __aicore__ void my_kernel(
    GM_ADDR x,
    GM_ADDR y,
    GM_ADDR workSpace,
    GM_ADDR tilingGm)
{
    KERNEL_TASK_TYPE_DEFAULT(KERNEL_TYPE_AIV_ONLY);
    MyOpAIV op;
    op.Init(x, y, tilingGm);
    op.Process();
}

KERNEL_TASK_TYPE_DEFAULT(KERNEL_TYPE_AIV_ONLY) 告诉硬件只启动 AIV（AI Vector）核，不启动 AIC（AI Cube）核。这是因为纯数据搬运算子不需要矩阵乘能力。MyOpAIV 是算子的核心类，Init 负责初始化（解析 Tiling、设置 GM 地址、分配 Buffer），Process 负责执行计算逻辑。

Kernel 内部流程

{
    // 1. 获取当前核编号
    auto blockIdx = AscendC::GetBlockIdx();

    // 2. 解析 Tiling 数据（直调方式）
    __gm__ MyTilingData* tilingPtr =
        reinterpret_cast<__gm__ MyTilingData*>(tilingGm);
    uint32_t totalN = tilingPtr->totalN;
    uint32_t perCoreN = tilingPtr->perCoreN;

    // 3. 计算当前核负责的数据范围
    uint32_t offset = blockIdx * perCoreN;

    // 4. 搬入 → 计算 → 搬出
    // ...（后续章节详细讲解）
}

框架宏 vs 手动解析：上面的 reinterpret_cast 是直调方式的写法。如果使用 aclnn 框架，则通过 GET_TILING_DATA(tilingData, tiling) 宏来解析。两种方式本质上都是把 GM 中的 Tiling 结构体读出来，只是封装程度不同。详见 2.4 直调开发方式。

每个 AI Core 通过 GetBlockIdx() 知道自己的编号，再结合 Tiling 数据算出自己该处理哪段数据。这就是 SPMD 模型在 Kernel 内部的具体体现。

2.3.3 TilingData 结构体

TilingData 是 Host 和 Kernel 之间的"通信协议"——Host 侧把计算好的 Tiling 参数填入这个结构体，Kernel 侧从 GM 中读取并解析它。

你可以把 Tiling 理解为雪王奶茶的分单策略：公司团建点了 totalN 杯奶茶，有 coreNum 家门店来分（核数），每家分 perCoreN 杯，最后可能剩下 remainder 杯分给某几家门店多喝几杯。

典型定义

struct SswapTilingData {
    uint32_t totalN;     // 向量总元素数
    uint32_t perCoreN;   // 每个核处理的基础元素数
    uint32_t remainder;  // 余数（totalN % coreNum）
    uint32_t tileSize;   // 每次迭代搬运的元素数（受 UB 容量限制）
};

constexpr uint32_t ELEMENTS_PER_BLOCK = 32 / sizeof(float);

ELEMENTS_PER_BLOCK 是 32 Bytes 对齐要求下的最小搬运单位——对于 float（4 Bytes），每次至少搬 8 个元素。

Q&A：常见疑问

Q：为什么 Kernel 函数必须加 extern "C"？

A：C++ 编译器会对函数名做 name mangling（名称修饰），把参数类型编码到函数名中。例如 my_kernel(int, float) 可能变成 _Z9my_kernelif。Host 侧通过原始函数名查找 Kernel，如果名字被修饰了就找不到。extern "C" 告诉编译器保持原始函数名。

Q：TilingData 结构体为什么不能包含指针？

A：TilingData 通过 GM 从 Host 传递到 Kernel。Host 侧的指针指向 CPU 内存地址空间，而 Kernel 运行在 NPU 上，两者的地址空间不同。因此 TilingData 只能包含基本数据类型（如 uint32_t），不能包含指针。数据本身通过 GM_ADDR 参数单独传递。

Q：KERNEL_TYPE_AIV_ONLY 和 KERNEL_TYPE_DEFAULT 有什么区别？

A：KERNEL_TYPE_AIV_ONLY 只启动 AI Vector 核，适合纯向量运算（如 add、swap）。KERNEL_TYPE_DEFAULT 同时启动 AIV 和 AIC（AI Cube）核，适合需要矩阵乘的场景。选错了会导致计算单元不可用。
