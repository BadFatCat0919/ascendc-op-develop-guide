1.1 算子

1.1.1 算子是什么

想象你在用丑图秀秀修图。一张自拍从导入到导出，要经过好几步操作：美颜、瘦脸、大眼、加滤镜。每一步操作就是一个算子（Operator）——它接收上一步的图像（数据），处理一下，传给下一步。你写的 PyTorch 模型就是那张修图流程图，框架就是丑图秀秀的引擎，而算子就是每一个实际执行的修图步骤。torch.add(a, b) 就是"调亮度"，torch.matmul(A, B) 就是"应用滤镜矩阵"。

用更正式的话说：算子是对张量（多维数组）执行的一种数学运算，它是深度学习框架中最小的计算单元。你在 PyTorch 里写 torch.add(a, b)，这个 add 就是一个算子；写 torch.matmul(A, B)，这个 matmul 也是一个算子。

1.1.1.1 算子 vs 函数 vs 层

这三个词经常混用，但它们的视角不同：

概念视角举例
函数编程概念add(a, b) 是一个 C++ 函数
层（Layer）网络结构概念"卷积层"是网络架构图里的一个方框
算子（Operator）硬件执行概念Conv2D 算子是调度 NPU 完成卷积运算的代码集合（包含 Host 侧和 Kernel 侧）

一个"层"往往包含多个算子。比如一个典型的"卷积层"实际上由三个算子串联而成：Conv2D（卷积计算）→ BiasAdd（加偏置）→ ReLU（激活函数）。框架在调度时，会把它们拆成独立的算子节点依次执行。

1.1.1.2 算子在框架中的角色

当你在 PyTorch 里写模型代码时，框架会在内部构建一张计算图（Compute Graph）——图中的每个节点就是一个算子，边代表数据的流向。

  • 前向传播：框架按拓扑序依次执行每个算子，从输入数据算出预测结果。
  • 反向传播：框架执行每个算子的"梯度算子"，从损失函数倒推出每个参数的梯度。

1.1.1.3 常见算子分类

按照计算特征，算子可以分为以下几大类：

类别典型算子主要计算单元特点
逐元素（Element-wise）Add、Mul、ReLU、SigmoidVector最简单，每个元素独立计算，互不依赖
归约（Reduction）Sum、Max、SoftmaxVector沿某个轴做聚合，比如对一行求和
矩阵乘（MatMul）GEMM、GEMVCube计算密集型，算力是瓶颈
卷积（Convolution）Conv2DCube本质也是矩阵运算（im2col + GEMM）
数据搬运（Data Move）Transpose、Gather、SwapMTE不涉及数值计算，搬运效率是关键

以 swap 算子为例：swap 是一种数据搬运算子，功能是交换两个向量的对应元素——swap(x, y) 执行后，x 变成原来的 y，y 变成原来的 x。它不做任何加减乘除，纯粹是把数据从 A 搬到 B、从 B 搬到 A。对于这类算子，关键不在于"算什么"，而在于"怎么搬得更快"。

1.1.1.4 为什么需要"开发算子"

框架自带的算子库已经很丰富了，为什么还要自己写算子？主要有三个原因：

  1. 框架没覆盖：研究界不断涌现新的算子和机制，比如 DeepSeek 推出的 MLA（Multi-head Latent Attention）注意力机制，框架里还没有现成实现，得自己开发对应的算子。
  2. 性能优化：通用算子为了兼容各种场景，不一定是最快的。针对特定硬件（如 A5 芯片）和特定形状（如固定 batch size）做定制优化，可以快很多。
  3. 算子融合：把多个小算子融合成一个大算子，减少数据在显存和片上缓存之间的来回搬运，显著提升效率。

Q：算子融合具体能带来多大的性能提升？

A：以"Conv2D + BiasAdd + ReLU"为例，如果不融合，三个算子各自从显存读数据、计算、写回显存，数据要搬运 3 次。融合后，数据只需从显存读 1 次，在片上缓存（UB）里依次完成三个操作，最后写回 1 次。搬运开销减少约 2/3，而搬运往往是算子的主要耗时——尤其在数据量大、计算简单的场景下。

关于融合算子的详细介绍（包括分类、知名算子如 FlashAttention 和 MC2、以及设计挑战），请参阅 1.5 融合算子。
