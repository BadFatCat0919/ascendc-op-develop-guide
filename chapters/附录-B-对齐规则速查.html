Ascend C 的数据搬运（DataCopy）有一个硬性要求：每次搬运的数据长度必须是 32 字节（32B）的整数倍。 这条规则贯穿了整个算子开发过程，理解它、遵守它，能帮你避免 80% 以上的"莫名其妙"的错误。

为什么是 32 字节

Da Vinci 核的 MTE（Memory Transfer Engine）在 GM 和 UB 之间搬运数据时，硬件总线宽度是 256 bit = 32 字节。每次 DMA 传输固定搬运 32B，不支持更小的粒度。你可以把它想象成一条传送带——每次只能放一个标准尺寸的箱子，散件必须先装进箱子才能上输送带。

对齐粒度计算公式

对于数据类型 T，对齐粒度（每次搬运的最少元素个数）为：

align_unit = 32 / sizeof(T)

搬运的元素个数必须是 align_unit 的整数倍。

各数据类型对齐粒度

数据类型sizeof对齐粒度（元素数）每次最少搬运
FP32（float）4B88 个 float = 32B
FP16（half）2B1616 个 half = 32B
BF16（bfloat16_t）2B1616 个 bfloat16 = 32B
FP8（float8_t）1B3232 个 float8 = 32B
INT648B44 个 int64 = 32B
INT324B88 个 int32 = 32B
INT162B1616 个 int16 = 32B
INT81B3232 个 int8 = 32B

sswap 示例：FP32 的对齐粒度

sswap 处理的是 FP32 数据，sizeof(float) = 4，所以：

align_unit = 32 / 4 = 8 个元素

sswap 在 sswap_tiling_data.h 中定义了这个值：

constexpr uint32_t ELEMENTS_PER_BLOCK = 32 / sizeof(float);  // = 8

这意味着 sswap 每次 DataCopy 搬运的元素个数必须是 8 的倍数。如果要搬运 100 个元素：

  • 对齐部分：(100 / 8) * 8 = 96 个元素，用 DataCopy 直接搬运。
  • 尾部：100 - 96 = 4 个元素，用 DataCopyPad 补齐到 8 个元素后搬运（多出的 4 个位置填 0，写回时只写前 4 个）。

对齐的两个层面

1. 搬运长度对齐

DataCopy 的第三个参数（元素个数）必须是 align_unit 的整数倍。如果不满足，硬件会报错或静默截断。

// ❌ 错误：搬运 10 个 float，不是 8 的倍数
DataCopy(dst, src, 10);

// ✅ 正确：搬运 8 个 float（对齐部分）
DataCopy(dst, src, 8);
// 剩余 2 个用 DataCopyPad 处理
DataCopyPad(dst[8], src[8], copyParams, padParams);

2. Buffer 地址对齐

UB 上的 Buffer 起始地址也必须是 32B 对齐的。使用 TPipe::InitBuffer 分配的 Buffer 会自动对齐，但如果你手动从 UB 中划分空间（如 arch22 的做法），就需要自己保证对齐。

向上对齐与向下对齐

在 Tiling 设计中，经常需要把某个数值对齐到 align_unit：

向下对齐（floor）：取不超过原值的最大对齐值。用于计算"安全搬运量"。

uint32_t aligned = (value / align_unit) * align_unit;

向上对齐（ceil）：取不小于原值的最小对齐值。用于计算"需要分配的空间"。

uint32_t aligned = ((value + align_unit - 1) / align_unit) * align_unit;

sswap 的 Tiling 函数中，perCoreN 使用的是向下对齐——确保每个核分到的数据量是 8 的倍数，多出来的部分（remainder）留给最后一个核处理。

repeat 与对齐的关系

Vector 指令的 repeatTimes 参数和对齐粒度密切相关。每次 repeat 处理的数据量 = repeatTimes × 128 元素（对于 FP16）或 repeatTimes × 64 元素（对于 FP32）。所以 repeat 的步长本身就隐含了对齐要求。
