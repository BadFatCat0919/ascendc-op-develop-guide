4.2 数据搬运 API 详解

数据搬运是算子执行的前提——数据必须先到达计算单元附近的 Buffer 才能被处理。本节详细介绍 Ascend C 中的四类数据搬运 API：DataCopy、DataCopyPad、CopyIn 和 CopyOut，包括它们的对齐要求、性能特性和使用场景。

4.2.1 DataCopy

DataCopy 是最常用的数据搬运 API，用于在 GM（全局内存）和 UB（统一缓冲区）之间拷贝数据。它由 MTE 引擎执行，不占用 Vector 或 Cube 计算资源。

函数签名

template <typename T>
__aicore__ inline void DataCopy(const LocalTensor<T>& dst,
                                 const LocalTensor<T>& src,
                                 const uint32_t count);
参数类型说明
dstLocalTensor<T>目标 Tensor（GM 或 UB 上的 LocalTensor）
srcLocalTensor<T>源 Tensor（GM 或 UB 上的 LocalTensor）
countuint32_t拷贝的元素个数（不是字节数）

对齐要求

DataCopy 的 count 参数有严格的对齐要求，这是由 MTE 引擎的硬件特性决定的：

数据类型元素大小count 对齐要求说明
FP16 / BF162 字节16 的倍数（32 字节对齐）最少搬运 16 个 FP16 元素
FP32 / INT324 字节8 的倍数（32 字节对齐）最少搬运 8 个 FP32 元素
INT81 字节32 的倍数（32 字节对齐）最少搬运 32 个 INT8 元素

核心原则：DataCopy 的字节数必须是 32 字节的倍数。如果实际数据量不满足对齐要求，需要用 DataCopyPad 补齐，或者用掩码机制处理尾部。

使用示例

DataCopy(ubIn, gmIn, 128);
DataCopy(gmOut, ubOut, 128);

DataCopy 重载形式

DataCopy 还支持带 stride 的重载形式，用于非连续数据的搬运：

template <typename T>
__aicore__ inline void DataCopy(const LocalTensor<T>& dst,
                                 const LocalTensor<T>& src,
                                 const DataCopyParams& params);
DataCopyParams 字段说明
blockCount搬运的数据块数
blockLen每个数据块的元素个数
srcStride源端块间步长（元素数）
dstStride目标端块间步长（元素数）
 
DataCopyParams params;
params.blockCount = 4;
params.blockLen = 32;
params.srcStride = 64;
params.dstStride = 32;
DataCopy(ubIn, gmIn, params);

4.2.2 DataCopyPad

DataCopyPad 在搬运数据的同时自动填充 padding，用于解决数据量不满足对齐要求的场景。

函数签名

template <typename T>
__aicore__ inline void DataCopyPad(const LocalTensor<T>& dst,
                                    const GlobalTensor<T>& src,
                                    const DataCopyPadParams& params);
DataCopyPadParams 字段说明
blockCount数据块数
blockLen每块有效元素个数
padCount每块末尾填充的元素个数
padValue填充值（通常为 0）

使用场景

当 tile 的尾部数据不足一个对齐块时，DataCopyPad 可以自动补齐：

DataCopyPadParams padParams;
padParams.blockCount = 1;
padParams.blockLen = 100;
padParams.padCount = 28;
padParams.padValue = 0;
DataCopyPad(ubIn, gmIn, padParams);

上述代码将 100 个 FP16 元素从 GM 搬到 UB，并在末尾填充 28 个零，使总数达到 128（满足 16 对齐）。

4.2.3 CopyIn / CopyOut

CopyIn 和 CopyOut 专用于 Cube 矩阵计算场景，在 GM 和 L0A/L0B/L0C 之间搬运数据。

CopyIn

template <typename T>
__aicore__ inline void CopyIn(const LocalTensor<T>& dst,
                               const GlobalTensor<T>& src,
                               const uint32_t count);

将数据从 GM 搬到 L0A 或 L0B（Cube 的输入 Buffer）。dst 的类型决定了目标位置：如果 dst 绑定到 L0A，则搬到 L0A；如果绑定到 L0B，则搬到 L0B。

CopyOut

template <typename T>
__aicore__ inline void CopyOut(const GlobalTensor<T>& dst,
                                const LocalTensor<T>& src,
                                const uint32_t count);

将 Cube 计算结果从 L0C 搬回 GM。

对齐要求

CopyIn/CopyOut 的对齐要求与矩阵维度相关。对于 FP16 矩阵，通常要求搬运元素数为 16 的倍数。具体的对齐规则取决于目标架构（A2/A5）和矩阵形状。

4.2.4 性能优化建议

对齐优先

在 Tiling 设计阶段，尽量让每个 tile 的数据量满足 32 字节对齐。这样可以直接使用 DataCopy，避免 DataCopyPad 的额外开销。

减少搬运次数

每次 DataCopy 调用都有固定的启动开销。将多个小块合并为一次大块搬运，比多次小块搬运更高效。

双缓冲重叠

利用双缓冲机制，让 MTE 搬运和 Vector/Cube 计算重叠执行。当 Vector 计算当前 tile 时，MTE 同时搬运下一个 tile 的数据。

避免不必要的 GM 中转

如果一个算子需要多步计算（如先乘再加），让数据留在 UB 中连续执行，不要中间搬回 GM。每次 GM↔UB 搬运都有延迟。

Q&A

Q：DataCopy 的 count 参数是元素个数还是字节数？

A：count 是元素个数，不是字节数。例如搬运 128 个 FP16 元素，count = 128，实际搬运 256 字节。这一点容易与 aclrtMemcpy（字节数）混淆，需要特别注意。

Q：为什么不直接用 DataCopyPad 代替 DataCopy？

A：DataCopyPad 有额外的 padding 计算开销。当数据量已经对齐时，DataCopy 更高效。最佳实践是：在 Tiling 阶段设计好对齐的 tile 大小，运行时用 DataCopy；只有尾部不对齐的数据才用 DataCopyPad。

Q：DataCopy 可以在 Vector 计算的同时执行吗？

A：可以。DataCopy 由 MTE 引擎执行，Vector 计算由 Vector 单元执行，它们是不同的硬件引擎，可以并行工作。这正是双缓冲机制的基础：MTE 搬运下一批数据的同时，Vector 计算当前批数据。
