4.4 swap 算子 Tiling 设计

Tiling（分块）是 Ascend C 算子开发的核心概念。由于 AI Core 的片上内存（UB）有限，我们需要把大规模数据切分成小块，分批次处理。本节将详细讲解 swap 算子的 Tiling 设计。

4.4.1 TilingData 结构设计

TilingData 是 Host 和 Kernel 之间的"通信协议"——Host 计算好 Tiling 参数，通过 TilingData 传递给 Kernel。

swap_tiling_data.h 完整代码

// swap_tiling_data.h
#ifndef SWAP_TILING_DATA_H
#define SWAP_TILING_DATA_H

#include <cstdint>

// Tiling 数据结构定义
// Host 侧计算这些值，传递给 Kernel 侧使用
struct SwapTilingData {
    int32_t totalN;      // 总元素个数（N）
    int32_t perCoreN;    // 每个 AI Core 处理的元素个数
    int32_t remainder;   // 最后一个 Core 需要额外处理的元素个数
    int32_t tileSize;    // 每次搬运的元素个数（UB 分块大小）
    int32_t coreCount;   // 使用的 AI Core 数量
};

#endif // SWAP_TILING_DATA_H

各字段含义

字段类型含义计算方式
totalNint32_t总元素个数直接取参数 N
perCoreNint32_t每个 Core 处理的基础元素数N / coreCount
remainderint32_t最后一个 Core 的额外元素数N % coreCount
tileSizeint32_tUB 单次搬运的元素数UB容量 / 元素大小 / 2
coreCountint32_t使用的 AI Core 数量min(N/minPerCore, maxCore)

4.4.2 多核切分策略

多核切分的目标是把 N 个元素均匀分配到多个 AI Core 上并行处理。

切分算法

// 1. 确定 Core 数量
int32_t maxCoreCount = 20;           // 硬件上限
int32_t minElementsPerCore = 256;     // 每个 Core 至少处理 256 个元素
tiling.coreCount = (N + minElementsPerCore - 1) / minElementsPerCore;
if (tiling.coreCount > maxCoreCount) {
    tiling.coreCount = maxCoreCount;
}

// 2. 计算每个 Core 的工作量
tiling.perCoreN = N / tiling.coreCount;
tiling.remainder = N % tiling.coreCount;

4.4.3 UB 空间规划

UB（Unified Buffer）是 AI Core 的片上高速缓存，容量有限（通常 256KB）。swap 需要在 UB 中同时存放 X 和 Y 两个 Buffer。

UB 空间分配

tileSize 计算公式

// UB 总容量（字节）
int32_t ubSizeBytes = 256 * 1024;  // 256KB

// 需要同时存 X 和 Y，所以每个 Buffer 只能用一半
// FP32 每个元素 4 字节
tiling.tileSize = ubSizeBytes / sizeof(float) / 2;  // = 32768

// 确保 tileSize 不超过 perCoreN（如果 perCoreN 很小，不需要用满 UB）
if (tiling.tileSize > tiling.perCoreN) {
    tiling.tileSize = tiling.perCoreN;
}

为什么 tileSize 要除以 2？

因为 swap 需要同时存储 X 和 Y 两个向量的数据到 UB 中。如果 UB 总容量是 256KB，那么 xBuf 和 yBuf 各占一半，每个 Buffer 只能用 128KB。

Q&A：常见问题

Q1：为什么 Core 数量不是越多越好？

A：如果 N 很小（比如 N=100），用 20 个 Core 意味着每个 Core 只处理 5 个元素。启动 Core 的开销可能比实际计算还大。所以代码里设置了 minElementsPerCore = 256，确保每个 Core 至少处理 256 个元素。

Q2：如果 tileSize 大于 perCoreN 会怎样？

A：如果 tileSize 大于 perCoreN，说明 UB 空间足够大，一次就能搬完一个 Core 的所有数据。此时应该把 tileSize 截断为 perCoreN，避免浪费 UB 空间。

Q3：为什么不把余数分散到所有 Core？

A：分散余数（如前 remainder 个 Core 各多处理 1 个元素）可以让负载更均衡，但会增加代码复杂度。对于 swap 这种简单算子，把余数集中给最后一个 Core 更简单，性能差异可以忽略。
