4.6 同步与流水线

AI Core 内部有多个执行单元并行工作：MTE2 负责从 GM 搬入数据，MTE3 负责把结果搬出 GM，Vector 做向量计算，Cube 做矩阵乘。这些单元各自有独立的硬件资源，可以同时执行不同的任务。

但并行带来了依赖问题。想象你在蓝鹿咖啡当值班经理：前台接单（MTE2 搬入数据）、咖啡师做咖啡（Vector 计算）、出餐口叫号取餐（MTE3 搬出结果），三条线同时运转。问题来了：咖啡师还没做好，出餐口就喊"您的美式好了"——顾客拿到的是一杯空气。更离谱的是，上一杯拿铁的杯子没洗就接着做卡布奇诺——顾客喝到的是拿铁味的卡布奇诺，这在算子里就叫"读到上一轮的脏数据"。

同步机制的作用就是在这些并行单元之间建立"先后顺序"的约束：告诉硬件"必须等 A 做完了，B 才能开始"。

4.6.2 Pipe 概念

每个执行单元有自己的指令流水线（Pipe）。你可以把 Pipe 理解为一条独立的"传送带"：传送带上的指令按顺序执行，但不同传送带之间是异步的。

Pipe执行单元功能
PIPE_MTE2MTE2GM → UB/L1 搬运（输入搬入）
PIPE_MTE3MTE3UB → GM 搬运（输出搬出）
PIPE_MTE1MTE1核内搬运（L1↔L0、L1↔UB）
PIPE_VVector向量计算
PIPE_CCube矩阵乘计算
PIPE_SQUScalar标量计算

同一 Pipe 内的指令是顺序执行的——PIPE_MTE2 上的第一条 DataCopy 一定在第二条之前完成。但不同 Pipe 之间没有天然的顺序保证——PIPE_MTE2 上的 DataCopy 和 PIPE_V 上的 Add 可能同时执行，也可能先后执行，取决于硬件调度。

这就是为什么我们需要显式的同步机制：在 Pipe 之间建立依赖关系，确保数据在被使用之前已经准备好了。

4.6.3 SetFlag / WaitFlag：经典同步机制（A5 以前的架构）

在 A5 以前的架构上，Pipe 之间的同步通过 SetFlag 和 WaitFlag 这对原语实现。你可以把它们想象成蓝鹿咖啡的叫号器：

  • SetFlag：咖啡师做完一杯，按下叫号器——"XX号，您的美式好了！"
  • WaitFlag：出餐口盯着叫号器，灯亮了才能取餐——"我得等那杯做完才能叫号"。

(1) 使用示例

// MTE2 搬完数据后设置信号
SetFlag(PIPE_MTE2, PIPE_V, flag);

// Vector 等待 MTE2 搬完再计算
WaitFlag(PIPE_V, PIPE_MTE2, flag);

// 计算完成后设置信号
SetFlag(PIPE_V, PIPE_MTE3, flag);

// MTE3 等待计算完再搬出
WaitFlag(PIPE_MTE3, PIPE_V, flag);

这段代码建立了一条完整的同步链：MTE2 搬完 → Vector 计算 → MTE3 搬出。每一步都要等前一步的信号才能开始。

(2) 典型同步链

SetFlag/WaitFlag 需要严格配对使用：每个 SetFlag 必须有对应的 WaitFlag，否则会死锁或数据竞争。在复杂的多流水算子中，管理这些信号旗标会变得非常繁琐。

Q：为什么 A5 的 BufferID 机制比 A5 以前的架构的 SetFlag/WaitFlag 更好？

A：A5 以前的架构的 SetFlag/WaitFlag 需要开发者手动配对——一个流水段 Set 了某个 flag，另一个流水段必须 Wait 同一个 flag。如果配对错了就会死锁或数据竞争。A5 的 BufferID 让硬件自动追踪 Buffer 的依赖关系，开发者只需要正确使用 EnQue/DeQue 语义，大幅降低了出错概率。

4.6.4 A5 的 BufferID 同步

A5（DAV_3510）引入了一种更优雅的同步机制：BufferID 同步。它从根本上简化了同步代码的编写方式。

在 BufferID 机制下，每个 Buffer 有一个唯一的硬件 ID。当你通过队列（TQue）操作 Buffer 时，硬件会自动追踪 Buffer 的依赖关系：如果 MTE2 正在往某个 Buffer 写数据，Vector 试图读这个 Buffer 时会自动等待，不需要手动写 SetFlag/WaitFlag。

这消除了 A5 以前的架构上 set/wait 强制配对的需求，大大简化了多流水算子的同步代码。开发者只需要正确使用队列的 EnQue/DeQue 语义，硬件会自动处理底层的同步。

不过需要注意：A5 以前的架构仍然需要使用 SetFlag/WaitFlag，BufferID 是 A5 独有的特性。如果你的算子需要同时支持 A2 和 A5，就需要用条件编译区分两种同步方式。

4.6.5 流水线并行设计

流水线的核心思想是"搬运和计算重叠执行"。就像蓝鹿咖啡的早高峰：咖啡师在做上一杯拿铁的同时，前台已经在接下一单，出餐口正在叫上一杯的号。三个步骤同时进行，整体效率远高于"接完单→做完→叫完号→再接→再做→再叫"的串行模式。

(1) 单 Buffer（无流水线）

最简单的模式：所有操作串行执行，一块 Buffer 从头用到尾。

时间线：
[搬运A] [计算A] [搬出A] [搬运B] [计算B] [搬出B]
                                    ↑ 大量空闲时间

每个阶段都要等前一个阶段完全结束才能开始。MTE2 搬入时，Vector 和 MTE3 都在闲着；Vector 计算时，MTE2 和 MTE3 都在闲着。硬件利用率很低。

(2) 双 Buffer（经典流水线）

把 Buffer 分成两块，交替使用。当 Buffer[0] 在做计算时，Buffer[1] 可以同时在搬入下一轮数据：

时间线：
Buffer[0]: [搬运] [计算] [搬出]          [搬运] [计算] [搬出]
Buffer[1]:        [搬运] [计算] [搬出]          [搬运] [计算] [搬出]
                   ↑ 搬运和计算重叠，利用率更高

双 Buffer 的关键是让不同阶段的操作在不同 Buffer 上同时执行。MTE2 往 Buffer[1] 搬数据的同时，Vector 在 Buffer[0] 上做计算，MTE3 在搬出更早一轮的结果。三个单元同时工作，硬件利用率大幅提升。

(3) N-Buffer

理论上，Buffer 越多，流水线并行度越高。3-Buffer、4-Buffer 可以让更多的操作重叠执行。但 Buffer 数量受限于 UB 空间——每多一块 Buffer，就要多占一份 UB 空间，留给每块 Buffer 的空间就更小，每轮能处理的数据量也越少。

实践中，双 Buffer 是最常用的选择，在流水线效率和 UB 空间之间取得了较好的平衡。对于搬运延迟远大于计算时间的算子，可以考虑 3-Buffer 或更多。

4.6.6 避免数据冲突

数据冲突是指同一块 Buffer 同时被读和写，或者同时被两个操作写入。这会导致数据竞争，结果不可预测。

双 Buffer 通过交替使用天然避免了冲突：当 Buffer[0] 被 Vector 读取时，Buffer[1] 正在被 MTE2 写入，两者互不干扰。下一轮反过来，Buffer[1] 被读取，Buffer[0] 被写入。

多队列设计也可以隔离不同阶段的数据。比如输入队列和输出队列分开管理，搬入操作只影响输入队列的 Buffer，搬出操作只影响输出队列的 Buffer，两者不会冲突。

4.6.7 流水线设计最佳实践

搬运和计算时间均衡：流水线的效率取决于最慢的那个阶段。如果搬运需要 100 个周期，计算只需要 10 个周期，那计算单元大部分时间都在等搬运，流水线形同虚设。理想的状况是搬运和计算时间接近，两者都能充分利用。

使用 TPipe 管理生命周期：TPipe 类封装了 Buffer 初始化、队列管理和流水线控制的全部逻辑。通过 pipe.InitBuffer() 初始化队列，框架会自动处理 Buffer 的分配和对齐。

使用 TQue 队列管理 Buffer：TQue 队列的 EnQue/DeQue 语义不仅管理 Buffer 的分配和释放，还隐含了同步语义（尤其在 A5 上）。正确使用队列可以让同步代码几乎为零。
