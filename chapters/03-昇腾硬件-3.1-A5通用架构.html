3.1.1 昇腾芯片概览

昇腾（Ascend）是华为自研的 AI 处理器系列，从训练到推理覆盖了多种场景。在算子开发中，我们最关心的是芯片的 NpuArch 架构代号，因为它直接决定了我们能使用哪些指令和编程模型。

代号NpuArch芯片型号说明
A2DAV_2201Ascend910B1~B4训练/推理芯片
A3DAV_2201Ascend910_93训练/推理芯片
A5DAV_3510Ascend950DT/PR新一代推理芯片（本文核心）

A2 和 A3 共用 DAV_2201 架构，因此算子代码基本兼容——同一份 arch22 目录下的代码通常可以在这两代芯片上直接运行。A5 则采用全新的 DAV_3510 架构，新增了 SIMT 编程模型、RegBase 向量计算、多种低精度数据类型等重要特性，这些是 A2/A3 所不具备的。

在 ops-blas 算子仓中，不同架构的实现按目录区分：arch22 目录存放 A2/A3 的实现，arch35 目录存放 A5 的实现。本文以 A5（DAV_3510）架构为主线讲解硬件结构，因为它是当前最新的标准架构，也是算子开发的主要目标平台。A5 相比 A2/A3 的差异特性将在 3.2 A5 新特性 中专门介绍。

3.1.2 AI Core：Da Vinci 核

AI Core 是昇腾 NPU 的基本计算单元，地位类似 GPU 中的 SM（Streaming Multiprocessor）。每个 AI Core 内部集成了三种计算单元，可以把它想象成一把王者荣耀里的团战阵容：Cube 是坦克（扛伤害，做最重的矩阵乘），Vector 是射手（持续输出，做逐元素运算），Scalar 是辅助（指挥走位，做控制流和地址计算）。

三大计算单元

计算单元类比擅长场景算力 (A5 Server)
Cube吕布大招（一刀劈一片）矩阵乘加（16×16×16 MAC 阵列）432T (BF16)
Vector流水线工人逐元素运算、归约、类型转换54T (FP16)
Scalar管理员控制流、地址计算-

Cube 是算力担当，专门做矩阵乘法。它内部是一个 16×16×16 的 MAC（乘累加）阵列，每个时钟周期可以完成一次 16×16 的矩阵乘运算。对于 FP8 精度，阵列扩大为 16×32×16，算力翻倍；对于 MXFP4 精度，阵列进一步扩大为 16×64×16，算力再翻倍。

Vector 是通用计算单元，宽度为 256 Bytes（2048 bit），每次可以处理 256 Bytes 的数据（对于 float 就是 64 个元素，对于 half 就是 128 个元素）。当数据总量超过一次处理的宽度时，Vector 通过 repeat（重复执行）和 mask（掩码选择）机制来处理任意长度的向量。

Scalar 负责控制流和地址计算等标量操作，不参与大规模数据计算。

sswap 是纯数据搬运算子（交换两个向量的元素），只做 CopyIn 和 CopyOut，不涉及加减乘除。因此它只使用 Vector 单元来驱动数据搬运（MTE2 搬入、MTE3 搬出），完全不需要 Cube 单元。这也是为什么 sswap 的 kernel 入口使用 KERNEL_TASK_TYPE_DEFAULT(KERNEL_TYPE_AIV_ONLY)——只启动 AIV（AI Vector）核，不启动 AIC（AI Cube）核。

Cube 单元详解

Cube 的核心是一个脉动阵列（Systolic Array），在 FP16/BF16 精度下规模为 16×16×16。你可以把它理解为王者里吕布的大招：蓄力（数据从 L0A/L0B 送入）→ 一刀劈下去（16×16 的矩阵乘一步完成）→ 范围内的敌人全被打到（结果写入 L0C）。每个时钟周期，阵列中的每个乘加单元都在做一次乘法和一次累加，数据像波浪一样从一端流向另一端。

FP8 精度下阵列规模变为 16×32×16（K 维度翻倍），MXFP4 下变为 16×64×16（K 维度再翻倍），精度越低、吞吐越高。

Vector 单元详解

Vector 单元的物理宽度是 256 Bytes（2048 bit），一次指令操作的数据量由 VL（Vector Length）决定，最大为 256 Bytes。以 float（4 Bytes）为例，一次可以处理 64 个元素；以 half（2 Bytes）为例，一次可以处理 128 个元素。

当向量长度超过单次处理能力时，Vector 通过 repeat 参数指定重复执行次数，通过 mask 参数指定每次执行时哪些元素有效。这套机制让 Vector 能够处理任意长度的向量，而不需要程序员手写循环。

3.1.3 存储层次

昇腾的存储层次类似 CPU 的 L1/L2/L3 cache，但有一个关键区别：缓存由程序员显式管理。CPU 的 cache 对程序员透明，由硬件自动替换；而昇腾的各级 Buffer 需要程序员通过 API 显式搬运数据。

把 AI Core 想象成一家雪王奶茶的门店：GM 是城市郊区的总仓库——东西多到堆成山（64GB），但叫个货拉拉过来得等半天。L1 是门店后厨的冷库——容量还行（512KB），走两步就到。L0A/L0B 是吧台上的原料盒——容量小但伸手就够着。UB 是调饮师面前的操作台——所有调配动作都在这里完成。你的工作就是合理安排"什么时候从总仓进货、什么提前备到冷库、什么摆在吧台上"，让调饮师（Vector 单元）不用等料就能连续出杯。

各层存储容量

BufferA2 (910B2)A5 (950PR)用途
GM64 GB112~128 GB全局存储
L2192 MB112~128 MB跨核共享
L1512 KB512 KBCube 输入缓存
L0A64 KB64 KBCube 左矩阵
L0B64 KB64 KBCube 右矩阵
L0C128 KB256 KBCube 输出（A5 翻倍）
UB192 KB248 KBVector 工作区
BT1 KB4 KBFixPipe Bias

注意 A5 相比 A2 有两处显著增大：L0C 从 128 KB 翻倍到 256 KB（Cube 输出缓冲区更大，减少写回次数），UB 从 192 KB 增加到 248 KB（Vector 工作区更大，可以放更多中间数据）。

Q：为什么不直接把所有数据都放在 UB 里？

A：UB 容量有限（A5 只有 248KB），而实际算子要处理的数据量通常是 MB 甚至 GB 级别。所以需要分批次从 GM 搬运数据到 UB，处理完再搬回去。Tiling 的核心任务之一就是规划这个分批搬运的策略。

运行时必须通过 GetMemSizeInBytes 等接口获取实际容量，禁止硬编码具体数值。不同芯片版本的容量可能不同，硬编码会导致移植时出错。

sswap 的工作区就是 UB。它从 GM 把 x 和 y 的数据分别搬入 UB 中的两个队列（inQueueX_ 和 inQueueY_），然后交叉写回 GM——x 的数据写到 y 的位置，y 的数据写到 x 的位置。整个过程不经过 L1/L0A/L0B/L0C，因为 sswap 不使用 Cube 单元。

3.1.4 MTE 数据搬运引擎

MTE（Memory Transfer Engine）是昇腾芯片中专门负责数据搬运的硬件引擎。你可以把它理解为狗西购物的物流系统——它不做任何加工，只负责把包裹从一个地方搬到另一个地方。MTE2 是干线运输（从总仓 GM 到城市分拣中心 UB），MTE3 是派送（从门店 UB 送到顾客手里 GM），MTE1 是店内调拨（从冷库 L1 到吧台 L0A/L0B）。

引擎搬运路径说明
MTE1L1 ↔ L0A/B/C, L1 ↔ UB核内搬运
MTE2GM → L1 / UB从全局搬入
MTE3UB → GM / L1搬出到全局

三个引擎各司其职：MTE2 负责"进货"（从 GM 搬数据到核内），MTE3 负责"出货"（从核内搬数据回 GM），MTE1 负责核内各级 Buffer 之间的内部调拨。三者可以并行工作，这就是为什么在 Ascend C 中可以实现"搬入-计算-搬出"的流水线重叠。

sswap 的数据流非常清晰——CopyIn 阶段通过 MTE2 将 x 和 y 从 GM 搬入 UB（inQueueX_ 和 inQueueY_），CopyOut 阶段通过 MTE3 将 UB 中的数据交叉写回 GM（inQueueX_ 的内容写到 yGM，inQueueY_ 的内容写到 xGM）。

3.1.5 Cube 算力公式

了解算力公式有助于你评估算子的理论性能上限。Cube BF16/FP16 的理论峰值算力（TFLOPS）由以下公式计算：

TFLOPS = M × K × N × 核数 × cube_freq(MHz) × 2 ÷ 10⁶

其中 M×K×N 是 Cube MAC 阵列的规模（FP16/BF16 下为 16×16×16 = 4096），×2 是因为一次 MAC（乘累加）算 2 次浮点运算。

芯片核数频率Cube BF16 算力
Ascend910B2241.8 GHz353T
Ascend950PR PCIE281.65 GHz378T
Ascend950PR Server321.65 GHz432T

对于 FP8 精度，MAC 阵列扩大为 16×32×16 = 8192，算力翻倍。对于 MXFP4，阵列进一步扩大为 16×64×16 = 16384，算力再翻倍。精度越低、吞吐越高——这是量化加速的基本原理。
