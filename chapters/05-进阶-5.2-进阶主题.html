📘 Explanation ← 返回目录

本文介绍 Ascend C 在 A5（Ascend 950 / DAV_3510）上引入的一系列高级特性。这些特性面向有经验的开发者，能够进一步释放硬件性能。文中会以 sswap 算子为例，重点说明跨架构兼容的设计思路。

5.2.1 SIMD-RegBase 深入

在前面的章节中，我们使用的都是 MemBase 编程模型——通过 UB 地址访问数据，API 接受 LocalTensor 作为操作数。MemBase 的好处是简单直观，缺点是编译器无法对寄存器做深度优化。

A5 新增了 RegBase 编程模型，让你直接操作向量寄存器（Vector Register），类似于写 GPU 的 PTX 或 CPU 的 intrinsics。RegBase 模式下，数据从 UB 加载到寄存器组中，所有计算都在寄存器上完成，最后再写回 UB。

寄存器组

RegBase 模式提供了多种专用寄存器：

寄存器类型用途
向量寄存器（VR）存放向量数据，是主要的计算载体
VFScalar标量寄存器，存放标量操作数
地址寄存器存放 UB 地址，用于 Load/Store
对齐寄存器存放对齐信息，辅助非对齐访问
掩码寄存器存放 mask，控制哪些元素参与计算

__VEC_SCOPE__ 声明

__VEC_SCOPE__ 是一个作用域标记，告诉编译器"这个代码块内的寄存器分配是局部的"。在 scope 内分配的寄存器，在 scope 结束后会被自动释放，可以被后续代码复用。

MicroAPI — 细粒度寄存器操作

RegBase 提供了比 MemBase 更细粒度的 API，称为 MicroAPI。比如你可以直接操作寄存器的某几个 lane，或者手动控制寄存器的复用策略。

OOO 双发 — 乱序执行

A5 的 Vector 单元支持 Out-of-Order 双发射：两条没有数据依赖的指令可以在同一个时钟周期同时发射。这在 MemBase 模式下由编译器自动调度，而在 RegBase 模式下你可以手动安排指令顺序来最大化双发射的利用率。

RegBase 代码示例

__VEC_SCOPE__ {
    RegTensor<float, 128> regA;
    RegTensor<float, 128> regB;
    regA.Load(ubA);
    regB.Load(ubB);
    regA.Add(regB);
    regA.Store(ubC);
}

这段代码的含义是：从 UB 加载两个 128 元素的向量到寄存器，做逐元素加法，结果写回 UB。整个过程都在寄存器上完成，没有中间的 UB 读写。

sswap 不使用 RegBase，因为它是纯搬运算子，没有 Vector 计算阶段。RegBase 的价值在于计算密集型算子——比如 GEMM 的内核循环、Reduce 的归约树等。

5.2.2 SIMT 混合编程

传统的 Ascend C 算子使用 SIMD 模式——一个核内的所有计算单元（Vector、Scalar、MTE）协同处理同一份数据。A5 引入了 SIMT（Single Instruction Multiple Threads）模式，让你可以像写 CUDA 一样用线程并行处理数据。

更强大的是，A5 支持在同一个 Kernel 中混合使用 SIMT 和 SIMD：

  • SIMT 部分适合处理离散内存访问（如 Gather/Scatter）和分支密集的逻辑。
  • SIMD 部分适合处理密集的向量计算和规整的数据搬运。

混合编程的声明方式

通过 __global__ __aicore__ 声明混合核函数。SIMT 的 Vector Function 通过 VF_CALL 调用，SIMD 的 Vector Function 通过 asc_vf_call 调用。

混合编程场景

典型的混合编程场景包括：

  • SIMT 做控制流，SIMD 做数据计算。 比如用 SIMT 线程并行遍历稀疏矩阵的非零元素位置，然后调用 SIMD 的 Vector 指令做实际的乘加运算。
  • SIMT 处理 Gather/Scatter，SIMD 处理密集计算。 比如 Embedding 查表用 SIMT（地址不连续），后续的 LayerNorm 用 SIMD（数据连续）。

LAUNCH_BOUND 与寄存器数量

SIMT 模式下，线程数和每个线程可用的寄存器数是一个 trade-off——线程越多，每个线程分到的寄存器就越少：

线程数范围每线程可用寄存器数
1025~204816
513~102432
257~51264
1~256127

选择合适的线程数需要根据算子的寄存器压力来决定。如果寄存器不够用，编译器会 spill 到 UB，性能反而会下降。

5.2.3 NDDMA 高维 DMA

传统的 DataCopy 是一维搬运——从 GM 的一段连续地址搬到 UB 的一段连续地址。但在深度学习中，很多场景需要搬运多维张量的某个切片，或者在搬运的同时做转置。

A5 新增的 NDDMA（N-Dimensional DMA） 指令支持高维数据搬运，可以在一次 DMA 操作中完成多维 stride 访问和转置。

适用场景

NDDMA 特别适合 last-axis 长度较短的场景。比如一个形状为 [N, D] 的张量，如果 D = 16（last axis 很短），传统 DataCopy 每次只能搬 16 个元素，并行度很低。通过 NDDMA 可以先做转置，把 last axis 变长，凑足并行度后再做计算。

搬入 (NDDMA) → 计算 (last 轴 reduce 变非 last 轴 reduce) → 搬出 (Transpose + DataCopyPad)

sswap 是一维向量操作，不涉及多维搬运，因此不需要 NDDMA。但对于 GEMM 的分块搬运、Attention 的 QKV 切片等场景，NDDMA 可以显著减少搬运次数。

5.2.4 CCU 通算融合

CCU（Collective Communication Unit） 是 A5 新增的专用通信单元。在分布式训练中，AllReduce、AllGather 等集合通信操作通常需要占用 AI Core 来完成数据搬运和归约——这意味着通信和计算在抢同一块算力。

CCU 的出现改变了这个局面：通信操作由 CCU 独立完成，不占用 AI Core 的计算资源。 这意味着你可以在 AI Core 做 MatMul 的同时，让 CCU 在后台做 AllReduce，两者完全并行。

三种通信范式

方式说明计算核占用
AIV + UBMem经典方式，AI Core 通过 UB 中转完成通信占用
AIV 直驱 URMAAI Core 发起异步通信，但仍需参与调度占用
AIV + CCUCCU 完成同步、Reduce、搬运全流程不占用

CCU 模式特别适合"计算 + 通信"重叠的场景，比如大模型训练中的 Megatron 并行策略。

5.2.5 跨架构兼容

在实际工程中，同一个算子往往需要同时支持 A2/A3（arch22）和 A5（arch35）。ops-blas 仓的做法是在算子目录下按架构分子目录：

blas/swap/sswap/
├── arch22/          # A2/A3 实现
│   ├── sswap_host.cpp
│   └── sswap_kernel.cpp
└── arch35/          # A5 实现
    ├── sswap_host.cpp
    ├── sswap_kernel.cpp
    └── sswap_tiling_data.h

编译时通过 --soc 参数选择目标架构，构建系统会自动选择对应的子目录。

架构差异对比：以 sswap 为例

sswap 的 arch22 和 arch35 实现展示了两个架构之间的核心差异：

数据搬运方式不同。 arch22 使用底层的 gm_to_ub_align / ub_to_gm_align 函数，需要手动计算 nBurst、leftPaddingNum、rightPaddingNum 等参数。arch35 使用高层的 DataCopy + DataCopyPad API，对齐细节由框架自动处理。

同步方式不同。 arch22 需要手动编写 SET_FLAG / WAIT_FLAG 来协调 MTE2 和 MTE3 流水线，还要管理 EVENT_ID0 / EVENT_ID1 两个事件 ID。arch35 使用 TQueBind 的 EnQue / DeQue，同步是隐式的。

Buffer 管理不同。 arch22 手动从 UB 中划分出 ping/pong 四块 Buffer（ub_x_ping、ub_x_pong、ub_y_ping、ub_y_pong），手动管理偏移量。arch35 使用 TPipe::InitBuffer 自动分配，通过 TQueBind 管理生命周期。

核数获取不同。 arch22 在 Host 侧硬编码了 DEFAULT_VECTOR_CNT = 40。arch35 通过 aclrtGetDeviceInfo(ACL_DEV_ATTR_VECTOR_CORE_NUM) 动态获取。

架构兼容模式

如果你的算子需要在 Tiling 函数中根据架构选择不同的切分策略，可以使用运行时分支：

NpuArch npuArch = ascendcPlatform.GetCurNpuArch();
if (npuArch == NpuArch::DAV_3510) {
    // A5 特殊逻辑（可用 RegBase/SIMT）
} else {
    // A2/A3 通用逻辑（仅 MemBase）
}

在 Kernel 侧，可以用编译宏做条件编译：

#if __NPU_ARCH__ == 3510
    // A5 专用代码
#else
    // 通用代码
#endif

不过对于 sswap 这种简单算子，ops-blas 选择了更干净的做法——直接为两个架构各写一套独立的实现，共享同一份 TilingData 定义和接口签名。这样每个架构的代码都是最简洁的，不会因为兼容逻辑而变得复杂。

5.2.6 与框架对接

算子开发完成后，需要注册到 CANN 框架才能被上层应用调用。常见的对接方式包括：

CANN 框架直调。 通过 aclblas 接口直接调用，就像 sswap 的 aclblasSswap 一样。这是最直接的方式，适合 C/C++ 应用。

PyTorch 自定义算子。 通过 torch-npu 的自定义算子注册机制，将 Ascend C 算子包装成 PyTorch 的 torch.ops 调用。

MindSpore 算子对接。 通过 MindSpore 的自定义算子框架，将 Ascend C 算子注册为 MindSpore 的 Primitive。

具体的对接方式取决于你的目标框架，这里不展开讨论。核心要点是：无论对接哪个框架，底层的 Ascend C Kernel 代码都是一样的，变的只是 Host 侧的封装层。
