4.2b 运行与查看结果

上一节我们完成了 swap 算子的编译，本节将运行 ST（System Test）测试，查看测试结果，并学会理解输出日志和排查常见错误。

运行 ST 测试

ops-blas 仓库自带 ST 测试框架，可以自动验证算子的正确性。

# 运行 swap 算子的 ST 测试
cd test/swap
bash run_test.sh

# 或者使用统一测试入口
cd ../..
bash test/run_all_tests.sh --ops=swap

理解输出日志

编译和运行过程中会产生大量日志，学会快速定位关键信息很重要。

查看 NPU 执行日志

# NPU 运行时日志在 plog 目录
ls plog/
# 输出：
# plog-xxxx-xxxxxx.log

# 查看日志中的 swap 相关信息
grep -i "swap" plog/plog-*.log | head -20

# 关键日志字段：
# [INFO] Kernel launch: swap_kernel
# [INFO] Block count: 8
# [INFO] Execution time: 12 us

日志级别说明

日志级别含义处理方式
[INFO]正常信息一般无需关注
[WARN]警告信息检查是否影响功能
[ERROR]错误信息必须修复
[FATAL]致命错误程序终止，必须修复

编译日志关键字段

# 编译成功的关键标志
[INFO] Build success!

# 如果编译失败，关注 error 信息
[ERROR] swap_kernel.cpp:42: undefined reference to 'DataCopy'
# 说明：API 使用错误，检查头文件包含

[ERROR] swap_kernel.cpp:15: expected ';' after expression
# 说明：语法错误，检查代码

常见错误码解读

运行时错误码是定位问题的关键线索。以下是 swap 算子开发中最常遇到的错误码：

错误码含义常见原因解决方法
161001Tiling 参数错误TilingData 计算逻辑有误检查 totalN、blockCount 等参数是否合理
361001Kernel 执行超时Kernel 死循环或同步问题检查 for 循环边界和 Pipe 同步点
561001内存分配失败UB 空间超限减小 Tiling 分块大小，检查 Buffer 分配
561002Kernel 查找失败芯片型号与编译不匹配确认 --soc 参数与当前设备一致
 
# 运行时错误码示例
[ERROR] ACL Error: 161001
# 含义：Tiling 参数错误
# 解决：检查 TilingData 的计算逻辑

[ERROR] ACL Error: 361001
# 含义：Kernel 执行超时
# 解决：检查 Kernel 是否有死循环或同步问题

[ERROR] ACL Error: 561001
# 含义：内存分配失败
# 解决：检查 UB 空间是否超限

Q&A：测试与日志常见问题

Q：测试通过了但性能数据比预期差很多，怎么办？

A：首先确认测试时是否有其他进程占用 NPU（用 npu-smi info 查看）。其次检查 Tiling 参数是否合理——如果 blockCount 太小，多核并行度不够；如果单次搬运数据量太小，带宽利用率会很低。性能优化详见第 5.2 节。

Q：plog 日志文件太多，如何快速找到有用的信息？

A：使用 grep 过滤关键词。常用模式：grep -i "error\|warn" plog/plog-*.log 查找所有错误和警告；grep "swap" plog/plog-*.log 查找与 swap 算子相关的日志。

Q：测试用例中 float16_edge_case 的 N=33 有什么特殊含义？

A：N=33 是一个非对齐的数据规模。Ascend C 的 Vector 指令通常要求数据按 256B 对齐，N=33 会触发尾块处理逻辑（tail handling），用于验证算子在边界条件下的正确性。
