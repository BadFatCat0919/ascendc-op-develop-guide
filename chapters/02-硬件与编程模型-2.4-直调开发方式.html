2.4.1 直调 vs aclnn 对比

直调（Direct Call）是最简单的算子调用方式——直接从 Host 侧调用 Kernel 函数，不经过 aclnn 框架的封装。

直调适合学习和快速验证。你不需要理解 aclnn 的两段式接口（先获取 workspace 大小、再执行计算），只需要准备好数据、算好 Tiling、启动 Kernel 就行。生产环境最终需要对接 aclnn 框架，但入门时先掌握直调可以让你更快理解算子的运行机制。

维度直调aclnn
复杂度低 — 直接调用 Kernel高 — 两段式接口 + 框架注册
灵活性高 — 完全自主控制中 — 受框架约束
框架对接需要自己处理框架自动调度
Tiling 传递手动 malloc + memcpy框架自动管理
适用场景学习、验证、原型开发生产环境、模型部署

2.4.2 Host 侧代码详解

一个完整的直调 Host 侧入口函数如下：

#include "acl/acl.h"

int main() {
    // 1. 初始化 ACL
    aclInit(nullptr);
    aclrtSetDevice(0);

    // 2. 准备输入数据（在 GM 上分配内存）
    size_t size = n * sizeof(float);
    void* devX = nullptr;
    void* devY = nullptr;
    aclrtMalloc(&devX, size, ACL_MEM_MALLOC_HUGE_FIRST);
    aclrtMalloc(&devY, size, ACL_MEM_MALLOC_HUGE_FIRST);

    // 3. 拷贝数据到 GM
    aclrtMemcpy(devX, size, hostX, size, ACL_MEMCPY_HOST_TO_DEVICE);

    // 4. 计算 Tiling 并写入 GM
    MyTilingData tiling;
    tiling.totalN = n;
    tiling.perCoreN = n / coreNum;
    tiling.remainder = n % coreNum;
    tiling.tileSize = computeTileSize(n, coreNum);

    void* devTiling = nullptr;
    aclrtMalloc(&devTiling, sizeof(MyTilingData), ACL_MEM_MALLOC_HUGE_FIRST);
    aclrtMemcpy(devTiling, sizeof(MyTilingData),
                &tiling, sizeof(MyTilingData),
                ACL_MEMCPY_HOST_TO_DEVICE);

    // 5. 启动 Kernel
    my_op_kernel<<<coreNum, stream>>>(devX, devY, nullptr, devTiling);

    // 6. 拷贝结果回 CPU
    aclrtMemcpy(hostY, size, devY, size, ACL_MEMCPY_DEVICE_TO_HOST);

    // 7. 清理
    aclrtFree(devX);
    aclrtFree(devY);
    aclrtFree(devTiling);
    aclrtResetDevice(0);
    aclFinalize();
}

流程非常直观：初始化设备 → 分配 GM 内存 → 拷入数据 → 启动 Kernel → 拷出结果 → 清理资源。

Kernel 启动语法

<<<coreNum, stream>>> 是 Ascend C 的 Kernel 启动语法：

  • coreNum — 指定启动多少个 AI Core，每个 Core 独立执行一次 Kernel 函数
  • stream — 执行流，用于管理异步操作的执行顺序

封装启动函数

Host 侧通常还会封装一个 C 风格的启动函数，把 Kernel 启动的细节封装起来：

void my_kernel_do(GM_ADDR x, GM_ADDR y, GM_ADDR workSpace,
                  GM_ADDR tilingGm, uint32_t numBlocks, void* stream)
{
    my_kernel<<<numBlocks, nullptr, stream>>>(x, y, nullptr, tilingGm);
}

这样 Host 侧的上层函数（如 aclblasSswap）只需要传入参数和核数就可以启动算子，不用关心 <<<>>> 语法的细节。

2.4.4 动手试试：写一个最简 Kernel

让我们跟着写一个最简单的 Kernel 骨架，把前面讲的概念串起来：

// kernel/add_kernel.cpp
#include "kernel_operator.h"

extern "C" __global__ __aicore__ void add_kernel(
    GM_ADDR a, GM_ADDR b, GM_ADDR c,
    GM_ADDR workspace, GM_ADDR tiling)
{
    // 获取 tiling 数据
    GET_TILING_DATA(tilingData, tiling);
    auto blockIdx = AscendC::GetBlockIdx();

    // 计算当前核的数据范围
    uint32_t offset = blockIdx * tilingData.tileSize;
    uint32_t length = tilingData.tileSize;

    // 声明 UB Buffer
    AscendC::TPipe pipe;
    AscendC::TQue<AscendC::QuePosition::VECIN, 1> queIn;
    AscendC::TQue<AscendC::QuePosition::VECOUT, 1> queOut;

    // 初始化 Buffer
    pipe.InitBuffer(queIn, 2, length * sizeof(float));
    pipe.InitBuffer(queOut, 1, length * sizeof(float));

    // 搬入
    auto inA = queIn.AllocTensor<float>();
    auto inB = queIn.AllocTensor<float>();
    AscendC::DataCopy(inA, a + offset, length);
    AscendC::DataCopy(inB, b + offset, length);
    queIn.EnQue(inA);
    queIn.EnQue(inB);

    // 计算
    auto valA = queIn.DeQue<float>();
    auto valB = queIn.DeQue<float>();
    auto valC = queOut.AllocTensor<float>();
    AscendC::Add(valC, valA, valB, length);
    queOut.EnQue(valC);

    // 搬出
    auto outC = queOut.DeQue<float>();
    AscendC::DataCopy(c + offset, outC, length);

    // 释放
    queIn.FreeTensor(valA);
    queIn.FreeTensor(valB);
    queOut.FreeTensor(outC);
}

这段代码展示了 Kernel 的基本骨架：搬入 → 计算 → 搬出。

  • TPipe 是流水线管理器，负责分配和管理 UB 空间
  • TQue 是队列，用来组织 UB 中的 Buffer——VECIN 队列存放搬入的数据，VECOUT 队列存放待搬出的结果
  • AllocTensor 从队列中分配一块 Buffer
  • EnQue 将填好数据的 Buffer 入队（表示"我写完了，可以给别人用了"）
  • DeQue 从队列中取出一块 Buffer（表示"我要开始读了"）
  • FreeTensor 释放 Buffer

这套 EnQue/DeQue 机制实现了流水线的同步：搬入线程把数据 EnQue 到 VECIN，计算线程从 VECIN DeQue 出来做计算，算完 EnQue 到 VECOUT，搬出线程再从 VECOUT DeQue 搬回 GM。

swap 算子的特殊之处：swap 的 Kernel 结构更简单——它没有"计算"步骤，只有"搬入"和"搬出"。swap 的巧妙之处在于交叉写回：搬入时 x 的数据进入 inQueueX_，y 的数据进入 inQueueY_；搬出时 inQueueX_ 的内容写到 yGM，inQueueY_ 的内容写到 xGM，全程不需要任何算术运算。

现在你只需要理解整体流程，不用纠结每个 API 的细节。后续章节会逐一讲解 TPipe、TQue、DataCopy 等 API 的详细用法。

Q&A：常见疑问

Q：直调方式中，Tiling 数据是怎么从 Host 传到 Kernel 的？

A：Host 侧先在 CPU 内存中填好 TilingData 结构体，然后通过 aclrtMemcpy 把它拷贝到 GM（HBM）上。Kernel 启动时通过 tilingGm 参数拿到这块 GM 的地址，再用 reinterpret_cast 把它转成 TilingData 结构体指针来读取。

Q：<<<coreNum, stream>>> 中的 stream 可以为 nullptr 吗？

A：可以。传 nullptr 表示使用默认执行流。在简单场景下（如单算子测试），用默认流就够了。多算子并发或需要精确控制执行顺序时，才需要创建和管理多个 stream。

Q：为什么直调方式适合学习但不适合生产？

A：直调方式让你直接看到算子运行的每一步，有助于理解底层机制。但生产环境需要 aclnn 框架提供的自动调度、workspace 管理、算子注册、多框架适配等能力。直调方式缺少这些基础设施，不适合直接用于生产部署。
