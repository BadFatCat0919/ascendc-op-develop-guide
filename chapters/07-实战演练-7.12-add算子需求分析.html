4.8 add 算子需求分析

4.8.1 功能说明

add 算子是最基础的 Element-wise（逐元素）算子。给定两个相同长度的向量 A 和 B，逐元素相加得到输出向量 C：

C[i] = A[i] + B[i],  i = 0, 1, ..., N-1

虽然计算逻辑只有一行加法，但 add 算子麻雀虽小五脏俱全——它包含了 aclnn 框架开发的全部环节：文件结构、TilingData 设计、Kernel 实现、Host 侧接口注册、CMakeLists 配置、编译和测试。

算子分类

add 属于 Element-wise 算子，特点是：

  • 输入输出的每个元素独立计算，无元素间依赖
  • 计算强度低（一次加法），瓶颈在内存带宽
  • 适合用 Vector 单元处理，不需要 Cube 单元
  • 数据流简单：搬入 → 计算 → 搬出

4.8.2 aclnn 两段式接口简介

aclnn 是 CANN 提供的算子接口框架，上层框架（如 PyTorch）通过它来调用底层算子。它采用两段式接口设计：

为什么要分两段？因为有些算子需要额外的临时内存（workspace）来做中间计算。第一段告诉调度器"我需要多大的 workspace"，调度器分配好内存后，第二段才真正执行计算。对于 add 这种简单算子，workspace 大小为 0，但接口形式仍然保留。

两段式接口的命名规则：

接口命名格式作用
第一段aclnnXxxGetWorkspaceSize计算 workspace 大小和 Tiling 数据
第二段aclnnXxx执行算子计算

其中 Xxx 是算子名的大驼峰形式，比如 add 算子对应 aclnnAdd。

4.8.3 与直调方式的对比

在前面 4.3 节中，我们用直调方式跑通了 swap 算子。aclnn 框架方式和直调方式的关键差异如下：

维度直调方式aclnn 框架
Kernel 启动Host 侧直接 <<<coreNum, stream>>>框架自动调度，通过 executor 启动
Tiling 计算Host 侧手动计算，手动 aclrtMemcpy 到 GM框架调用 Tiling 函数，自动传递到 Kernel
Workspace自行分配和管理两段式接口自动管理
参数校验自行实现框架提供统一的校验机制
上层对接需要自己封装 C 接口框架自动生成 aclnn 接口，PyTorch 直接调用
适用场景学习、验证、原型开发生产环境、正式算子入库

建议：入门阶段先用直调方式理解算子的运行机制（参考 4.3 节），掌握后再切换到 aclnn 框架进行正式开发。两者的 Kernel 侧代码几乎完全相同，主要区别在 Host 侧。

4.8.4 文件结构

aclnn 框架下的 add 算子，文件结构和直调方式基本一致：

blas/blas1/
└── add/
    └── arch35/
        ├── add_tiling_data.h     # TilingData 结构体（Host/Kernel 共用）
        ├── add_kernel.cpp        # Kernel 侧：AscendC 类 + Kernel 入口
        └── add_host.cpp          # Host 侧：aclnn 两段式接口实现

三个文件的职责：

  • add_tiling_data.h：定义 Host 和 Kernel 之间的通信协议
  • add_kernel.cpp：Ascend C Kernel 实现，包含搬入→计算→搬出流程
  • add_host.cpp：Tiling 计算 + aclnn 两段式接口注册

Q&A

Q：add 算子这么简单，有必要用 aclnn 框架吗？

A：有必要。虽然 add 的逻辑简单，但在生产环境中，算子需要被 PyTorch 等上层框架调用，必须通过 aclnn 框架注册接口。aclnn 框架还提供了统一的参数校验、内存管理和调度机制，是正式算子入库的必经之路。

Q：workspace 为 0 的算子还需要两段式接口吗？

A：需要。两段式接口是 aclnn 框架的统一规范，即使 workspace 为 0，接口形式仍然保留。这样做的好处是上层框架可以用统一的流程调用所有算子，不需要针对有无 workspace 做特殊处理。
