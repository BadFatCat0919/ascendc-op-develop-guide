在实际工程中，同一个算子往往需要同时支持 A2/A3（arch22）和 A5（arch35）。ops-blas 仓的做法是在算子目录下按架构分子目录：

blas/swap/sswap/
├── arch22/          # A2/A3 实现
│   ├── sswap_host.cpp
│   └── sswap_kernel.cpp
└── arch35/          # A5 实现
    ├── sswap_host.cpp
    ├── sswap_kernel.cpp
    └── sswap_tiling_data.h

编译时通过 --soc 参数选择目标架构，构建系统会自动选择对应的子目录。

arch22 vs arch35 差异

以 sswap 算子为例，两个架构之间的核心差异体现在以下几个方面：

数据搬运方式不同

arch22 使用底层的 gm_to_ub_align / ub_to_gm_align 函数，需要手动计算 nBurst、leftPaddingNum、rightPaddingNum 等参数。arch35 使用高层的 DataCopy + DataCopyPad API，对齐细节由框架自动处理。

同步方式不同

arch22 需要手动编写 SET_FLAG / WAIT_FLAG 来协调 MTE2 和 MTE3 流水线，还要管理 EVENT_ID0 / EVENT_ID1 两个事件 ID。arch35 使用 TQueBind 的 EnQue / DeQue，同步是隐式的。

Buffer 管理不同

arch22 手动从 UB 中划分出 ping/pong 四块 Buffer（ub_x_ping、ub_x_pong、ub_y_ping、ub_y_pong），手动管理偏移量。arch35 使用 TPipe::InitBuffer 自动分配，通过 TQueBind 管理生命周期。

核数获取不同

arch22 在 Host 侧硬编码了 DEFAULT_VECTOR_CNT = 40。arch35 通过 aclrtGetDeviceInfo(ACL_DEV_ATTR_VECTOR_CORE_NUM) 动态获取。

条件编译策略

Tiling 侧：运行时分支

如果你的算子需要在 Tiling 函数中根据架构选择不同的切分策略，可以使用运行时分支：

NpuArch npuArch = ascendcPlatform.GetCurNpuArch();
if (npuArch == NpuArch::DAV_3510) {
    // A5 特殊逻辑（可用 RegBase/SIMT）
} else {
    // A2/A3 通用逻辑（仅 MemBase）
}

Kernel 侧：编译宏

在 Kernel 侧，可以用编译宏做条件编译：

#if __NPU_ARCH__ == 3510
    // A5 专用代码（可使用 RegBase、SIMT 等高级特性）
#else
    // 通用代码（仅 MemBase）
#endif

目录级分离（推荐）

对于 sswap 这种简单算子，ops-blas 选择了更干净的做法——直接为两个架构各写一套独立的实现，共享同一份 TilingData 定义和接口签名。这样每个架构的代码都是最简洁的，不会因为兼容逻辑而变得复杂。

最佳实践

  1. 优先目录级分离：当两个架构的实现差异较大时（如使用不同的 API、不同的编程模型），为每个架构写独立的实现文件。代码更清晰，维护更容易。
  2. 共享 TilingData 定义：不同架构的实现可以共享同一份 TilingData 结构体定义和接口签名，保持一致性。
  3. 编译宏用于小差异：如果只是个别 API 调用或参数不同，可以用 #if __NPU_ARCH__ == 3510 做条件编译，避免代码重复。
  4. 运行时分支用于 Tiling：Tiling 函数在 Host 侧执行，可以用 GetCurNpuArch() 做运行时分支，根据架构选择不同的切分策略。
  5. 不要混合使用高级特性：arch22 的代码中不要使用 RegBase、SIMT、NDDMA、CCU 等 A5 独有的特性，否则编译会失败。

Q&A

Q: 为什么 sswap 选择目录级分离而不是条件编译？

因为 sswap 在 arch22 和 arch35 上的实现差异很大：搬运 API 不同、同步方式不同、Buffer 管理方式不同。如果用条件编译，代码会充满 #if 分支，可读性很差。目录级分离让每个架构的代码都是最简洁的。

Q: 可以在 arch22 的代码中使用 RegBase 吗？

不可以。RegBase 是 DAV_3510（arch35）独有的编程模型。在 arch22 的代码中使用 RegBase API 会导致编译失败。

Q: 如何判断当前编译的目标架构？

在 Kernel 侧，可以使用 __NPU_ARCH__ 编译宏判断。在 Host 侧（Tiling 函数中），可以使用 ascendcPlatform.GetCurNpuArch() 运行时获取。
