3.2 编程模型

3.2.1 并行编程模型总览

编程模型是硬件能力和软件表达之间的抽象层——它决定了程序员如何描述并行计算。选对编程模型，代码写起来事半功倍；选错了，要么性能上不去，要么根本实现不了。

昇腾 NPU 支持以下几种编程模型：

编程模型全称核心思想昇腾支持
SIMDSingle Instruction Multiple Data一条指令同时处理多个数据✅ 所有架构
SIMTSingle Instruction Multiple Threads一条指令调度多个线程✅ 仅 A5
SPMDSingle Program Multiple Data多个核运行同一份代码，处理不同数据✅ Ascend C 默认模型
MPMDMultiple Program Multiple Data多个核运行不同代码✅ 通过 PyPTO 框架支持

下面从外到内、从粗到细逐一讲解。

3.2.2 SPMD：多核并行的基础

SPMD 是昇腾多核并行的核心模型。它的意思是：所有 AI Core 运行同一份 Kernel 代码，但每个核处理不同的数据块。

可以用公司团建来类比：20 个同事（20 个 AI Core）做同一套试卷（同一份 Kernel 代码），但每人分到不同的题号（不同的数据块）。

每个核通过 AscendC::GetBlockIdx() 获取自己的编号，再结合 Tiling 参数算出自己该处理哪段数据。这就是 SPMD 的全部含义——代码共享，数据分工。

SPMD 是外层的多核并行模型。在每个核内部，还可以选择 SIMD 或 SIMT 作为单核的计算方式。因此昇腾的编程模型可以理解为两层嵌套：SPMD（多核）嵌套 SIMD/SIMT（单核）。

以 swap 算子为例，Host 侧获取可用的 Vector 核数，然后将 n 个元素均分给各个核。每个核拿到自己的编号后，算出自己负责的数据范围，只处理自己负责的那段数据。这就是典型的 SPMD。

3.2.2.1 SPMD vs MPMD

维度SPMD（Ascend C）MPMD（PyPTO）
程序所有核运行同一份代码不同核运行不同代码
编程复杂度低高
灵活性中（通过 if/else 区分行为）高
昇腾支持Ascend C 默认模型通过 PyPTO 框架支持
适用场景单个算子开发复杂融合算子、整网模型开发

SPMD 是 Ascend C 算子开发的基础，日常开发单个算子时只需要掌握 SPMD 就够了。如果你的需求更复杂——比如需要不同核运行不同的计算逻辑（MPMD），或者要开发复杂的融合算子乃至整个模型网络，可以考虑使用 PyPTO（Parallel Tensor/Tile Operation）框架。PyPTO 采用基于 Tile 的编程模型，通过多层次编译优化，将 Python 代码编译为高效的 NPU 可执行代码，并以 MPMD 方式调度到设备上的处理器核。

3.2.3 SIMD：传统向量编程

SIMD 的核心思想是"一条指令处理多个数据"。Vector 单元是一个 128 bit 宽的 SIMD 单元，一条 Add 指令就可以同时完成 64 个 float 的加法。

类比：想象你弟弟打了一辆网约车，车上一共 4 个座位。不管坐满 4 个人还是只坐 1 个人，对司机来说操作都是一样的——踩油门、打方向盘、到目的地，花的时间也固定。这就是 SIMD——一条指令（一趟车）同时处理多个数据（多个乘客），不管实际装了多少数据，执行时间都一样。传统的 CPU 就相当于只有 1 个座位的车，每次只能拉 1 个乘客，要拉 64 个乘客就得跑 64 趟。而 Vector 单元是一辆 64 座的大巴，一趟全拉走。

3.2.3.1 SIMD 的两个子模型

SIMD 在昇腾上进一步分为两个子模型，区别在于中间计算结果存放在哪里：

子模型别名中间数据位置适用架构
MemBaseBasic APIUB（Unified Buffer）所有架构
RegBaseVF Fusion API寄存器仅 A5

MemBase 是经典模式，所有架构通用；RegBase 是 A5 新增的高性能模式，适合多步融合计算。下面分别讲解。

3.2.4 SIMD-MemBase：经典模式

MemBase 是最基础、最通用的编程模式。它的核心特征很简单：所有中间数据都存储在 UB（Unified Buffer）中。

3.2.4.1 编程流程

1. 从 GM 搬数据到 UB (DataCopy, MTE2)
2. 在 UB 上做计算 (Add/Mul/Reduce...)
3. 从 UB 搬结果回 GM (DataCopy, MTE3)

这个三步流程就是 MemBase 的全部：搬入 → 计算 → 搬出。UB 既是计算的"工作台"，也是中间结果的"暂存区"。

MemBase 的优点是所有架构都支持，编程模型简单直观，API 丰富（TPipe、TQue 等流水线管理工具都可以使用）。缺点是当需要多步融合计算时（比如先做 Add 再做 Mul 再做 Cast），每一步的中间结果都要写回 UB 再读出来，UB 的读写带宽可能成为瓶颈。

swap 算子选择 SIMD-MemBase，因为它是纯数据搬运，不需要多步融合计算。swap 只需要把 x 和 y 的数据从 GM 搬入 UB，再从 UB 交叉写回 GM。这种"搬入即搬出"的场景，MemBase 完全够用，不需要 RegBase 的寄存器级优化。

3.2.5 SIMD-RegBase：寄存器级计算（A5 新增）

RegBase 是 A5 新增的高性能编程模式。它的核心特征是：中间计算结果保存在寄存器中，而不是 UB。

为什么要这样做？因为在多步融合计算中，MemBase 模式下每一步都要把中间结果写回 UB、下一步再读出来，UB 的读写带宽成为瓶颈。RegBase 让中间结果留在寄存器里，下一步直接从寄存器读取，完全绕过了 UB 的带宽限制。只有最开始的数据搬入和最终的结果搬出才需要访问 UB。

3.2.5.1 RegBase vs MemBase

维度MemBaseRegBase
中间数据位置UB寄存器
UB 访问次数每步都读写仅首尾读写
多步融合效率低高
指令双发不支持支持（OOO 双发）
非 32B 对齐不支持支持
架构支持所有仅 A5
编程复杂度低中

RegBase 还有一个独特优势：指令双发（Out-of-Order 双发）。硬件可以在一个时钟周期内同时发射两条不相关的指令，进一步提升计算单元的利用率。此外，RegBase 支持非 32B 对齐的数据访问，这在处理某些特殊数据布局时非常有用。

3.2.5.2 RegBase 代码形态

RegBase 的代码风格与 MemBase 有明显不同。MemBase 使用 block/repeat 参数来描述批量操作（"这条指令重复执行 N 次"），而 RegBase 使用 for-loop 显式循环（程序员自己写循环，每次处理一小块数据）。

RegBase 使用 __VEC_SCOPE__ 声明向量寄存器的作用域，使用 __simd_vf__ 声明 VF（Vector Function）函数。这些关键字告诉编译器：这个函数内部的变量应该分配到向量寄存器中，而不是 UB。

3.2.6 SIMT：线程级并行（A5 新增）

SIMT 是 A5 引入的全新编程模型。在 SIMT 模式下，每个线程处理一个或少量数据元素，硬件按 Warp（32 线程一组）进行调度执行。

继续用打车的类比来理解 SIMD 和 SIMT 的区别：SIMD 是一辆大巴，一趟拉走所有乘客，但所有人必须去同一个目的地——如果有人要去不同的地方，大巴就得挨个送，效率大打折扣。SIMT 是每人开一辆车，每个人可以直接开到自己的目的地，灵活得多，但调度成本更高（要管更多车）。所以：所有人去同一个地方的时候（规整的大块数据），大巴（SIMD）成本低；所有人需求不同的时候（离散、不规则的数据访问），各自开车（SIMT）成本低。

Q：SIMT 比 SIMD 更先进，是不是应该优先用 SIMT？

A：不是。SIMT 和 SIMD 各有适用场景。SIMD（大巴）适合规整的大块数据，一条指令处理多个元素，效率极高。SIMT（各自开车）适合离散、不规则的数据访问，灵活但调度成本更高。选择哪种取决于算子的数据访问模式，而不是哪个更"先进"。

3.2.6.1 SIMT 线程层次

层级概念说明设置方式
启动核数Block 数量启动多少个 AI CoreTiling 侧 SetBlockDim
启动线程数单核线程数每个 AI Core 内部启动多少线程Kernel 侧 constexpr 常量
Warp32 线程分组硬件自动将 32 个线程分为一组硬件自动分组

Warp 的概念——硬件自动将 32 个线程分为一组，组内线程同步执行同一条指令。

3.2.6.2 SIMT 内存空间

内存说明访问修饰符
寄存器/栈线程私有默认
UB核内共享__ubuf__
GM全局内存__gm__

SIMT 的一个重要特性是支持直接读写 GM，不需要像 SIMD 那样显式调用 DataCopy 搬入搬出。这意味着你可以像写普通 C++ 代码一样直接通过指针访问全局内存，编程体验更直观。

SIMT 模式下可用的 UB 容量为 256KB - 8KB（系统预留）- 32KB（DCache，SIMT 模式下的数据缓存，详见 7.3）= 216KB。

3.2.6.3 SIMT 适用场景

SIMT 擅长处理离散、不规则的访问模式：

  • 适合：Gather/Scatter（离散地址读写）、Hash 计算、随机数生成、排序（支持原子操作）、稀疏计算。
  • 不适合：大块 dense 矩阵乘（用 SIMD + Cube 更高效）、长向量的顺序计算（SIMD 的宽向量更合适）。

3.2.6.4 SIMT 函数调用层级

SIMT 和 SIMD 的 VF（Vector Function）不能跨模式调用——SIMT 的 VF 函数只能被 SIMT 代码调用，SIMD 的 VF 函数只能被 SIMD 代码调用。如果需要在 SIMT 代码中调用 SIMD 的能力，需要通过 asc_vf_call 机制进行模式切换。

3.2.7 编程模型选型指南

3.2.7.1 按架构选择

架构可用模型
A5 以前 (arch22, DAV_2201)SIMD-MemBase
A5 (arch35, DAV_3510)SIMD-MemBase + SIMD-RegBase + SIMT

A5 以前的架构只有 SIMD-MemBase 一种选择，没有选型烦恼。A5 有三种模型可选，需要根据算子特点做出合理选择。

3.2.7.2 按算子类型选择

算子类型推荐模型原因
简单逐元素 (Add/Mul)SIMD-MemBase简单直接，所有架构通用
多步融合 (Gelu+Cast)SIMD-RegBase减少 UB 读写，性能更好
矩阵乘 (MatMul)SIMD + CubeCube 单元最高效
离散访问 (Gather)SIMT支持 GM 直接访问
排序/HashSIMT支持原子操作和线程级控制

swap 算子选择 SIMD-MemBase 的原因很直接——它是纯数据搬运算子，没有多步融合计算（不需要 RegBase），没有离散访问（不需要 SIMT），只需要把数据从 GM 搬入 UB 再交叉写回 GM。MemBase 的流水线工具恰好提供了搬入-搬出的完整管理能力，是最合适的选择。
