4.7 swap 算子编译与测试

代码写完后，我们需要编译并在 NPU 上运行测试。本节将介绍编译命令、测试方法和常见问题排查。

4.7.1 编译命令

ops-blas 提供了统一的构建脚本 build.sh，支持按算子名和芯片版本编译。

基本编译命令

# 编译 swap 算子（指定目标芯片）
bash build.sh --ops=swap --soc=Ascend910B

# 编译所有算子
bash build.sh --soc=Ascend910B

# 清理编译产物
bash build.sh --clean

编译参数说明

参数说明示例
--ops指定要编译的算子名--ops=swap
--soc指定目标芯片版本--soc=Ascend910B
--clean清理编译产物--clean
--debug启用调试模式--debug

编译产物

build/
├── swap/
│   ├── swap_kernel.o          # Kernel 目标文件
│   ├── swap_host.o            # Host 目标文件
│   └── libswap.so             # 动态链接库
└── test/
    └── swap_test              # 测试可执行文件

4.7.2 运行测试

编译成功后，运行测试程序验证 swap 算子的正确性。

运行命令

# 运行 swap 测试
./build/test/swap_test

# 指定测试用例（通过 GTest 过滤器）
./build/test/swap_test --gtest_filter="SwapTest.BasicSwap"

# 运行所有测试并输出详细信息
./build/test/swap_test --gtest_print_time=1

测试用例说明

测试用例说明预期结果

LargeN大规模数据（N=100000）所有元素正确交换

4.7.4 常见问题

问题 1：编译报错 "kernel_operator.h: No such file"

原因：CANN 环境变量未配置。

# 解决方案：source CANN 环境变量
source /usr/local/Ascend/ascend-toolkit/set_env.sh

问题 2：运行时报错 "Failed to allocate device memory"

原因：NPU 内存不足或设备未初始化。

# 检查 NPU 设备状态
npu-smi info

# 检查是否有其他进程占用 NPU
npu-smi info -t usages -i 0

问题 3：测试结果全为 0

原因：Kernel 未正确执行，可能是 TilingData 传递错误。

// 检查 TilingData 是否正确传递
// 在 Kernel 入口函数中添加调试输出
printf("totalN=%d, perCoreN=%d, tileSize=%d\n", 
       tiling.totalN, tiling.perCoreN, tiling.tileSize);

问题 4：部分元素未交换

原因：余数处理不正确，最后一个 Core 的数据没有完全处理。

// 检查 Init 函数中的余数处理
if (coreIdx == tiling.coreCount - 1) {
    myN += tiling.remainder;  // 确保最后一个 Core 处理余数
}

问题 5：aclrtSynchronizeDevice 超时

原因：Kernel 死循环或卡死。

// 检查 Process 函数的循环条件
while (processed < myN) {  // 确保循环能终止
    int32_t currentSize = ...;
    SingleIteration(processed, currentSize);
    processed += currentSize;  // 确保 processed 递增
}

4.7.5 调试技巧

1. 使用 printf 调试

在 Kernel 中可以使用 printf 输出调试信息（注意：输出可能乱序）。

// 在 Kernel 中添加调试输出
__aicore__ inline void Init(...) {
    printf("Core %d: myN=%d, tileSize=%d\n", 
           GetBlockIdx(), myN, tileSize);
}

2. 使用 plog 日志

对于生产环境，建议使用 plog 日志框架。

#include "log_inner.h"

// 记录日志
ASCEND_LOG_INFO("swap kernel: coreIdx=%d, myN=%d", coreIdx, myN);

3. 使用 Simulator 调试

如果没有 NPU 硬件，可以使用 CANN Simulator 进行仿真调试。

# 使用 Simulator 编译
bash build.sh --ops=swap --soc=Ascend910B --simulator

# 运行仿真
./build/test/swap_test --simulator

4.7.6 性能测试

功能验证通过后，可以使用 profiling 工具测试性能。

# 使用 msprof 采集性能数据
msprof op --kernel=swap_kernel_entry --input=swap_input.json

Q&A：常见问题

Q1：为什么编译时需要指定 --soc 参数？

A：不同芯片版本（如 Ascend910A、Ascend910B）的 AI Core 架构不同，支持的指令集和 UB 大小也不同。编译器需要根据目标芯片生成对应的机器码。

Q2：测试失败时应该先检查什么？

A：按以下顺序排查：1) 检查编译是否成功（Kernel 二进制是否生成）→ 2) 检查 TilingData 是否正确传递 → 3) 检查 Kernel 是否正确执行（添加 printf 调试）→ 4) 检查数据拷贝是否正确（H2D 和 D2H）。

Q3：如何在没有 NPU 的环境下调试？

A：使用 CANN Simulator。Simulator 可以在 CPU 上模拟 NPU 的执行行为，支持断点调试和单步执行。虽然性能数据不准确，但功能验证完全可用。
