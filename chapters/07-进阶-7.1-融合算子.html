1.5 融合算子

7.1.1 融合算子的定义

在前面几节中，我们了解了算子的基本概念、数据排布、数据类型和量化技术。现在我们来看一个在实际工程中极其重要的概念——融合算子（Fused Operator）。

所谓融合算子，就是把多个原本独立执行的小算子合并成一个大算子，在同一个 Kernel 里完成所有计算。这不是简单的"把代码粘在一起"，而是一种深层的性能优化策略。

为什么要融合？核心原因是减少数据搬运。NPU 的计算单元（Vector / Cube）速度极快，但从 Global Memory（显存）搬运数据到片上缓存（UB）的速度相对较慢。如果每个算子都独立执行，数据就要反复在显存和 UB 之间搬运，搬运时间可能远超计算时间。

举个直观的例子：一个典型的卷积层包含 Conv2D → BiasAdd → ReLU 三个算子。如果不融合：

  • Conv2D 从显存读输入，计算后写回显存
  • BiasAdd 从显存读 Conv2D 的输出，加偏置后写回显存
  • ReLU 从显存读 BiasAdd 的输出，做激活后写回显存

数据搬运了 3 次。如果融合成一个 "Conv2D_BiasAdd_ReLU" 算子：

  • 从显存读输入到 UB
  • 在 UB 里依次完成 Conv2D、BiasAdd、ReLU
  • 最后写回显存 1 次

数据搬运减少到 1 次，搬运开销减少约 2/3——尤其在数据量大、计算简单的场景下，搬运往往是算子的主要耗时，融合带来的性能提升非常显著。

Q：融合算子是不是越多越好？把所有算子都融合成一个超大 Kernel 不是最快吗？

A：不是。融合有边界：（1）UB 空间有限（A5 是 248KB），融合太多算子会导致中间结果放不下；（2）融合后的 Kernel 复杂度增加，调试和优化难度上升；（3）有些算子之间有数据依赖（如归约操作需要全局同步），无法简单融合。实际工程中需要在性能、可维护性、UB 空间之间找平衡。

7.1.2 融合算子的分类

按照融合的计算类型组合方式，融合算子可以分为以下几大类：

(1) 计算 + 激活融合

这是最常见、最经典的融合模式。矩阵运算（如 MatMul、Conv2D）后面紧跟一个激活函数（如 ReLU、Sigmoid、GELU），两者融合在一起。

融合算子组成典型场景
MatMulReLUMatMul + ReLU全连接层 + 激活
Conv2D_BiasAdd_ReLUConv2D + BiasAdd + ReLU标准卷积层
MatMulGELUMatMul + GELUTransformer 中的 FFN 层
Conv2DSigmoidConv2D + Sigmoid检测/分割网络

这类融合的核心思路是：矩阵运算的输出结果不写回显存，而是直接留在 UB 中，紧接着做激活计算，最后一次性写回。因为激活函数是逐元素操作（每个元素独立计算），不需要额外的 UB 空间来存放中间结果，所以融合代价很小，收益很大。

(2) 计算 + 归一化融合

矩阵运算后接归一化操作（如 LayerNorm、BatchNorm）也是一种常见的融合模式。

融合算子组成典型场景
MatMulLayerNormMatMul + LayerNormTransformer 子层输出
ConvBatchNormConv2D + BatchNormCNN 中的卷积层
MatMulRMSNormMatMul + RMSNormLLaMA 等大模型

与激活融合相比，归一化融合更具挑战性——归一化操作通常需要先做一次归约（如求均值、方差），然后再对每个元素做标准化。这意味着需要在 UB 中保存中间结果（均值、方差），并且归约操作本身需要多步计算。

(3) 多步计算融合

将多个计算步骤融合成一个完整的流程，典型代表是各种注意力机制和复杂网络模块的融合实现。

融合算子组成典型场景
FlashAttentionQKT → Scale → Mask → Softmax → ×VTransformer 注意力
FusedResidualMatMul + Add（残差连接）+ LayerNormTransformer 残差块
FusedMLPMatMul + GELU + MatMulTransformer FFN 模块

这类融合的复杂度最高，需要精心设计数据流和流水线，但性能收益也最大——因为它消除了多个中间张量的显存读写。

(4) 注意力机制融合

这是大模型时代最重要的一类融合。注意力机制（Attention）涉及多个矩阵运算和 Softmax，如果不融合，中间结果（如注意力矩阵 QKT）的规模是 O(N2)，对显存的压力极大。

融合算子特点典型场景
FlashAttention分块计算注意力，避免 O(N2) 中间矩阵通用 Transformer
MC2计算 + 通信融合，重叠 AllReduce 与计算分布式大模型训练
PagedAttention分页管理 KV Cache，减少显存碎片大模型推理

这类融合算子通常是大模型推理和训练的核心性能瓶颈，也是 Ascend C 开发中最具挑战性的算子。

7.1.3 知名融合算子介绍

(1) MC2

MC2（Multi-Head Attention with Computation and Communication fusion） 是昇腾 NPU 上专为分布式大模型训练设计的高性能注意力算子。

解决什么问题？ 在分布式训练中，多卡并行需要 AllReduce 通信来同步梯度。传统做法是"先计算、后通信"——计算和通信串行执行，通信时间完全暴露在关键路径上。对于大模型（如 GPT、LLaMA），通信时间可能占到总训练时间的 30% 以上。

核心思想： MC2 将注意力计算和 AllReduce 通信融合在同一个算子中，让两者重叠执行——当一部分 AI Core 在做矩阵乘法时，另一部分数据正在通过 HCCS（华为芯片间互联）进行跨卡通信。这样通信时间被"隐藏"在计算时间里，整体性能大幅提升。

MC2 是昇腾在大模型训练场景下的核心竞争力之一，它充分利用了 DaVinci 架构中 AI Core 和 HCCS 通信引擎可以并行工作的硬件特性。

(2) FlashAttention

FlashAttention 是近年来最有影响力的注意力优化算法之一，由 Tri Dao 等人于 2022 年提出。它的核心思想是：通过分块（Tiling）计算注意力，避免在显存中存储 O(N2) 的完整注意力矩阵。

标准 Attention 的问题：

Attention(Q, K, V) = Softmax(Q × KT / √d) × V

标准实现需要先把完整的 Q × KT 矩阵（大小为 N×N）写到显存，再做 Softmax，最后乘以 V。当序列长度 N 很大时（如 N=8192），这个中间矩阵需要 8192×8192×2B = 128MB 的显存——而且每个注意力头都要存一份。

FlashAttention 的做法： 把 Q、K、V 分成小块（Block），每次只加载一小块到片上缓存（UB），在 UB 内完成 QKT → Scale → Softmax → ×V 的全部计算，然后把结果写回。通过巧妙的数值技巧（在线 Softmax 算法），保证分块计算的结果与标准实现完全一致。

FlashAttention 在昇腾 NPU 上的实现是 Ascend C 开发中的经典案例——它需要精心设计 Tiling 策略（如何分块）、UB 空间分配（每块多大才能放得下）以及流水线（如何让数据搬运和计算重叠）。后面的章节会详细讲解这些技术。

(3) 其他常见融合算子

除了 MC2 和 FlashAttention，还有许多在实际模型中广泛使用的融合算子：

融合算子组成说明
FusedLayerNormLayerNorm（均值 + 方差 + 标准化 + 缩放偏移）将 LayerNorm 的多步归约和标准化融合为单个 Kernel，减少中间数据的显存搬运
FusedGELUGELU 激活（含近似计算）GELU 的计算涉及 erf 函数或近似多项式，融合后避免中间结果搬运
FusedRMSNormRMSNorm（均方根 + 标准化）LLaMA 等大模型的核心组件，比 LayerNorm 更轻量但仍需融合优化
FusedSoftmaxSoftmax（Max → Sub → Exp → Sum → Div）将 Softmax 的 5 步操作融合，避免每步都读写显存
FusedResidualAddAdd + LayerNorm残差连接后紧跟归一化，是 Transformer 中出现频率最高的模式之一

这些融合算子的共同特点是：将多个访存密集型操作合并，让中间结果在片上缓存中流转，只在最终输出时写回显存。这也是为什么理解 UB 空间规划和流水线设计对于算子开发者如此重要。

7.1.4 融合算子的设计挑战

设计一个融合算子远比设计单个简单算子复杂。以下是开发者需要面对的核心挑战：

(1) UB 空间规划

融合算子需要在 UB 中同时存放多个中间结果。以 FlashAttention 为例，UB 需要同时容纳：

  • Q 的分块数据
  • K 的分块数据
  • V 的分块数据
  • QKT 的中间结果
  • Softmax 的中间变量（max 值、exp 累加和）
  • 最终输出的累加结果

A5 芯片的 UB 大小为 248KB，如何在这有限的空间内合理分配各部分数据，是 Tiling 设计的核心问题。分块太大放不下了，分块太小则计算效率低——需要精确计算每种数据类型的占用量，找到最优的平衡点。

(2) 流水线设计

融合算子通常包含多个计算步骤，理想情况下应该让这些步骤重叠执行——当 Cube 单元在做矩阵乘法时，Vector 单元可以同时做激活计算，MTE 可以同时搬运下一批数据。这就是流水线（Pipeline）设计。

设计好流水线需要深入理解 DaVinci 架构中各硬件单元的并行能力：哪些单元可以同时工作、数据在单元之间如何流转、同步信号如何设置。这也是后面"同步与流水线"章节要重点讲解的内容。

(3) 数值稳定性

融合后的计算流程可能引入数值精度问题。例如：

  • FlashAttention 的分块 Softmax 需要使用"在线 Softmax"算法来保证数值一致性
  • 多个 FP16 计算步骤融合后，累积误差可能比分开执行更大
  • 归一化操作中的方差计算在低精度下容易溢出

开发者需要在融合带来的性能收益和精度损失之间做出权衡。

(4) 多核并行策略

融合算子通常计算量更大，需要多个 AI Core 并行处理。如何切分数据（Tiling）、如何分配给不同的 Core、如何处理 Core 之间的数据依赖，都是需要仔细设计的问题。不合理的并行策略可能导致负载不均衡（有些 Core 忙、有些 Core 闲），反而降低整体性能。

小结：融合算子是 Ascend C 开发中最有价值也最具挑战的部分。它要求开发者不仅理解单个算子的实现，还要具备全局视角——规划 UB 空间、设计流水线、保证数值稳定性、优化并行策略。好消息是，后面章节讲解的所有核心技术（Tiling、内存管理、同步机制）都是为了解决这些挑战而准备的。
