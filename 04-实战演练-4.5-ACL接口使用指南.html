4.5.1 ACL 运行时 API 概览

ACL（Ascend Computing Language）是昇腾 NPU 的运行时编程接口，类似于 CUDA 的 Runtime API。算子的 Host 侧代码通过 ACL 来管理设备、分配内存、控制执行流。本节汇总了算子开发中最常用的 ACL API，方便随时查阅。

ACL API 按功能分为以下几个模块：

模块核心 API作用
初始化aclInit, aclFinalizeACL 运行时的初始化和销毁
设备管理aclrtSetDevice, aclrtResetDevice选择和管理 NPU 设备
内存管理aclrtMalloc, aclrtMemcpy, aclrtFreeDevice 内存的分配、拷贝和释放
流管理aclrtCreateStream, aclrtSynchronizeStream异步执行流的管理
事件管理aclrtCreateEvent, aclrtRecordEvent流间同步和事件记录
错误处理aclGetRecentErrMsg获取最近的错误信息

4.5.2 初始化与销毁

ACL 运行时在使用前必须初始化，使用完毕后必须销毁。这是所有 ACL 调用的前提。

aclInit

aclError aclInit(const char *configPath);
参数类型说明
configPathconst char*配置文件路径，传 nullptr 使用默认配置

返回值：ACL_SUCCESS（0）表示成功，其他值表示失败。

调用时机：程序启动时调用一次，在所有 ACL 操作之前。

aclFinalize

aclError aclFinalize();

调用时机：程序退出前调用一次，在所有 ACL 操作之后。必须与 aclInit 成对出现。

// 标准用法
int main() {
    aclInit(nullptr);       // 初始化
    // ... 所有 ACL 操作 ...
    aclFinalize();          // 销毁
    return 0;
}

4.5.3 设备管理

一个服务器可能有多张 NPU 卡，设备管理 API 用来选择当前要使用哪张卡。

aclrtSetDevice

aclError aclrtSetDevice(int32_t deviceId);
参数类型说明
deviceIdint32_t设备 ID，从 0 开始编号

作用：设置当前线程使用的 NPU 设备。后续的内存分配、Kernel 启动等操作都会在这个设备上执行。

aclrtResetDevice

aclError aclrtResetDevice(int32_t deviceId);

作用：重置指定设备，释放该设备上的所有资源。必须与 aclrtSetDevice 成对使用。

aclrtGetDeviceCount

aclError aclrtGetDeviceCount(uint32_t *count);

作用：获取当前可用的 NPU 设备数量。

// 典型用法：选择第一张卡
aclrtSetDevice(0);
// ... 执行算子 ...
aclrtResetDevice(0);

注意：aclrtSetDevice 是线程级的——每个线程可以设置不同的设备。如果你在多线程环境中使用，每个线程都需要单独调用 aclrtSetDevice。

4.5.4 内存管理

内存管理是算子开发中使用频率最高的 ACL API 模块。Host 侧需要在 Device（NPU HBM）上分配内存来存放输入、输出和 Tiling 数据。

aclrtMalloc

aclError aclrtMalloc(void **devPtr, size_t size, aclrtMemMallocPolicy policy);
参数类型说明
devPtrvoid**输出参数，返回分配的 Device 内存地址
sizesize_t分配的字节数
policyaclrtMemMallocPolicy内存分配策略

常用分配策略：

策略说明
ACL_MEM_MALLOC_HUGE_FIRST优先从 Huge Page 分配（推荐，性能更好）
ACL_MEM_MALLOC_HUGE_ONLY只从 Huge Page 分配，不够则失败
ACL_MEM_MALLOC_NORMAL_ONLY只从普通页分配

aclrtMemcpy

aclError aclrtMemcpy(void *dst, size_t destMax,
                      const void *src, size_t count,
                      aclrtMemcpyKind kind);
参数类型说明
dstvoid*目标地址
destMaxsize_t目标缓冲区大小（安全检查）
srcconst void*源地址
countsize_t拷贝的字节数
kindaclrtMemcpyKind拷贝方向

拷贝方向：

枚举值说明
ACL_MEMCPY_HOST_TO_DEVICECPU → NPU（把输入数据送到 Device）
ACL_MEMCPY_DEVICE_TO_HOSTNPU → CPU（把计算结果取回 Host）
ACL_MEMCPY_DEVICE_TO_DEVICENPU → NPU（Device 内部拷贝）
ACL_MEMCPY_HOST_TO_HOSTCPU → CPU（一般不用）

aclrtFree

aclError aclrtFree(void *devPtr, size_t size);

作用：释放 aclrtMalloc 分配的 Device 内存。必须与 aclrtMalloc 成对使用，否则会导致内存泄漏。

// 完整的内存管理示例
size_t dataSize = n * sizeof(float);
float* devInput = nullptr;
float* devOutput = nullptr;

// 分配 Device 内存
aclrtMalloc((void**)&devInput, dataSize, ACL_MEM_MALLOC_HUGE_FIRST);
aclrtMalloc((void**)&devOutput, dataSize, ACL_MEM_MALLOC_HUGE_FIRST);

// CPU → NPU：把输入数据送到 Device
aclrtMemcpy(devInput, dataSize, hostInput, dataSize, ACL_MEMCPY_HOST_TO_DEVICE);

// ... 启动 Kernel 执行计算 ...

// NPU → CPU：把结果取回 Host
aclrtMemcpy(hostOutput, dataSize, devOutput, dataSize, ACL_MEMCPY_DEVICE_TO_HOST);

// 释放 Device 内存
aclrtFree(devInput, dataSize);
aclrtFree(devOutput, dataSize);

4.5.5 流管理

Stream（流）是 NPU 上的异步执行队列。同一个 Stream 内的操作按提交顺序执行，不同 Stream 之间的操作可以并行。

aclrtCreateStream

aclError aclrtCreateStream(aclrtStream *stream);

作用：创建一个新的执行流。创建后可以用它来启动 Kernel 和执行内存拷贝。

aclrtDestroyStream

aclError aclrtDestroyStream(aclrtStream stream);

作用：销毁指定的执行流，释放相关资源。

aclrtSynchronizeStream

aclError aclrtSynchronizeStream(aclrtStream stream);

作用：阻塞等待指定 Stream 中的所有操作完成。这是最常用的同步方式——Kernel 启动是异步的，调用后不会等 Kernel 跑完就返回，必须用 SynchronizeStream 来等待。

aclrtGetCurrentContext

aclError aclrtGetCurrentContext(aclrtContext *context);

作用：获取当前线程的 Context。Context 是设备上的运行环境，一个设备可以有多个 Context。

// 流管理的典型用法
aclrtStream stream = nullptr;
aclrtCreateStream(&stream);

// 使用 stream 启动 Kernel（异步，立即返回）
my_kernel<<<coreNum, nullptr, stream>>>(devA, devB, devC, tilingGm);

// 等待 Kernel 执行完成
aclrtSynchronizeStream(stream);

// 此时才能安全地拷贝结果
aclrtMemcpy(hostOutput, size, devOutput, size, ACL_MEMCPY_DEVICE_TO_HOST);

// 销毁流
aclrtDestroyStream(stream);

常见错误：启动 Kernel 后没有调用 aclrtSynchronizeStream 就直接拷贝结果。由于 Kernel 是异步执行的，此时结果还没算完，拷贝回来的数据是未初始化的垃圾值。

4.5.6 事件管理

Event（事件）用于流间同步——当一个 Stream 的操作依赖另一个 Stream 的结果时，用 Event 来建立依赖关系。

aclrtCreateEvent

aclError aclrtCreateEvent(aclrtEvent *event);

作用：创建一个事件对象。

aclrtRecordEvent

aclError aclrtRecordEvent(aclrtEvent event, aclrtStream stream);

作用：在指定 Stream 中记录一个事件。当 Stream 执行到该事件时，事件被标记为"已完成"。

aclrtStreamWaitEvent

aclError aclrtStreamWaitEvent(aclrtStream stream, aclrtEvent event);

作用：让指定 Stream 等待某个事件完成后才继续执行后续操作。

aclrtSynchronizeEvent

aclError aclrtSynchronizeEvent(aclrtEvent event);

作用：阻塞等待指定事件完成。

aclrtDestroyEvent

aclError aclrtDestroyEvent(aclrtEvent event);

作用：销毁事件对象。

// 流间同步示例：Stream B 依赖 Stream A 的结果
aclrtEvent event = nullptr;
aclrtCreateEvent(&event);

// Stream A 执行计算
kernelA<<<coreNum, nullptr, streamA>>>(...);
aclrtRecordEvent(event, streamA);  // 在 Stream A 中记录事件

// Stream B 等待事件完成后再执行
aclrtStreamWaitEvent(streamB, event);
kernelB<<<coreNum, nullptr, streamB>>>(...);

// 清理
aclrtSynchronizeStream(streamA);
aclrtSynchronizeStream(streamB);
aclrtDestroyEvent(event);

4.5.7 错误处理

ACL API 的返回值类型是 aclError（即 int32_t），ACL_SUCCESS（0）表示成功，其他值表示各种错误。

aclGetRecentErrMsg

const char* aclGetRecentErrMsg();

作用：获取最近一次 ACL 错误的描述信息。当 API 返回非 0 值时，调用此函数可以获取人类可读的错误描述。

// 错误处理的标准模式
aclError ret = aclrtMalloc(&devPtr, size, ACL_MEM_MALLOC_HUGE_FIRST);
if (ret != ACL_SUCCESS) {
    printf("aclrtMalloc failed: %s\n", aclGetRecentErrMsg());
    // 处理错误...
    return ret;
}

常用错误码

错误码含义常见原因
ACL_SUCCESS (0)成功—
ACL_ERROR_INVALID_PARAM参数无效传了空指针、非法尺寸等
ACL_ERROR_MEMORY_ALLOCATION内存分配失败HBM 空间不足
ACL_ERROR_INTERNAL内部错误驱动异常、设备故障
161xxx运行时配置错误环境变量缺失、版本不匹配
361xxxTiling 错误TilingData 计算有误
561xxxKernel 查找失败Kernel 名拼写错误或未编译

更详细的错误码列表请参考附录 C。

4.5.8 常用数据结构和枚举

aclrtMemcpyKind（拷贝方向）

typedef enum {
    ACL_MEMCPY_HOST_TO_HOST     = 0,
    ACL_MEMCPY_HOST_TO_DEVICE   = 1,
    ACL_MEMCPY_DEVICE_TO_DEVICE = 2,
    ACL_MEMCPY_DEVICE_TO_HOST   = 3
} aclrtMemcpyKind;

aclrtMemMallocPolicy（内存分配策略）

typedef enum {
    ACL_MEM_MALLOC_HUGE_FIRST  = 0,  // 优先 Huge Page
    ACL_MEM_MALLOC_HUGE_ONLY   = 1,  // 仅 Huge Page
    ACL_MEM_MALLOC_NORMAL_ONLY = 2   // 仅普通页
} aclrtMemMallocPolicy;

aclError（返回值类型）

typedef int32_t aclError;
#define ACL_SUCCESS 0

aclrtStream / aclrtEvent（不透明句柄）

typedef void* aclrtStream;
typedef void* aclrtEvent;

Stream 和 Event 都是不透明指针（opaque handle），你不需要知道它们的内部结构，只需要通过 ACL API 来创建、使用和销毁。

4.5.9 完整的 Host 侧模板

下面是一个使用直调方式的完整 Host 侧代码模板，把前面介绍的所有 API 串起来：

#include "acl/acl.h"
#include <cstdio>
#include <cstdlib>

#define ACL_CHECK(call) do { \
    aclError ret = (call); \
    if (ret != ACL_SUCCESS) { \
        fprintf(stderr, "ACL Error at %s:%d - %s\n", \
                __FILE__, __LINE__, aclGetRecentErrMsg()); \
        exit(1); \
    } \
} while(0)

int main() {
    const int n = 1024;
    const size_t size = n * sizeof(float);

    // 1. 初始化
    ACL_CHECK(aclInit(nullptr));
    ACL_CHECK(aclrtSetDevice(0));

    // 2. 准备 Host 数据
    float* hostA = (float*)malloc(size);
    float* hostB = (float*)malloc(size);
    float* hostC = (float*)malloc(size);
    for (int i = 0; i < n; i++) {
        hostA[i] = 1.0f;
        hostB[i] = 2.0f;
    }

    // 3. 分配 Device 内存
    void *devA = nullptr, *devB = nullptr, *devC = nullptr;
    ACL_CHECK(aclrtMalloc(&devA, size, ACL_MEM_MALLOC_HUGE_FIRST));
    ACL_CHECK(aclrtMalloc(&devB, size, ACL_MEM_MALLOC_HUGE_FIRST));
    ACL_CHECK(aclrtMalloc(&devC, size, ACL_MEM_MALLOC_HUGE_FIRST));

    // 4. 准备 Tiling 数据并传到 Device
    MyTilingData tilingData = { .totalLength = n, .tileSize = n / coreNum, ... };
    void* tilingGm = nullptr;
    ACL_CHECK(aclrtMalloc(&tilingGm, sizeof(MyTilingData), ACL_MEM_MALLOC_HUGE_FIRST));
    ACL_CHECK(aclrtMemcpy(tilingGm, sizeof(MyTilingData),
                           &tilingData, sizeof(MyTilingData),
                           ACL_MEMCPY_HOST_TO_DEVICE));

    // 5. 拷贝输入数据到 Device
    ACL_CHECK(aclrtMemcpy(devA, size, hostA, size, ACL_MEMCPY_HOST_TO_DEVICE));
    ACL_CHECK(aclrtMemcpy(devB, size, hostB, size, ACL_MEMCPY_HOST_TO_DEVICE));

    // 6. 创建 Stream 并启动 Kernel
    aclrtStream stream = nullptr;
    ACL_CHECK(aclrtCreateStream(&stream));
    my_kernel<<<coreNum, nullptr, stream>>>(devA, devB, devC, tilingGm);

    // 7. 等待 Kernel 完成
    ACL_CHECK(aclrtSynchronizeStream(stream));

    // 8. 拷贝结果回 Host
    ACL_CHECK(aclrtMemcpy(hostC, size, devC, size, ACL_MEMCPY_DEVICE_TO_HOST));

    // 9. 验证结果
    for (int i = 0; i < n; i++) {
        if (hostC[i] != 3.0f) {
            printf("Mismatch at %d: expected 3.0, got %f\n", i, hostC[i]);
            break;
        }
    }
    printf("All correct!\n");

    // 10. 清理资源
    ACL_CHECK(aclrtDestroyStream(stream));
    ACL_CHECK(aclrtFree(devA, size));
    ACL_CHECK(aclrtFree(devB, size));
    ACL_CHECK(aclrtFree(devC, size));
    ACL_CHECK(aclrtFree(tilingGm, sizeof(MyTilingData)));
    free(hostA); free(hostB); free(hostC);
    ACL_CHECK(aclrtResetDevice(0));
    ACL_CHECK(aclFinalize());

    return 0;
}

4.5.10 最佳实践和注意事项

资源管理

  1. 成对使用：aclInit/aclFinalize、aclrtSetDevice/aclrtResetDevice、aclrtMalloc/aclrtFree、aclrtCreateStream/aclrtDestroyStream 必须成对出现，否则会导致资源泄漏。
  2. 释放顺序：先释放 Stream/Event，再释放内存，最后 ResetDevice 和 Finalize。顺序反了可能导致段错误。
  3. 检查返回值：每个 ACL API 都可能失败，务必检查返回值。推荐使用上面的 ACL_CHECK 宏。

内存管理

  1. 优先使用 Huge Page：ACL_MEM_MALLOC_HUGE_FIRST 性能更好，因为大页内存减少了 TLB miss。
  2. 减少拷贝次数：Host-Device 之间的数据拷贝是性能瓶颈，尽量减少不必要的拷贝。如果数据已经在 Device 上，就不要搬回 Host 再搬回去。
  3. 对齐要求：aclrtMalloc 分配的内存地址是 512B 对齐的，但 aclrtMemcpy 的拷贝长度没有对齐要求。

异步执行

  1. Kernel 是异步的：<<<>>> 启动 Kernel 后立即返回，不等 Kernel 执行完。必须在拷贝结果前调用 aclrtSynchronizeStream。
  2. Memcpy 是同步的：aclrtMemcpy 默认是同步的（阻塞等待拷贝完成）。如果需要异步拷贝，使用 aclrtMemcpyAsync。
  3. 多流并行：不同 Stream 上的 Kernel 可以并行执行。如果算子之间没有数据依赖，可以放在不同 Stream 上提升吞吐。

调试技巧

  1. 开启日志：设置环境变量 ASCEND_GLOBAL_LOG_LEVEL=1 可以开启 ACL 的详细日志输出。
  2. plog 日志：当 ACL API 返回错误时，查看 $HOME/ascend/log/ 目录下的 plog 日志，通常能找到更详细的错误原因。
  3. aclGetRecentErrMsg()：在错误发生后立即调用，获取人类可读的错误描述。
