本文介绍 Ascend C 在 A5（Ascend 950 / DAV_3510）上引入的一系列高级特性。这些特性面向有经验的开发者，能够进一步释放硬件性能。文中会以 sswap 算子为例，重点说明跨架构兼容的设计思路。

5.2.1 SIMD-RegBase 深入

在前面的章节中，我们使用的都是 MemBase 编程模型——通过 UB 地址访问数据，API 接受 LocalTensor 作为操作数。MemBase 的好处是简单直观，缺点是中间数据必须驻留在 UB，编译器无法对寄存器做深度优化，UB 带宽成为性能瓶颈。

A5（DAV_3510）在 Vector 单元上引入了 RegBase 编程模型，与传统 MemBase 并存。RegBase 的核心思想是：中间数据存储在寄存器而非 UB，数据从 UB 加载到寄存器组中，所有计算都在寄存器上完成，最后再写回 UB。这类似于 GPU 的 PTX 或 CPU 的 intrinsics 编程风格。

核心优势

  • 寄存器内计算 — 减少 UB 访问带宽消耗，中间结果不占 UB 空间
  • OOO 指令双发 — 乱序执行双发射，Vector 性能显著提升
  • 支持非 32B 对齐 — 通过地址寄存器和对齐寄存器，可以处理 LB 非 32B 对齐的数据

寄存器组

RegBase 模式提供了五类专用寄存器，构成完整的向量寄存器体系：

寄存器类型用途说明
向量寄存器（VR）存放向量数据，是主要的计算载体每条 VL=256B，承载实际计算数据
VFScalar标量寄存器，存放标量操作数用于标量广播、立即数等场景
地址寄存器存放 UB 地址，用于 Load/Store指向 UB 中的数据位置
对齐寄存器存放对齐信息，辅助非对齐访问配合地址寄存器处理非 32B 对齐数据
掩码寄存器存放 mask，控制哪些元素参与计算实现条件计算和尾部处理

__VEC_SCOPE__ 声明向量寄存器作用域

__VEC_SCOPE__ 是一个作用域标记，告诉编译器"这个代码块内的向量寄存器分配是局部的"。在 scope 内分配的寄存器，在 scope 结束后会被自动释放，可以被后续代码复用。合理使用 __VEC_SCOPE__ 可以大幅降低寄存器压力，让编译器有更多优化空间。

__VEC_SCOPE__ {
    RegTensor<float, 128> regA;
    RegTensor<float, 128> regB;
    regA.Load(ubA);
    regB.Load(ubB);
    regA.Add(regB);
    regA.Store(ubC);
}
__VEC_SCOPE__ {
    RegTensor<float, 128> regX;
    regX.Load(ubD);
    regX.Mul(regX);
    regX.Store(ubE);
}

上面两个 __VEC_SCOPE__ 块各自独立分配寄存器，第一个块结束后其寄存器被释放，第二个块可以复用同一组物理寄存器。

MicroAPI — 细粒度寄存器操作

RegBase 提供了比 MemBase 更细粒度的 API，称为 MicroAPI。MemBase 的 API 以 block / repeat 参数控制批量操作，而 MicroAPI 允许你直接操作寄存器的某几个 lane，或者手动控制寄存器的复用策略。代码形态从 MemBase 的批量参数变为 RegBase 的 for-loop 显式循环：

维度MemBaseRegBase (MicroAPI)
操作粒度block/repeat 批量逐元素 / 逐 lane 精细控制
代码形态API 调用 + 参数for-loop 显式循环
中间数据驻留 UB驻留寄存器
寄存器管理编译器自动开发者手动 + __VEC_SCOPE__

OOO 双发 — 乱序执行

A5 的 Vector 单元支持 Out-of-Order 双发射：两条没有数据依赖的指令可以在同一个时钟周期同时发射。这在 MemBase 模式下由编译器自动调度，而在 RegBase 模式下你可以手动安排指令顺序来最大化双发射的利用率。

OOO 双发对 Vector 算力的提升是显著的。以 Ascend950PR PCIE 为例，Vector FP16 基础算力为 23.7T，启用 RegBase OOO 双发后翻倍至 47T。Server 版本从 27T 提升至 54T。

要充分利用 OOO 双发，需要将无依赖的指令交错排列：

__VEC_SCOPE__ {
    RegTensor<float, 128> a, b, c, d;
    a.Load(ubA);
    c.Load(ubC);
    b.Load(ubB);
    d.Load(ubD);
    a.Add(b);
    c.Add(d);
    a.Store(ubOut1);
    c.Store(ubOut2);
}

上述代码中，a.Add(b) 和 c.Add(d) 没有数据依赖，可以被硬件双发执行。

RegBase 与 MemBase 详细对比

维度MemBaseRegBase
数据驻留UB（Unified Buffer）向量寄存器
API 风格LocalTensor + block/repeatRegTensor + for-loop / MicroAPI
中间数据开销占用 UB 空间不占 UB，驻留寄存器
编译器优化空间有限（UB 访问不可消除）大（寄存器分配 + 指令调度）
OOO 双发编译器自动调度可手动编排最大化
对齐要求32B 对齐支持非 32B 对齐（地址+对齐寄存器）
适用架构所有架构仅 DAV_3510（A5）
学习曲线较低较高（需理解寄存器管理）

完整代码示例

下面是一个更完整的 RegBase 示例，展示了向量加法的完整流程：

__aicore__ inline void VectorAddRegBase(
    const LocalTensor<float>& ubA,
    const LocalTensor<float>& ubB,
    const LocalTensor<float>& ubC,
    uint32_t elemCount)
{
    __VEC_SCOPE__ {
        RegTensor<float, 256> regA;
        RegTensor<float, 256> regB;
        regA.Load(ubA);
        regB.Load(ubB);
        regA.Add(regB);
        regA.Store(ubC);
    }
}

这段代码的含义是：从 UB 加载两个 256 元素的向量到寄存器，做逐元素加法，结果写回 UB。整个过程都在寄存器上完成，没有中间的 UB 读写。

适用场景

RegBase 的价值在于计算密集型算子，典型场景包括：

  • GEMM 内核循环 — 矩阵乘法的累加在寄存器中完成，避免频繁的 UB 读写
  • Reduce 归约树 — 多级归约的中间结果驻留寄存器，减少 UB 带宽压力
  • 融合算子 — 多个计算步骤串联时，中间结果全部在寄存器中流转

sswap 不使用 RegBase，因为它是纯搬运算子，没有 Vector 计算阶段。对于搬运为主的算子，MemBase 已经足够高效。

5.2.2 SIMT 混合编程

传统的 Ascend C 算子使用 SIMD 模式——一个核内的所有计算单元（Vector、Scalar、MTE）协同处理同一份数据。A5（DAV_3510）引入了 SIMT（Single Instruction Multiple Threads）模式，让你可以用线程并行处理数据，编程范式从"矢量编程（VF 内连续计算）"变为"标量编程（线程视角）"。

更强大的是，A5 支持在同一个 Kernel 中混合使用 SIMT 和 SIMD：

  • SIMT 部分适合处理离散内存访问（如 Gather/Scatter）和分支密集的逻辑。
  • SIMD 部分适合处理密集的向量计算和规整的数据搬运。

SIMT vs SIMD 硬件能力差异

维度SIMTSIMD
编程范式标量编程（线程视角）矢量编程（VF 内连续计算）
控制逻辑Warp Scheduler 硬件分支调度软件展开循环
GM 离散访问支持直接访问（核内 DCache）不支持，需先搬入 UB
寄存器类型标量寄存器（per-Thread）矢量寄存器（per-VF）
寄存器组织128 KB 寄存器堆，按线程并发数切分多条 VL=256 B 矢量寄存器

SIMT 独有硬件（DAV_3510 新增）：Warp Scheduler（每 AIV 4 个）、SIMT Register File、SIMT DCache（最大 128KB，复用 UB 作 Cacheline，128B 粒度）。SIMD/SIMT 共享：ALU / ICache / Unified Buffer。

SIMT 线程层次

AscendC SIMT 编程模型采用三级线程层次：

层级概念对应 CUDA设置方式
启动核数Block 数量Grid 中的 Block 数tiling 侧 SetBlockDim
启动线程数单核线程数单 Block 线程数kernel 侧 constexpr 常量
Warp32 线程分组Warp硬件自动分组

核数（Block Dim）：通过 tiling 侧 SetBlockDim 接口设置，切分公式为 总核数 = ceil(输出元素总数 / 单核最少处理元素数)。

线程数：单核最大支持 2048 线程，必须在编译期确定（使用 constexpr 或字面量），禁止从 tiling 数据动态获取。LAUNCH_BOUND(N) 和 Simt::Dim3(N) 必须使用同一个编译期常量。默认值 1024。

Warp 调度：内部按 32 线程分组形成 warp，warp 是硬件调度的最小单位。每个 AIV 核包含 4 个 Warp Scheduler，调度器编号为 warp_id % 4。同一时刻一个 AIV 核只执行一个线程块任务，线程块内多个 Warp 被依次调度执行。

SIMT 内存空间

AscendC SIMT 编程模型包含三类内存空间：

内存类型说明访问范围地址空间修饰符
寄存器和栈每个线程独立线程私有无（默认）
Unified Buffer (UB)本地内存核内共享__ubuf__
Global Memory (GM)全局内存所有线程可访问__gm__

寄存器和栈：每个线程独享一组寄存器，寄存器数量受线程数影响，超出寄存器容量的变量会栈溢出。

Unified Buffer (UB)：核内共享的本地内存，总量 256KB。SIMT 算子不能使用全部 UB 空间，需为 DCache 预留 ≥32KB，实际可用 UB = 256KB - 8KB(预留) - 32KB(DCache) = 216KB。在 SIMT VF 中通过 __ubuf__ 指针访问。

Global Memory (GM)：所有核所有线程均可访问。SIMT VF 支持直接读写 GM，无需显式 Load/Store，通过 __gm__ 指针访问，在 kernel 入口通过 GM_ADDR 参数传入。这是 SIMT 相比 SIMD 的一大优势——SIMD 模式下数据必须先搬入 UB 才能被计算单元访问。

SIMT 支持直接读写 GM

SIMT 模式下，线程可以通过 __gm__ 指针直接读写 Global Memory，这是通过核内 DCache 实现的。DCache 最大 128KB，复用 UB 空间作为 Cacheline，以 128B 为粒度缓存 GM 数据。

__simt_vf__ __aicore__ LAUNCH_BOUND(512) inline void GatherKernel(
    __gm__ const float* src,
    __gm__ float* dst,
    __gm__ const int32_t* indices,
    uint32_t count)
{
    uint32_t tid = Simt::GetThreadId();
    uint32_t stride = Simt::GetBlockSize();
    for (uint32_t i = tid; i < count; i += stride) {
        int32_t idx = indices[i];
        dst[i] = src[idx];
    }
}

上述 Gather 示例中，每个线程通过 indices 读取不连续的地址，SIMT 的 DCache 会自动处理这些离散访问，无需手动搬入 UB。

SIMT 函数调用层级

混合编程中，函数调用遵循严格的层级结构：

核函数 (__global__ __aicore__)
  ├── __aicore__ 函数
  ├── SIMD VF (__simd_vf__) ← 通过 asc_vf_call 调用
  │     └── __simd_callee__ 子函数
  └── SIMT VF (__simt_vf__) ← 通过 VF_CALL 调用
        └── __simt_callee__ 子函数
  • 核函数（__global__ __aicore__）：算子入口，由框架调用，负责 tiling 数据解析和场景分发
  • __aicore__ 函数：核内主逻辑函数，负责 UB buffer 初始化、GM 地址转换，调用 Simt::VF_CALL 启动 SIMT VF
  • SIMT VF（__simt_vf__）：线程级并行计算函数，每个线程独立执行，以 stride 间隔遍历数据
  • __simt_callee__：SIMT VF 内部的辅助函数，必须带修饰符

调用约束：__simt_vf__ 内只能调用 __simt_callee__ 函数和 constexpr 函数；__simd_vf__ 内只能调用 __simd_callee__ 函数和 constexpr 函数。不可跨模式调用（SIMT VF 不能调用 SIMD callee，反之亦然）。

混合编程的声明方式

通过 __global__ __aicore__ 声明混合核函数，DAV_3510 上提供 __global__ 核函数语法和 <<<...>>> 启动方式（等价 CUDA 风格）。SIMT 的 Vector Function 通过 VF_CALL 调用，SIMD 的 Vector Function 通过 asc_vf_call 调用。

VF_CALL vs asc_vf_call

调用方式目标 VF 类型使用场景
Simt::VF_CALLSIMT VF（__simt_vf__）启动线程级并行计算
asc_vf_callSIMD VF（__simd_vf__）启动矢量级计算

在同一个 __global__ __aicore__ 核函数中，可以先调用 VF_CALL 执行 SIMT 逻辑（如 Gather），再调用 asc_vf_call 执行 SIMD 逻辑（如密集 Vector 计算），实现混合编程。

LAUNCH_BOUND 与寄存器数量

LAUNCH_BOUND(N) 限定每个 VF Block 使用的最大线程数，线程数直接影响每线程可用寄存器数：

线程数范围每线程可用寄存器数
1025~204816
513~102432
257~51264
1~256127

选择合适的线程数需要根据算子的寄存器压力来决定。如果寄存器不够用，编译器会 spill 到 UB，性能反而会下降。

算子类型建议线程数原因
搬运类算子2048 / 1024内存带宽受限，更多线程隐藏延迟
计算类算子512 / 1024寄存器压力较大，需平衡并行度和寄存器

寄存器压力随 VF 复杂度增加而增大。极低复杂度（NONE/1D）的 VF 使用 uint32_t 索引时最多支持 1024 线程，最高复杂度（ND 运行时循环）使用 uint64_t 索引时仅支持 128 线程。

典型混合编程场景

场景一：SIMT 做控制流，SIMD 做数据计算。 用 SIMT 线程并行遍历稀疏矩阵的非零元素位置，然后调用 SIMD 的 Vector 指令做实际的乘加运算。

场景二：SIMT 处理 Gather/Scatter，SIMD 处理密集计算。 Embedding 查表用 SIMT（地址不连续），后续的 LayerNorm 用 SIMD（数据连续）。

场景三：SIMT 处理分支密集逻辑。 如排序、Hash 插入、随机数生成等包含大量条件分支的操作，SIMT 的 Warp Scheduler 可以高效处理分支发散。

混合编程代码示例

__global__ __aicore__ void MixedKernel(
    GM_ADDR input, GM_ADDR output,
    GM_ADDR indices, GM_ADDR tiling)
{
    TilingData tilingData;
    GET_TILING_DATA_WITH_STRUCT(tilingData, tiling);

    __gm__ const float* gmInput = (__gm__ const float*)input;
    __gm__ float* gmOutput = (__gm__ float*)output;
    __gm__ const int32_t* gmIndices = (__gm__ const int32_t*)indices;

    Simt::VF_CALL<GatherSimt, LAUNCH_BOUND_VAL>(
        gmInput, gmOutput, gmIndices, tilingData.elemCount);

    asc_vf_call<ComputeSimd>(
        ubInput, ubOutput, tilingData.elemCount);
}

__simt_vf__ __aicore__ LAUNCH_BOUND(512) inline void GatherSimt(
    __gm__ const float* src,
    __gm__ float* dst,
    __gm__ const int32_t* indices,
    uint32_t count)
{
    uint32_t tid = Simt::GetThreadId();
    uint32_t stride = Simt::GetBlockSize();
    for (uint32_t i = tid; i < count; i += stride) {
        dst[i] = src[indices[i]];
    }
}

SIMT 适用与不适用场景

适用场景：Gather/Scatter、Hash 插入、随机数、排序（含原子操作）。

不适用场景：

  • 大块 dense BF16/FP16 矩阵乘 / 卷积 — SIMD + Cube 流水更高效
  • 长向量顺序计算 — SIMD 矢量化指令吞吐更高
  • DAV_3510 之前架构 — SIMT 硬件不存在，无法使用

5.2.3 NDDMA 高维 DMA

传统的 DataCopy 是一维搬运——从 GM 的一段连续地址搬到 UB 的一段连续地址。但在深度学习中，很多场景需要搬运多维张量的某个切片，或者在搬运的同时做转置。传统方式需要多次 DataCopy 配合手动地址计算，代码复杂且效率低。

A5（DAV_3510）新增的 NDDMA（N-Dimensional DMA） 指令支持高维数据搬运，可以在一次 DMA 操作中完成多维 stride 访问和转置。NDDMA 配合 DAV_3510 新增的 ND-DMA Cache，大幅提升离散访问与转置效率。

核心思想

NDDMA 的核心思想是：将多维张量的非连续切片在一次 DMA 操作中完成搬运，避免多次 DataCopy 和手动地址计算。它支持在搬运过程中同时完成 stride 访问和维度转置，将原本需要多步操作的流程压缩为一条 DMA 指令。

适用场景：last-axis 较短时

NDDMA 特别适合 last-axis 长度较短的场景。比如一个形状为 [N, D] 的张量，如果 D = 16（last axis 很短），传统 DataCopy 每次只能搬 16 个元素（FP32 下仅 64B），并行度很低，计算效率仅 16/64 = 25%。

通过 NDDMA 可以先做转置，把 D 轴转置到高轴凑足并行度后再计算，最后通过 Transpose / DataCopyPad 搬出。

使用流程

NDDMA 的典型使用流程分为三步：

搬入 (NDDMA) → 计算 (last 轴 reduce 变非 last 轴 reduce) → 搬出 (Transpose + DataCopyPad)
  1. 搬入：使用 NDDMA 指令将多维张量的切片从 GM 搬入 UB，同时完成维度转置，使短轴不再位于 last axis
  2. 计算：在 UB 中对转置后的数据执行 Vector 计算，此时 last axis 已变长，并行度充足
  3. 搬出：通过 Transpose + DataCopyPad 将结果转置回原始维度顺序并搬出到 GM

NDDMA 与传统 DataCopy 对比

维度传统 DataCopyNDDMA
搬运维度一维连续搬运多维 stride 搬运
转置能力无，需额外 Transpose 操作搬运过程中自动转置
last-axis 短时并行度低，效率差通过转置凑足并行度
代码复杂度需手动计算多维地址偏移一条指令完成多维搬运
硬件支持所有架构仅 DAV_3510（配合 ND-DMA Cache）

代码示例

以形状为 [N, D]（D=16，FP32）的张量归约为例，展示 NDDMA 的使用方式：

__aicore__ inline void ReduceShortLastAxis(
    const LocalTensor<float>& ubSrc,
    const LocalTensor<float>& ubDst,
    uint32_t n, uint32_t d)
{
    NDDMA nddma;
    nddma.SetSrc(gmSrc, sizeof(float));
    nddma.SetDim(2);
    nddma.SetShape(0, n);
    nddma.SetShape(1, d);
    nddma.SetStride(0, d * sizeof(float));
    nddma.SetStride(1, sizeof(float));
    nddma.SetDst(ubTransposed);
    nddma.Run();

    ReduceSum(ubTransposed, ubDst, n, d);

    DataCopyPad(gmDst, ubDst, copyPadParams);
}

上述代码中，NDDMA 一次性完成了 [N, D] 张量从 GM 到 UB 的搬运和转置，使得原本 last axis 仅 16 个元素的问题变为高轴归约，并行度大幅提升。

适用与不适用场景

sswap 是一维向量操作，不涉及多维搬运，因此不需要 NDDMA。NDDMA 的典型适用场景包括：

  • GEMM 分块搬运 — 矩阵分块时的多维 stride 访问
  • Attention QKV 切片 — 从大张量中提取 Q/K/V 切片
  • 短 last-axis 归约 — 如 D=16 的 embedding 归约
  • 多维转置 — 需要在搬运过程中改变维度顺序

5.2.4 CCU 通算融合

CCU（Collective Communication Unit） 是 A5（DAV_3510）在 IO-Die 上新增的专用通信引擎。在分布式训练中，AllReduce、AllGather 等集合通信操作通常需要占用 AI Core 来完成数据搬运和归约——这意味着通信和计算在抢同一块算力。

CCU 的出现改变了这个局面：通信操作由 CCU 独立完成，不占用 AI Core 的计算资源。 这意味着你可以在 AI Core 做 MatMul 的同时，让 CCU 在后台做 AllReduce，两者完全并行。

核心思想

CCU 的核心思想是将集合通信从 AI Core 的计算流程中解耦。传统方式下，AIV（Vector Core）需要亲自参与通信的数据搬运和归约操作，消耗计算周期。CCU 模式下，AIV 只需发起通信请求，后续的同步、Reduce、搬运全流程由 CCU 独立完成。

三种通信范式

方式说明计算核占用
AIV + UBMem经典方式，AIV 写 UB Mem 触发通信占用
AIV 直驱 URMA异步通信，AIV 直接发起 URMA占用
AIV + CCUCCU 完成同步、Reduce、搬运不占用

AIV + UBMem：经典方式，AIV 通过 UB 中转完成通信。AIV 将数据写入 UB 的特定区域来触发通信，整个过程需要 AIV 参与数据搬运和归约，占用计算资源。

AIV 直驱 URMA：AIV 直接发起 URMA（Unified Remote Memory Access）异步通信请求。相比 UBMem 方式减少了 UB 中转，但 AIV 仍需参与通信调度和状态轮询。

AIV + CCU：AIV 发起通信请求后，CCU 接管全部通信流程——包括跨节点同步、数据 Reduce（如 AllReduce 的求和）、数据搬运。AIV 可以立即返回执行计算任务，通信与计算完全并行。

KFC 调度模型变化

CCU 的引入改变了 Kernel Function Call 的调度路径：

模式调度路径
传统AICore (KFC) ↔ AICPU (KFC) → SDMA
CCUAICore (KFC) ↔ CCU (KFC) → URMA

CCU 模式下，通信任务通过 KFC 直接调度到 CCU，绕过了 AICPU 中转环节，通信任务展开更快，AICore 发起到通信开始的静态时延更小。

CCU 对算子开发的优势

  • 通信不占用 AI Core 计算算力 — AI Core 可以全速执行计算，通信在 CCU 上并行完成
  • 通信任务展开更快 — AICore 发起到通信开始的静态时延更小
  • 天然零拷贝 — 片上 buffer + Reduce 单元保序，无需额外的数据拷贝

适用场景

CCU 模式特别适合"计算 + 通信"重叠的场景：

  • AllReduce + MatMul 融合 — 大模型训练中，梯度 AllReduce 与下一层 MatMul 并行执行
  • AllGather + MatMul 融合 — 模型并行中，参数 AllGather 与当前层计算并行
  • Megatron 并行策略 — Tensor Parallelism 中的 AllReduce 与计算重叠

CCU 使用前置条件

条件说明
架构仅 DAV_3510（IO-Die 上配备 CCU）；前代架构无此选项
算法通算融合算子（如 AllReduce / AllGather 与 MatMul 融合）
拓扑集群间通信需支持 URMA 的硬件互联

代码示例

以下是通算融合的概念性代码，展示 CCU 模式下 AllReduce 与 MatMul 并行的编程模式：

__global__ __aicore__ void FusedAllReduceMatMul(
    GM_ADDR gradOutput, GM_ADDR weights,
    GM_ADDR nextInput, GM_ADDR tiling)
{
    TilingData tilingData;
    GET_TILING_DATA_WITH_STRUCT(tilingData, tiling);

    CcuAllReduce(gradOutput, tilingData.gradSize);

    asc_vf_call<MatMulSimd>(
        ubA, ubB, ubC,
        tilingData.m, tilingData.n, tilingData.k);
}

在上述代码中，CcuAllReduce 发起 CCU 通信后立即返回，不阻塞后续的 MatMul 计算。CCU 在后台完成梯度 AllReduce，AI Core 同时执行矩阵乘法，两者完全并行。

5.2.5 跨架构兼容

在实际工程中，同一个算子往往需要同时支持 A2/A3（arch22）和 A5（arch35）。ops-blas 仓的做法是在算子目录下按架构分子目录：

blas/swap/sswap/
├── arch22/          # A2/A3 实现
│   ├── sswap_host.cpp
│   └── sswap_kernel.cpp
└── arch35/          # A5 实现
    ├── sswap_host.cpp
    ├── sswap_kernel.cpp
    └── sswap_tiling_data.h

编译时通过 --soc 参数选择目标架构，构建系统会自动选择对应的子目录。

架构差异对比：以 sswap 为例

sswap 的 arch22 和 arch35 实现展示了两个架构之间的核心差异：

数据搬运方式不同。 arch22 使用底层的 gm_to_ub_align / ub_to_gm_align 函数，需要手动计算 nBurst、leftPaddingNum、rightPaddingNum 等参数。arch35 使用高层的 DataCopy + DataCopyPad API，对齐细节由框架自动处理。

同步方式不同。 arch22 需要手动编写 SET_FLAG / WAIT_FLAG 来协调 MTE2 和 MTE3 流水线，还要管理 EVENT_ID0 / EVENT_ID1 两个事件 ID。arch35 使用 TQueBind 的 EnQue / DeQue，同步是隐式的。

Buffer 管理不同。 arch22 手动从 UB 中划分出 ping/pong 四块 Buffer（ub_x_ping、ub_x_pong、ub_y_ping、ub_y_pong），手动管理偏移量。arch35 使用 TPipe::InitBuffer 自动分配，通过 TQueBind 管理生命周期。

核数获取不同。 arch22 在 Host 侧硬编码了 DEFAULT_VECTOR_CNT = 40。arch35 通过 aclrtGetDeviceInfo(ACL_DEV_ATTR_VECTOR_CORE_NUM) 动态获取。

架构兼容模式

如果你的算子需要在 Tiling 函数中根据架构选择不同的切分策略，可以使用运行时分支：

NpuArch npuArch = ascendcPlatform.GetCurNpuArch();
if (npuArch == NpuArch::DAV_3510) {
    // A5 特殊逻辑（可用 RegBase/SIMT）
} else {
    // A2/A3 通用逻辑（仅 MemBase）
}

在 Kernel 侧，可以用编译宏做条件编译：

#if __NPU_ARCH__ == 3510
    // A5 专用代码
#else
    // 通用代码
#endif

不过对于 sswap 这种简单算子，ops-blas 选择了更干净的做法——直接为两个架构各写一套独立的实现，共享同一份 TilingData 定义和接口签名。这样每个架构的代码都是最简洁的，不会因为兼容逻辑而变得复杂。

5.2.6 与框架对接

算子开发完成后，需要注册到 CANN 框架才能被上层应用调用。常见的对接方式包括：

CANN 框架直调。 通过 aclblas 接口直接调用，就像 sswap 的 aclblasSswap 一样。这是最直接的方式，适合 C/C++ 应用。

PyTorch 自定义算子。 通过 torch-npu 的自定义算子注册机制，将 Ascend C 算子包装成 PyTorch 的 torch.ops 调用。

具体的对接方式取决于你的目标框架，这里不展开讨论。核心要点是：无论对接哪个框架，底层的 Ascend C Kernel 代码都是一样的，变的只是 Host 侧的封装层。
